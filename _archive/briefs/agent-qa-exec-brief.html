vostride/agent-qa — what it is, and how it stacks up

source: github.com/vostride/agent-qa · 105★ · TS monorepo · created 2026-05-07 · active (pushed 2026-06-14)

Bottom line: agent-qa and your QE ecosystem are not competitors — they sit at different layers.

agent-qa is a horizontal, runtime agentic test-execution engine: write tests in plain English, an LLM agent drives the browser/app, heals around UI drift mid-run, and curates a memory that makes future runs smarter. It's a product you'd adopt.

Your QE ecosystem is a vertical, workflow-automation layer bonded to IMA 360: deterministic Playwright + GraphQL tests, plus a deep skills/orchestrator stack wired into Jira, Zephyr, ADO, SQL, and App Insights — the parts agent-qa doesn't touch. That integration depth is your moat.

Recommendation: don't replace anything. Mine agent-qa for three architectural ideas (runtime self-healing, a formal curated-memory subsystem, and an action/plan cache for token savings), and optionally pilot it as a self-healing smoke-runner for cross-browser/mobile — but clear the FSL license first.

What agent-qa actually does

A "self-improving Agentic QA harness with Memory." You describe a test in natural language; agent-qa's LLM agent works from on-screen roles, labels, and state to execute and assert it across web and mobile.

  • Natural-language tests as code — YAML with plain-English setup / steps / teardown (e.g. "Open the hooks demo page", "Verify hook-provided data is visible"). Versioned, diffable, reviewable.
  • Self-healing at runtime — when a click/fill/select fails, the agent re-observes the UI and tries another path in the same run, so tests survive UI drift instead of failing on first break.
  • A real memory subsystem — an LLM curator builds execution memory across product/suite/test tiers, each observation carrying a trust score and confirm/deprecate/delete lifecycle; it learns specifically from healed steps so future runs avoid the same mistake. Includes a security scanner over remembered text.
  • Action cache — reuses validated plans across similar runs to cut planner work, tokens, and runtime (the token-efficiency angle you noticed in your own loop).
  • Sandboxed hooks — Node / Bun / Python / Bash hooks in isolated Docker containers for env setup, API calls, fixture seeding, teardown.
  • Built for humans + machines — polished dashboard, CLI, JUnit/console reporters, PostHog analytics, an MCP server, and packaged skills (authoring / debug-fix / result-triage) for coding agents.
  • Bring-your-own LLM — OpenAI/Anthropic-compatible, Gemini, local models, and subscriptions including Codex and Claude Code.

Packages core (agent loop · memory · reporter · analytics · auth-state) · cli · android (Appium) · dashboard-server · web dashboard · subscription-auth.

Where they overlap (and what to borrow)

🩹 Runtime self-healing

Yours heals at author-time (fix-loop rewrites specs). agent-qa heals mid-run. Borrowable idea: a re-observe-and-retry wrapper for your flakiest Playwright steps before a hard fail — fewer false reds without rewriting code.

🧠 Curated memory with trust

agent-qa's memory has trust scores + confirm/deprecate/delete deltas + a security scanner. Your knowledge store is richer in domain depth but more manual. Worth lifting: a trust/lifecycle model and "learn from healed steps" pattern into your memory + session-review loop.

⚡ Action cache

You just praised the token efficiency of your research loop. agent-qa formalizes this with a validated-plan cache. A plan/observation cache keyed on stable selectors could cut tokens in any agentic step you add to your skills.

🔌 Packaged agent surface

Both ship skills + MCP. agent-qa proves the "QA capabilities as installable skills + MCP for any coding agent" packaging. Validates your skills-ecosystem bet; a reference for how to expose pieces externally.

Strategic options

  • A. Ideas only (lowest risk, recommended): treat agent-qa as an architecture reference. Adopt runtime-heal-before-fail, a memory trust model, and a plan cache inside your existing stack. No license/vendor exposure.
  • B. Complementary pilot: use agent-qa as a self-healing smoke/exploratory runner for cross-browser or (net-new) mobile coverage, with your orchestrators triggering it and your Zephyr/Jira layer recording results. Keep deterministic Playwright for regulated regression + SQL/GraphQL verification.
  • C. Not advised: replacing deterministic IMA 360 regression with NL agent runs — you'd trade auditability, precision, and DB-level verification for resilience you can get more cheaply via option A.

Risks & due diligence

  • License: FSL-1.1-ALv2 (Functional Source License), not OSI open source. Permits internal use; prohibits "Competing Use," converts to Apache-2.0 two years after each release. Fine for internal QA; legal should confirm before any product-adjacent use.
  • Maturity: ~5 weeks old, single-author copyright, fast-moving — expect churn and breaking changes.
  • Cost & non-determinism: LLM-driven runs cost tokens and vary run-to-run; the action cache helps but a large regression suite on agent execution is pricier and less repeatable than scripted Playwright.
  • Telemetry & sandbox: PostHog analytics and Docker-sandboxed hooks — confirm opt-out/data-handling and container posture against IGS policy.
