Verdict — the three things to fix first in this slice

The type files are small, readable, and correct in shape — but they are a claim, not a check. They are hand-maintained casts (executeQuery<Party>(...)) with nothing tying them to the SQL or the schema, in a repo whose own knowledge base records 32+ column-name drifts. Three issues stand out:

  1. [Medium] A type declares fields the query never returns. OpportunityForUpdate declares two BrokerOrganizationGroupId fields that neither backing query selects — so consuming code reads undefined while the compiler reports a number | null. The type actively lies.
  2. [Medium] No generation — drift against the schema is invisible. Every interface is hand-written to mirror a SELECT; a column rename (the repo’s most-documented failure class) produces no type error, just a runtime undefined. The cast trusts a shape nothing verifies.
  3. [Medium] Type coverage is partial and scattered. Four files cover party/account/opportunity/contract, but many finders return anonymous inline shapes, two interfaces live in the facade, and enum + Industrial results have no type file at all. There is no single home for “the SQL layer’s types.”

Net: the cheapest high-value win is fixing the one type-lie and adding a generation or assertion step so the cast stops being an act of faith.

Phase 1 — Surface Map

Public surface & change-radius

  • 4 type files — sql.types.account.ts (11 interfaces), sql.types.opportunity.ts (12), sql.types.party.ts (9), sql.types.contract.ts (3). All plain export interface, no generics, no imports.
  • Consumed by the facade as cast targets: executeQuery<Party>(...) (sql.db.ts:61) and ~100 siblings. The compiler trusts the cast; there is no runtime validation.
  • Not the whole picture — sql.db.ts also declares BulkTransferBatchItem (:198–205) and ValidUlob (:666–674) inline, and dozens of finders return anonymous { ... } shapes; queries.enum.ts and queries.industrial.ts results have no type file.

Change-radius: low and compile-time — a type change can’t break runtime, only the build. But the absence of a check means a schema change reaches runtime undetected (F1).

Dimension 1 — Architecture, Coupling & Boundaries

MEDIUM Type coverage is partial and has no single home — 4 files + inline interfaces + anonymous shapes

Evidence: src/types/sql/*.ts (4 files) · sql.db.ts:198–205 (BulkTransferBatchItem), 666–674 (ValidUlob), 357/495/545/566/578 (anonymous returns); no type file for enum/industrial results

const getOpportunityByUtilityCode = async (utilityCode): Promise<{ OpportunityId: number; PartyId: number }> => ...  // sql.db.ts:357 anon
const getTaskById = async (taskId): Promise<{ TaskId: number; TaskStatusId: number; OrganizationAgentId: number }> => ... // :495 anon

interface ValidUlob { ... }                       // sql.db.ts:666–674 — also in the facade

Why it matters: “The SQL types” is not one thing. The four src/types/sql/ files cover the party/account/opportunity/contract result shapes, but a large fraction of finders return anonymous inline object types (e.g. :357, :495, the SUT/audit shapes at :545/:566/:578), two named interfaces live in the facade itself (BulkTransferBatchItem, ValidUlob), and there is no type file for the enum reference data or for any queries.industrial.ts result. A maintainer asked “where are the SQL result types?” gets four files, a couple of facade declarations, and a scattering of inline literals. That scatter is what makes drift (F1) hard to police and is compounded by the physical split — the types sit in src/types/sql/ while the layer they describe sits in src/utils/sql/.

Severity rationale: Medium — no single source for the layer’s contract; makes both review and any generation step harder.

Remediation sketch: Consolidate: promote the inline/anonymous shapes (including BulkTransferBatchItem, ValidUlob, enum and Industrial results) into the types/sql/ set, and decide one home (co-locate with the finders during the TDD-015 F6 split, or keep types/sql/ as the single barrel).

Dimension 5 — Tech Debt & Maintainability

MEDIUM OpportunityForUpdate declares two fields the backing queries never return — a type-lie

Evidence: src/types/sql/sql.types.opportunity.ts:84–90 vs queries.opportunity.ts:662–681 and 685–708

// the TYPE claims broker-group fields:
export interface OpportunityForUpdate extends Opportunity {
  LineOfBusinessId: number; UtilityId: number; OrganizationAgentId: number;
  BrokerOrganizationGroupId1: number | null;   // sql.types.opportunity.ts:88
  BrokerOrganizationGroupId2: number | null;   // :89
}
// the QUERY selects neither — only OpportunityId/PartyId/PartyNumber/LineOfBusinessId/UtilityId/OrganizationAgentId:
SELECT TOP 100 o.OpportunityId, o.PartyId, p.PartyNumber, ldc.LineOfBusinessId, ldc.UtilityId,
       ldc.SalesRepresentativeOrganizationAgentId AS OrganizationAgentId
FROM ... // queries.opportunity.ts:662–681 (getReadyToReviewOpportunityForBrokerUpdate)

Why it matters: getReadyToReviewOpportunityForBrokerUpdate and getOpportunityWithBrokerForUpdate are typed as OpportunityForUpdate, which declares BrokerOrganizationGroupId1/2 as number | null. Neither query selects those columns. So any consumer that reads opp.BrokerOrganizationGroupId1 gets undefined at runtime while the compiler cheerfully reports number | null and offers no warning. This is the most concrete failure of a hand-written type contract: the type doesn’t just fail to catch drift (F1) — it asserts a field exists that the data never carries, turning the type system from a safety net into a source of false confidence.

Severity rationale: Medium — latent correctness; a consumer relying on those fields silently gets undefined, and the type hides it.

Remediation sketch: Either drop BrokerOrganizationGroupId1/2 from OpportunityForUpdate (the queries don’t provide them; brokers are supplied separately via getNonOrganicBrokerOrganizationGroupId), or add them to both SELECTs if a consumer genuinely needs them. Make the type match the query.

LOW Redundant field re-declaration in an extends interface

Evidence: src/types/sql/sql.types.opportunity.ts:43–47

export interface OpportunityWithResetableFields extends Opportunity {
  OpportunityId: number;  // already in Opportunity
  PartyId: number;        // already in Opportunity
  PartyNumber: string;    // already in Opportunity — all three are no-ops
}

Why it matters: OpportunityWithResetableFields extends Opportunity then re-declares the three fields it already inherits — a no-op that signals copy-paste and adds nothing the parent doesn’t. It also means the interface adds zero fields over Opportunity, so the distinct name implies a distinction that doesn’t exist in the type.

Severity rationale: Low — cosmetic; mild confusion only.

Remediation sketch: Drop the re-declared fields; if the interface adds nothing, alias it (type OpportunityWithResetableFields = Opportunity) or remove it.

LOW Loose optionals weaken the contract where the query always supplies the column

Evidence: src/types/sql/sql.types.contract.ts:1–14 vs queries.contract.ts:2–13

export interface Contract { ImaContractId: number; PartyId: number; PartyNumber: string;
  PartyName?: string; DealType?: string; }   // optional...
// ...but getRandomContract always SELECTs both p.PartyName and c.DealType (queries.contract.ts:7–8)

Why it matters: Contract.PartyName? and DealType? are optional, and ContractParty.PartyName?/ContractCount? likewise — yet the backing queries always select those columns. Optional-by-default types don’t catch a genuinely missing column (they just widen to undefined), so they trade away the one thing the type could do for you. It’s the inverse problem to F2: F2 over-claims, this under-claims.

Severity rationale: Low — weakens the contract without breaking it.

Remediation sketch: Mark a field optional only when a query genuinely omits it; otherwise make it required so a dropped column surfaces as a type error.

LOW Field-for-field duplicate interfaces

Evidence: src/types/sql/sql.types.account.ts:9–21 (ActiveAccount) vs 28–40 (InactiveAccount)

export interface ActiveAccount   { AccountId; AccountPartyId; UtilityLineOfBusinessId; LobName;
  UtilityName; UtilityCode; AccountNumber; AddressLine1; City; State; ZipCode; }
export interface InactiveAccount { /* the same 11 fields, identical */ }   // account types :28–40

Why it matters: ActiveAccount and InactiveAccount are identical (11 fields each), and each then has a *WithOpportunity extends. The active/inactive distinction is a runtime filter (LDCAccountStatusId), not a shape difference, so two identical interfaces is duplication that will drift independently.

Severity rationale: Low — duplication; no functional impact.

Remediation sketch: One shared LdcAccountRecord with both *WithOpportunity extending it (the comment at account:8 already notes they intentionally share a shape).

Dimension 2 — Test Coverage, Quality & Gaps

MEDIUM No generation or runtime assertion — the cast is an act of faith against a drift-prone schema

Evidence: all 4 files are hand-written interfaces; cast at sql.db.ts:61 (executeQuery<Party>) with no validation; contrast the GraphQL layer’s npm run codegen-int

const records = await sql.executeQuery<Party>(queriesParty.getCiParties());  // the <Party> cast is unchecked at runtime

Why it matters: Every type here is hand-authored to mirror a SELECT list, and executeQuery<T> is a blind cast — mssql returns whatever columns the query produced and TypeScript labels them T without checking. The repo’s own knowledge base records 32+ confirmed column-name drift findings (schema-constants.md: Address1→AddressLine1, StateCode→State, …). When such a rename lands, the SQL returns a differently-named column, the cast still “succeeds,” and the field reads undefined at runtime — no type error. Notably, the GraphQL side of this same suite is generated (npm run codegen-int → src/generated/graphql.ts), so the project already believes in generated types; the SQL layer is the un-generated exception, and it’s the layer pointed at the database most prone to drift. The SQL sanity spec (TDD-017) catches “zero rows,” not “wrong/renamed column,” so it does not close this gap.

Severity rationale: Medium — silent drift on the layer least protected against it; not High because the sanity spec catches the grossest breakages and the blast radius is test-trust, not production.

Remediation sketch: Add a lightweight guard — either generate the result interfaces from the query column lists, or add a runtime shape assertion in executeQuery in dev/CI (assert the recordset’s keys match the expected interface for a sampled row) so a renamed/dropped column fails loudly instead of reading undefined.

Dimensions 3, 4, 6, 7 — Security · Error Handling · Dependencies · Performance

No material findings — compile-time-only surface

These are pure export interface declarations: no runtime behavior, no untrusted input, no secrets, no control flow, no imports, no allocations. Security (3): no surface. Error handling (4): none to assess; runtime failure modes are the facade’s (TDD-015 F4). The only cross-cutting note is F1’s: the absence of generation is the real risk here, and it sits in coverage (Dim 2), not these four.
