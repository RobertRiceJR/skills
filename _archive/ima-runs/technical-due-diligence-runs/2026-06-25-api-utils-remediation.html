Verdict — should we make these changes?

Partly. The waiter change already solved the headline problem — so trim the remediation rather than ship all of it. The decisive fact: all 37 apiUtils call sites receive a response returned by waiters.waitForResponseSuccess(), and that function now asserts top-level errors[] is empty before the helper ever runs. The vacuous-pass-on-GraphQL-error gap the sweep flagged is therefore already closed upstream for every real caller. That reframes all five proposals:

  1. De-scope #1 (parseBody errors[] guard). Re-asserting top-level errors[] inside api.utils duplicates the waiter — a second source of truth that will drift. Its only non-redundant addition (assert data non-null) is itself transitively covered: GraphQL returns data:null ⇔ it populates errors[], which the waiter already catches.
  2. GO on #3 (logging) and #5 (doc) now — the clean wins. The full-payload logger.info(JSON.stringify(opportunities)) dumps (4 sites) are noise + customer data in logs and a behavior-free fix. The doc note (#5) is what actually prevents the #1 drift by codifying the waiter-vs-verifyNoErrors layering.
  3. #2 is worth doing only as a consistency cleanup — and only one way. Collapsing the three inconsistent null-data modes is real maintainability value, but it addresses a hole the waiter mostly closed; implement it as data-present-only, never non-empty, or it will newly fail tests that legitimately assert an empty filtered grid.

Recommended subset: ship #3 + #5 now; do #2 as a scoped consistency pass (data-present only); drop #1's errors[] re-check; fold #4 into #2 or skip it.

Phase 1 — Surface Map (the fact that drives the verdict)

Every apiUtils helper runs on a waiter-verified response

Pattern confirmed at opportunities.grid.e2e.spec.ts:32-36, 47-52; opportunity.edit.e2e.spec.ts:30-35; party.record.overview.e2e.spec.ts:85-88; summary.panel.activity.e2e.spec.ts:142-148

const promise  = waiters.getResponsePromise(page, API_ENDPOINTS.GET_OPPORTUNITIES);
await action();
const response = await waiters.waitForResponseSuccess(promise);   // ← asserts 2xx + no top-level errors[]
await apiUtils.assertAllOpportunitiesMatchCode(response, ...);        // ← receives the verified response

Consequence: by the time any apiUtils.assert* body runs, the response has already passed the waiter's top-level errors[] check. A GraphQL resolver error (200 + {data:null, errors:[…]}) now throws in waitForResponseSuccess — the helper never executes. The vacuous-pass gap the sweep flagged is closed upstream for all 37 sites. What remains for api.utils to defend against is narrow: a non-waiter-fed response (none today), or data partially-null without a top-level error (near-impossible per GraphQL spec).

Change-radius of editing api.utils: 37 calls across 8 spec files — modest, and all in one assert-after-filter idiom. A signature-preserving internal refactor (parse dedup, logging level) is low-blast; a behavioral assertion change (forcing non-empty, or re-throwing on shapes the waiter passed) reaches all 37.

Per-Change Evaluation

MEDIUMDE-SCOPE #1 — parseBody guard that re-asserts no top-level errors[]

Evidence: waiters.ts:54-60 (already does this) vs. proposed new guard in api.utils.ts

// waiters.ts:54-60 — ALREADY runs on every apiUtils-bound response:
if ((response.headers()['content-type'] ?? '').includes('json')) {
  const body = (await response.json().catch(() => ({}))) as { errors?: unknown[] };
  if (Array.isArray(body.errors) && body.errors.length > 0) { /* throws */ }
}

Soundness: the check itself is correct — but it is the same check the waiter now performs, on the same responses. Adding it to api.utils creates two copies of "no top-level errors[]" that must be kept in agreement (the exact duplicated-thing drift the repo catalogs as bug-patterns #19).

Blast radius / regression: low risk of breakage (it would just re-confirm what the waiter confirmed), but it is wasted work + a drift liability. The one thing it adds that the waiter doesn't — assert data is non-null — is transitively covered: a spec-compliant GraphQL server returns data:null only alongside a populated errors[], which the waiter already trips on.

Severity rationale: Medium — not dangerous, but it's effort spent re-solving a solved problem and seeds a second source of truth. The correct mitigation is #5 (document the contract), not #1.

Go/no-go: DE-SCOPE the errors[] re-check. If anything survives from #1, let it be a thin data-present assertion folded into #2 — not a duplicate error envelope check.

MEDIUMMODIFY #2 — collapse the 3 inconsistent null-data failure modes

Evidence: api.utils.ts:10 (crash), 168-169 / 174-175 (undefined-access), 55/97/136/155/199 (vacuous ?? [])

// three different reactions to a null/odd data shape, same root cause:
const noteActivities = noteBody.data.party.notesById;                 // :10  crash (no ?.)
const row = body?.data?.updateOpportunityFlag?.opportunityBrowserRow; // :168 then row.x → undefined-access
const opportunities = body?.data?.opportunities?.opportunitiesNew ?? []; // :55  vacuous pass

Soundness: real maintainability value — one predictable failure beats three. But note what live risk it closes: with the waiter in front, the null-data scenario these guard against is mostly already prevented (data:null ⇒ errors[] ⇒ waiter threw). So this is a consistency cleanup, not a bug fix — frame it that way so it isn't over-sold.

Blast radius / regression — the trap: the fix is safe only if the guard asserts data (and the addressed sub-field) is present, never that a collection is non-empty. Several tests legitimately exercise filters that can return zero rows; assertHasResults (:147-151) is the dedicated non-empty check. If #2 is implemented as "throw on empty array," it converts those into false REDs across the 37 sites. Implement as presence-only.

Severity rationale: Medium — the regression risk is real but entirely avoidable with the presence-only scoping; value is maintainability, not hole-closing.

Go/no-go: MODIFY & proceed — adopt as a presence-only guard (data + named sub-path defined), applied uniformly so :10/:168/:174 stop crashing and :55-class stop passing vacuously. This is where a small shared helper earns its place.

LOWGO #3 — logger.info full-payload dumps → debug + count

Evidence: api.utils.ts:56, 98, 137, 156

logger.info(`Opportunities - ${JSON.stringify(opportunities)}`);  // full records, at info, every grid assert

Soundness: unambiguous. This dumps entire opportunity records — names, IDs, financial fields — to the console at the default info level on every grid assertion. It is the concrete instance of waiters-finding F9 (info spam) and F3 (customer data in logs); the waiter only theorized it, this ships it. Logging-only — no behavioral effect on tests.

Blast radius / regression: none. Pure observability change.

Severity rationale: Low as a current issue, but the change is the highest value-to-risk of the five.

Go/no-go: GO now. Drop to logger.debug and log a count (e.g. `${opportunities.length} opportunities`), not the serialized payload. Mirror the [waiter]-style tag for consistency.

LOWFOLD-IN #4 — dedup the repeated const body = await response.json()

Evidence: api.utils.ts — repeated at :9, :42, :48, :54, :96, :135, :148, :154, :167, :173, :179, :190, :198 (~13 helpers)

Soundness: the repetition is real, but extracting a parseBody<T>() purely to remove the one-liner is modest value and churns ~13 functions. Its worth depends entirely on whether it carries the #2 presence-guard: a parseBody that parses and asserts data present is the natural home for #2; a parseBody that only saves a .json() call is cosmetic.

Blast radius / regression: low if signature-preserving (helpers keep their (response, …) signatures, call parseBody internally). Double-parsing is a non-issue — Playwright caches the body.

Severity rationale: Low — pure refactor; value contingent on #2.

Go/no-go: FOLD INTO #2 (make parseBody the presence-guard's home) or skip — don't do it as standalone cosmetic churn.

LOWGO #5 — doc note: verifyNoErrors (recursive, opt-in) vs the waiter (top-level, automatic)

Evidence: api.utils.ts:21-45 (collectErrors/verifyNoErrors); waiters.ts:54-60 (top-level check); callers opportunity.edit:35, party.record.relationships:31

Soundness: this is the change that prevents #1. There are now three error-checking layers and their division of labor is implicit: the waiter catches top-level errors[] automatically on every wait; verifyNoErrors/collectErrors is the recursive, opt-in deep check (catches nested business errors too) used where a fully-clean response is required; assertAccountsFailedToLinkIsEmpty (:197-201) checks the specific nested business-error channel. Writing that down stops a future contributor from "fixing" the apparent gap by re-adding a top-level check (i.e., re-introducing #1).

Blast radius / regression: none — documentation.

Severity rationale: Low effort, but high leverage against drift.

Go/no-go: GO now. A short header comment in api.utils.ts (and one line in the waiter's JSDoc pointing at verifyNoErrors for the deep check) closes the loop.
