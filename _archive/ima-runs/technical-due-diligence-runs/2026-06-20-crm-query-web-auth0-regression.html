Verdict — the three things that should most concern you

1. The data gateway's authentication fails open. Igs.Crm.Query.Web derives the entire user identity from a client-supplied IgsUserId header with no token validation (AuthenticateCurrentUserMiddleware.cs:64, HttpContextExtensions.cs:137-154), and serves it behind CORS that reflects any origin with AllowCredentials() (Startup.cs:106-112). The only thing standing between an attacker and full user impersonation + credentialed cross-origin access to every query/mutation (including deleteParty/mergeParties) is an upstream network control that is not enforced in this code. (C1, C2)

2. Five of the seven "Auth0" bugs are not Auth0. git-blame dates the upload-breaking body-read middleware to 2019, the getCurrentUserTeam crash to the HotChocolate-15 migration (2026-04), and the preset 400 to a HotChocolate input-generation contract defect. Only Genesys and Graph calendar plausibly relate to Auth0 — and only through the shared header→roles/username identity layer. Fixing Auth0 will not fix most of these; they were latent and the regression pass merely surfaced them. All seven are untested at every layer.

3. Controller failures are invisible in production. The global MVC exception filter returns 500s without TrackException (ApplicationExceptionFilterAttribute.cs:10-22) — which is exactly why the document-upload 500 (CI-7844) logged no exception. Combined with several mutations that catch and return a success-shaped default (DeleteParty, UpdateCommissionableStartDates), real write-path failures vanish.

Security & Trust Boundaries

CRITICAL C1 — Identity is a client-supplied header; no token validation on the gateway (impersonation if reachable)

AuthenticateCurrentUserMiddleware.cs:64 · HttpContextExtensions.cs:137-154 · Startup.cs:102

var windowsUserName = httpContext.GetIgsUserIdHeaderValue();   // IgsUserId / LegacyUserId header
// ...no JWT/cookie validated here; roles looked up from this string -> ClaimsPrincipal

Why it matters: The OIDC/Auth0/JwtBearer config lives only in the sibling DMZ app Igs.Crm.Web. On this gateway, identity is whatever arrives in the header. Anyone who can reach ima-query.*/graphql and set IgsUserId: <victim> becomes that user — including the roles that gate authorization. The header is also re-attached to downstream UserManagement/WebApi calls, propagating the spoofed identity.

Reach / severity: Runs on every request; feeds every resolver. App-layer auth fails open — the sole control is network isolation + the upstream gateway stripping/setting the header, which is not visible or enforced here. Critical (defense-in-depth absent on the data tier).

Remediation: Validate the Auth0 JWT on this service (bind the JwtBearer scheme already configured in Igs.Crm.Web) and derive identity from a verified claim; treat IgsUserId as an internal hop set only after the proxy strips any inbound copy.

CRITICAL C2 — CORS reflects any Origin with AllowCredentials() (spec-forbidden; browser-exploitable)

Startup.cs:106-112

builder.AllowAnyMethod().AllowAnyHeader()
       .SetIsOriginAllowed(origin => true).AllowCredentials();

Why it matters: Reflecting the caller's Origin while emitting Access-Control-Allow-Credentials: true is the one combination the CORS spec forbids — it defeats same-origin policy. Any site a logged-in user visits can make credentialed GraphQL calls to the gateway and read the responses (CSRF-with-readback over all queries/mutations). Compounds C1.

Reach / severity: Global, covers MapGraphQL + MapControllers. Critical — no attacker infra beyond a web page.

Remediation: Replace SetIsOriginAllowed(_ => true) with an explicit WithOrigins(...) allow-list when credentials are required.

HIGH H6 — Destructive mutations have no @authorize policy (authz is opt-in)

queryWeb.graphql:2080,2096,2114 — deleteParty / mergeParties / transferOpportunities

Why it matters: deleteParty (no directive at all), mergeParties, transferOpportunities are irreversible/data-moving and carry no schema-level @authorize, while peers (updateCommissionableStartDates, email-template mutations) do. The mechanism exists (RoleAuthorizationHandler.cs) — it's just not applied, so every new destructive mutation is unprotected by default. Elevates C1/C2 from "impersonate/read" to "destroy/merge."

Remediation: Add @authorize(policy: ...) to every state-changing mutation; add a schema-build test that fails CI if a Mutation field lacks one.

HIGH H7 — Genesys OAuth access + refresh tokens stored in the DB in plaintext

OrganizationAgentAuthorizationTokenConfiguration.cs:14-27 · GenesysService.cs:479-484

builder.Property(x => x.AccessToken)...IsUnicode(false);   // varchar, no converter
builder.Property(x => x.RefreshToken)...IsUnicode(false);  // 31-day refresh token

Why it matters: Live Genesys/PureCloud bearer + 31-day refresh tokens for every agent stored unencrypted. Anyone with read access to that table (DB cred leak, SQLi elsewhere, a backup, a read-only analyst/MCP connection) obtains usable agent tokens to place/record calls as those agents.

Remediation: Encrypt the columns (EF value converter backed by Data Protection / Always Encrypted / KMS) or store in a secret store keyed by agent.

MEDIUM M5 — CI-7847: Genesys "not authorized" branch returns IsAuthorized = true (inverted contract)

GenesysService.cs:191-201

Why it matters: The role gate correctly fails closed (no call placed when the user lacks Role.Genesys), but the denial response sets IsAuthorized = true while the token-missing branch sets it false. A client reading that flag misreads a role denial as success-with-message. The functional block is fine; the signal is wrong. Identity comes from the header (C1), so a spoofed Genesys-roled username passes this gate.

Error Handling & Failure Modes

HIGH H2 — Global MVC exception filter returns 500 without TrackException (controller failures vanish from App Insights)

ApplicationExceptionFilterAttribute.cs:10-22 (registered Startup.cs:69)

context.Result = new ContentResult { StatusCode = DetermineStatusCode(exception),
    ContentType = "text/plain", Content = exception.Message };  // no TrackException

Why it matters: Catch-all for every controller (Document, Account, ContractTransfer, …). Unhandled exceptions become a status code + plaintext and the request completes — never sent to App Insights (the GraphQL path does TrackException). This is the direct mechanism behind CI-7844 "the document 500 logs no exception": the multipart IOException bubbles here, becomes a 500, and is dropped. Operators are blind to a whole class of controller failures.

Remediation: TrackException in OnException (gated to 5xx) before building the ContentResult.

HIGH H1 — CI-7844/45: RequestLoggingMiddleware consumes the request body before the multipart handlers

RequestLoggingMiddleware.cs:28-44 (Startup.cs:101) → DocumentController.cs:103, AccountController.cs:122, MultipartFormDataHelper.cs:23

request.EnableBuffering();
var buffer = new byte[Convert.ToInt32(request.ContentLength)];
await request.Body.ReadAsync(buffer, 0, buffer.Length);   // drains the body
request.Body.Seek(0, SeekOrigin.Begin);                    // rewind attempt

Why it matters: Fine for JSON, but for multipart/form-data the read/rewind interacts with the form reader to throw IOException "Unexpected end of Stream, the content may have already been read by another component" at MultipartReaderStream.ReadAsync — the live INT signature. Every upload endpoint (documents, LOA, account/contract templates, bulk import) routes through this middleware first, so uploads break; with H2 the failure surfaces as a 500 with no logged exception.

Auth0 verdict: Coincidental — git blame: the body-read line dates to f7694df44a (Jason Bray, 2019-02-04), ~7 years before Auth0. Latent legacy bug, not introduced by the upgrade.

Remediation: Skip body capture for multipart content types (or log after _next from the already-buffered stream) in ReadRequestAsString.

Performance Hotspots

HIGH H11 — RequestLoggingMiddleware buffers the full request AND response in memory on every request

RequestLoggingMiddleware.cs:28-44 (request byte[]) · :67-74 (response MemoryStream)

var buffer = new byte[Convert.ToInt32(request.ContentLength)];   // 20 MB upload -> 20 MB LOH alloc
... Encoding.UTF8.GetString(buffer) ...                          // + a 2nd large alloc, then discarded

Why it matters: For a 20 MB upload (cap: DocumentController.cs:30-31) this allocates a ~20 MB LOH array + a full decoded string per request, all garbage; responses are buffered into a growable MemoryStream and the untruncated response is queued to RabbitMQ (the request is truncated to 8000 chars; the response is not). N concurrent uploads = N×~40 MB LOH churn → Gen2/LOH GC pressure. Disables response streaming for everyone.

Remediation: skip capture for multipart/large content; cap captured text via ArrayPool; only swap Response.Body when logging is needed; truncate the queued response like the request.
