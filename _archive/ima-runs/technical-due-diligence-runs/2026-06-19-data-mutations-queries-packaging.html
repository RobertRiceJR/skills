Technical Due Diligence · QE · TDD-004

GraphQL operation packaging — src/data/mutations + src/data/queries

Pricing the architectural risk in the test suite's GraphQL operation library as it grows. The library is structurally sound at the unit level (every module is a static, parameterized gql document) but the packaging has already begun to fail silently — a documented "load-bearing" duplicate is dead, two mutations live in the queries folder, and a shared fragment is copy-pasted three times.

Verdict — the three findings that most change the price of inheriting this code
  1. The duplicate that's supposed to be load-bearing is actually dead — and the live copy is a trap. upsertPartyAccountsMutation is exported from both party.mutations.ts and account.mutations.ts. A 24-line comment justifies keeping the party.mutations.ts copy "because three existing consumers depend on it." All five live consumers import the other copy; the documented one has zero importers. The strategy of "keep both, document why" failed silently — the justification rotted and nobody noticed. Worse, the dead copy is the one that trips Hot Chocolate HC0018 and collapses the whole response to data:null, one careless re-import away. → A2
  2. Two mutations live in the queries/ folder, so the directory name is no longer trustworthy. mergeParties and mergeOpportunities are GraphQL mutation operations imported via from 'queries/mergeParties'. As this set "grows fast," every reader has to open the file to learn whether a folder's contents match its name. → A1
  3. The shared ReferenceFields fragment is copy-pasted in three query files with no shared-fragment module. Byte-identical today; the drift surface grows linearly with exactly the thing you flagged as growing fast. → M1

Bottom line: nothing here loses production data — these are test fixtures. But the packaging has crossed the threshold where the folder layout actively misinforms readers and silently accumulates dead/duplicated definitions. Fix the four cheap correctness items now (A1, A2, M1, M2); then decide on the structural target (see Recommended Packaging).

Recommended Packaging (the ask)

TARGET Feature-first co-location with a shared-fragment module

Your own repo standard is feature-first (CLAUDE.md: "Feature-first: one folder per domain under src/tests/"). src/data/ is the lone layer-first island left in the tree, and that split is the root cause of A1, A4, and A5 — a merge operation has no obvious home when "queries" and "mutations" are the only two boxes, so it lands wherever the author guessed. Co-locating by domain removes the guess.

// TARGET — domain-first, layer rides the filename suffix (mirrors src/tests/)
src/data/
  party/        party.queries.ts   party.mutations.ts
  account/      account.queries.ts account.mutations.ts
  opportunity/  opportunity.queries.ts opportunity.mutations.ts
  contract/ · communications/ · dashboard/ · …
  _shared/
    fragments.ts        // ReferenceFields, SearchPartyFields, CurrentUserFields …
    reference-data.ts   // the 373-line getReferenceData god query, isolated

Conventions to lock in the same pass: one export-naming rule (<verb><Noun>Mutation / <query> — kill the bare mergeParties exports that force as …Mutation re-aliasing); one file-naming rule (<domain>.<layer>.ts, no snake_case); add a data/*-only alias policy (drop the redundant queries/* alias so there is exactly one way to spell each import).

Migration is mechanical but wide: 40 consumer files / 71 imports reference these modules. Do it as a codemod (move file → update barrel/alias → rewrite imports) in one branch, lean on npm run type-check as the gate. Stage it: (1) land the four correctness fixes now under the existing layout (zero structural risk); (2) introduce _shared/fragments.ts; (3) move domain-by-domain behind the codemod.

ALT Keep layer-first, just stop the leaks (lower effort, doesn't resolve the contradiction)

If a full move is out of budget: add a mutations/* alias to match queries/*, move the two misfiled merge mutations into mutations/, extract _fragments.ts, delete the dead duplicate, standardize names. Cheaper, but leaves src/data/ as a layer-first island disagreeing with src/tests/ — the same "where does a merge go?" ambiguity returns with the next cross-cutting operation.

Phase 1 — Surface Map

MAP What's actually here, and what depends on it

  • Module shape: every one of the 37 files is the same shape — import { gql } from 'graphql-tag' then one or more export const X = gql`…` template literals. No logic, no interpolation, no runtime branching. They are static GraphQL document definitions consumed by API specs ({ gql } fixture) and the data factories.
  • Entry points (consumers): 40 files import from these two folders — API specs under src/tests/**, the src/api/*.api.ts page-API wrappers, and src/dataFactory/*. 71 import statements total.
  • Highest-blast-radius modules: reference_data.ts (getReferenceData, imported by many specs for setup) and party.mutations.ts (15 exports — the de-facto grab-bag). A change to either radiates widely.
  • Alias surface: tsconfig.json defines queries/* → ./src/data/queries/* and data/* → ./src/data/*. Mutations have no dedicated alias. Net: queries are reachable by two spellings, mutations by one — the asymmetry behind A5.
  • No barrel: no index.ts in either folder; consumers reach individual files directly (explicit, tree-shakeable — fine; just means there's no curated public surface).

Dimension 1 — Architecture, Coupling & Boundaries

HIGH A2 · Duplicate upsertPartyAccountsMutation — the "load-bearing" copy is dead; the live copy is an HC0018 trap

Evidence: src/data/mutations/party.mutations.ts:43-66 (duplicate + justifying comment) vs src/data/mutations/account.mutations.ts:11-24

// party.mutations.ts:43  — the comment claims this copy is load-bearing:
// "Both copies are kept because three existing consumers (ERCOT factory + 2
//  API tests) already depend on reading `errors` on the upsertedAccounts side here."
export const upsertPartyAccountsMutation = gql`
  mutation upsertPartyAccounts($input: UpsertPartyAccountsInput!) {
    upsertPartyAccounts(input: $input) {
      upsertedAccounts { accountId requestId errors }   // reads errors[]
      ...

Reach (traced — all 5 live importers use the OTHER copy):

account.api.ts:8                          → account.mutations
dataFactory.ercot.account.ts:8           → account.mutations   // the "ERCOT factory" the comment cites
accounts.duplicate-validation.api.spec:4 → account.mutations
accounts.mutations.api.spec:7            → account.mutations
api.mixed.api.spec:9                      → account.mutations
party.mutations.ts copy                   → 0 importers

Why it matters: the comment is now factually false — the "ERCOT factory" it names imports from account.mutations, not here. The dead copy is the one whose upsertedAccounts[].errors selection trips Hot Chocolate HC0018 ("Cannot return null for non-nullable field") on success paths, collapsing the entire response to data:null. The next person who greps upsertPartyAccountsMutation, takes the first hit, and imports it gets a test that silently asserts against data:null. This is the packaging risk made concrete: a duplicate kept "for safety" became a dormant landmine.

Severity rationale: High — silently rots a test (false pass/fail) and the protective comment actively misleads. No money/auth multiplier (test fixture).

Remediation: delete party.mutations.ts:43-66; keep the single account.mutations.ts definition. If a consumer ever needs the errors selection, add it there.

HIGH A1 · Two mutation operations live in the queries/ folder — the directory name lies

Evidence: src/data/queries/mergeParties.ts:3-12, src/data/queries/mergeOpportunities.ts:3-12

export const mergeParties = gql`
  mutation mergeParties($input: MergePartiesInput!) { mergeParties(input: $input) { … } }`;

export const mergeOpportunities = gql`
  mutation mergeOpportunities($input: MergeOpportunitiesInput!) { … }`;

Reach: party-merge.api.spec.ts:5 → 'data/queries/mergeParties'; party.merge.e2e.spec.ts:9, party.merge.inheritance.e2e.spec.ts:9, party.record.myaccount.e2e.spec.ts:9 → 'queries/mergeParties'; opportunities.merge.e2e.spec.ts:8 → 'queries/mergeOpportunities'. Note the sibling mergeAccounts mutation correctly lives in mutations/ — so the three merge mutations are split across both folders with no rule.

Why it matters: the folder split is the suite's one signal for "is this a read or a write?" Two writes hiding in queries/ means import … from 'queries/mergeParties' reads as a query but mutates data. As the set grows, readers can't trust the folder; they must open each file. (mergePartiesPreview.ts is correctly a query — so the folder isn't wrong, just inconsistently populated.)

Severity rationale: High — boundary integrity is the primary value of the layer-first split; two leaks negate it and the count will grow.

Remediation: move mergeParties.ts + mergeOpportunities.ts into mutations/ (or, preferred, into their domain folder per the recommendation); rename exports to …Mutation (see M2) and update the 5 import sites.

MEDIUM A3 · Layer-first island contradicts the repo's feature-first standard (root cause of A1/A4/A5)

Evidence: CLAUDE.md — "Feature-first: one folder per domain under src/tests/"; src/tests/parties/, src/tests/accounts/, … are domain folders. src/data/{mutations,queries}/ is the only layer-first split in the source tree.

Why it matters: the two organizing principles disagree, so a cross-cutting operation (a merge, a preview, a bulk action) has two equally-plausible homes and lands by author whim — exactly what produced A1. Every new operation re-litigates "queries or mutations?" instead of "which domain?".

Severity rationale: Medium — not breaking, but it's the structural reason the leaks keep appearing; compounds as the set grows.

Remediation: adopt feature-first co-location (see Recommended Packaging). This dissolves A1, A4, and A5 at once.

MEDIUM A4 · Cross-domain leak: party.mutations.ts is a grab-bag of non-party operations

Evidence: src/data/mutations/party.mutations.ts exports, among 15 mutations: insertOpportunityMutation (L19), makeContactPrimaryOnOpportunityMutation (L31), insertNoteMutation (L68), createCallOutcomeMutation (L96), updateSalesOrCollectionsNoteMutation (L79).

Reach: opportunity.mutations.api.spec.ts:5 and opportunities.api.spec.ts:17 import insertOpportunityMutation from 'data/mutations/party.mutations' — while a sibling opportunity.mutations.ts already exists and also feeds those specs.

Why it matters: opportunity, note, and call-outcome writes have wandered into the party file, so an opportunity test imports its core mutation from party.mutations. Domain ownership is no longer inferable from the filename; a developer changing party mutations also owns opportunity/note/call-outcome surface they didn't expect.

Severity rationale: Medium — coupling that makes the highest-traffic file risky to touch; not currently broken.

Remediation: relocate insertOpportunity / makeContactPrimaryOnOpportunity to opportunity.mutations.ts; notes + call-outcomes to a communications/notes module. Update the ~4 import sites.

MEDIUM A5 · Alias asymmetry — the same module is imported via two different spellings

Evidence: tsconfig.json defines both queries/* and data/*; no mutations/*. So reference_data.ts is imported as both:

opportunities.api.spec.ts:9          import … from 'queries/reference_data'
accounts.mutations.api.spec.ts:10    import … from 'data/queries/reference_data'

Most acute in one file — party.merge.api.spec.ts imports the same export from the same file twice, via two alias spellings:

party.merge.api.spec.ts:5   import { mergeParties as mergePartiesMutation } from 'queries/mergeParties'
party.merge.api.spec.ts:10  import { mergeParties }                       from 'data/queries/mergeParties'

Why it matters: two spellings for one path means a grep / find-references / codemod over 'queries/…' silently misses every 'data/queries/…' site (and vice-versa). It also defeats import-dedup — hence the same export pulled twice above. This directly raises the cost of the very repackage you're contemplating.

Severity rationale: Medium — makes refactors error-prone (missed call sites) precisely when you want to refactor.

Remediation: pick one alias policy (recommend dropping queries/*, keep data/*), ESLint no-restricted-imports the other, codemod existing sites.

Dimension 2 — Test Coverage, Quality & Gaps

LOW C1 · No module-level schema-validity gate — a definition is only proven by a covering consumer

Why it matters: these modules are pure GraphQL documents; their correctness (valid against the deployed schema) is exercised only when a consuming spec runs and that spec's path is covered. An export that is mistyped but unimported — like A2's dead copy — sits indefinitely with no signal. npm run codegen-int validates operations against the schema, but it isn't a per-commit gate on this folder.

Severity rationale: Low — latent, not active; codegen catches it on the next regen.

Remediation: wire a lightweight "all gql documents parse + validate against the generated schema" check into knowledge:check/pre-commit; flag exported-but-unimported documents as dead.

No silent caps: a full dead-export sweep across all ~70 exports was not performed this run — only A2 was confirmed dead by exhaustive grep. Others may exist.

Dimension 3 — Security Surface & Trust Boundaries

PASS No material findings

Checked all 37 modules: each is a static gql template literal with no string interpolation — every variable is a declared GraphQL operation parameter ($input, $partyId, …), never spliced into the query text. There is no untrusted-input path through these files (the test suite, not the app, supplies the values; auth is handled by the fixture/Key Vault layer documented in CLAUDE.md, out of this scope). No injection, no secret handling, no SSRF surface in the operation library itself.

Dimension 4 — Error Handling & Failure Modes

HIGH (cross-ref A2) · The one real failure mode is packaging-induced: the dead duplicate's HC0018 null-collapse

Why it matters: selecting upsertedAccounts[].errors (the dead party.mutations.ts copy) makes the server return null for a non-nullable field on some success paths, which Hot Chocolate escalates to a top-level error and collapses the response to data:null. A spec importing that copy then asserts against null — a failure mode created entirely by the duplicate-and-document packaging choice. Full detail and remediation under A2.

Observation (informational, server-side): mergeAccounts.ts:3-7 documents that the payload echoes only the surviving ids and does not confirm the merge happened (verify via the deleted from-account + AcceptedAccountMerge row). That's a correct, valuable comment about CRM behavior — not a packaging defect, but worth keeping co-located with the operation wherever it moves.

Dimension 5 — Tech Debt & Maintainability

HIGH M1 · ReferenceFields fragment copy-pasted in three files — no shared-fragment module

Evidence: identical fragment ReferenceFields on ReferenceType { id code name __typename } at reference_data.ts:299, search.ts:82, getRecentSearches.ts:29.

fragment ReferenceFields on ReferenceType { id code name __typename }   // ×3, byte-identical

Why it matters: reference data is the most-reused shape in the schema; three hand-maintained copies will drift the moment one query needs an extra field (e.g. someone adds isEnabled to one copy). The drift surface scales with the number of queries — i.e. with exactly the growth you flagged. This is the single strongest argument for a _shared/fragments.ts module.

Severity rationale: High — silent divergence across the suite's most-used shape; grows with the codebase.

Remediation: extract ReferenceFields (and SearchPartyFields, CurrentUserFields, the *Browser*Fields family) into _shared/fragments.ts; compose via fragment import. Codegen already supports shared fragments.

MEDIUM M2 · Inconsistent export naming → consumers re-alias at the call site

Evidence: most exports carry a …Mutation suffix (createPartyMutation, validateAccountsMutation); the merge + bulk-template exports do not (mergeParties, mergeAccounts, mergeOpportunities, bulkUpdateOpportunitiesFromTemplate, createOpportunitiesTask). Consumers patch it locally:

party.merge.e2e.spec.ts:9        import { mergeParties as mergePartiesMutation } …
opportunities.merge.e2e.spec.ts:8  import { mergeOpportunities as mergeOpportunitiesMutation } …

Why it matters: the missing suffix is repaired by hand at every call site (as …Mutation) — duplicated effort, and a reader can't tell read-vs-write from the bare imported symbol. It's the export-naming twin of A1.

Severity rationale: Medium — recurring per-call-site tax; trivially preventable.

Remediation: one naming rule — mutations end in Mutation, queries are bare verbs. Rename the ~5 offenders, drop the now-redundant as aliases.

LOW M3 · File-naming conventions disagree within and across both folders

Evidence: queries/ mixes camelCase (getAccountById.ts), snake_case (reference_data.ts), bare nouns (search.ts, general.ts), and verb files (logPartySearch.ts). mutations/ mixes <domain>.mutations.ts, <domain>.<sub>.mutations.ts (7 opportunity.* files), and a bare mergeAccounts.ts with no suffix.

Why it matters: no predictable filename means no predictable import path — you can't autocomplete or guess where a thing lives, which slows every addition.

Severity rationale: Low — friction, not breakage.

Remediation: standardize on <domain>.<layer>.ts (kebab/lower-camel domain), no snake_case; fold the orphans (see M4).

LOW M4 · general.ts is a catch-all with no domain home

Evidence: src/data/queries/general.ts exports only getUser (currentUser + CurrentUserFields fragment).

Why it matters: "general" is where things go when nobody decides where they belong — it only grows and never shrinks. getUser has a clear home (a user/session module or _shared).

Severity rationale: Low — one small file today; a magnet for future orphans.

Remediation: rename to user.queries.ts (or move getUser to _shared); ban a generic "general/misc" bucket by convention.

LOW M5 · Opportunity mutations sprawl across 7 files with no sub-package grouping

Evidence: opportunity.activity, opportunity.bulk, opportunity.bulk-template, opportunity.flag, opportunity.transfer, opportunity.contact, opportunity.mutations — plus insertOpportunityMutation stranded in party.mutations.ts (A4). Several hold 1–3 exports (opportunity.mutations.ts = 1).

Why it matters: this is the "growing large and fast" pattern in miniature — split-per-subfeature with no folder to contain it, so the flat mutations/ namespace keeps widening. Two near-twins (opportunity.bulk vs opportunity.bulk-template) already invite confusion.

Severity rationale: Low now; this is the trajectory the whole review is about.

Remediation: under feature-first, these collapse into opportunity/opportunity.mutations.ts (+ optional opportunity.bulk.mutations.ts if a file genuinely warrants splitting) — the domain folder is the grouping the flat layout lacks.

Dimension 6 — Dependency Health & Version Risk

LOW D1 · Single runtime dependency (graphql-tag) in maintenance mode

Evidence: all 37 files import { gql } from 'graphql-tag' and nothing else.

Why it matters: graphql-tag is stable and ubiquitous but effectively feature-frozen; the modern path is the codegen-provided gql/typed-document-node. No vulnerability or version trap in the used surface today. (The repo's known graphql-codegen version trap lives in codegen config, not these files — out of scope here.)

Severity rationale: Low — no known impact; noted for the eventual typed-document-node migration.

Remediation: none required now; if/when you adopt TypedDocumentNode, this folder is the natural cut-over point.

Dimension 7 — Performance Hotspots

MEDIUM P1 · getReferenceData is a 373-line god query — every consumer over-fetches ~60 collections to use one

Evidence: src/data/queries/reference_data.ts:3-285 — a single getReferenceData query selecting ~60 reference collections (account types, statuses, utilities, LOBs, NAICS codes, task types, transfer reasons, aging buckets, …) plus 14 fragment definitions.

Reach (hot path): imported for setup by reference.data.api.spec.ts, opportunities.api.spec.ts:9, accounts.mutations.api.spec.ts:10, opportunities.bulk-update-template.authz.api.spec.ts:5, and more.

Why it matters: a spec that needs only opportunityStatuses still pulls all ~60 collections — a large response, more server work, slower test setup, multiplied across every consuming spec and every shard. It's also the single most likely file to hit a merge conflict (every reference-data change touches it).

Severity rationale: Medium — measurable setup-time/payload tax on a frequently-run path; not correctness.

Remediation: isolate it in _shared/reference-data.ts (it doesn't belong to one domain); consider splitting into a few purpose-scoped reference queries so specs fetch only what they assert on. Lower priority than the correctness fixes.

Observation (server-side, already documented): mergePartiesPreview.ts:3-7 notes the preview has no pagination and times out on large parties (CI-7633). Real perf risk, but it's the resolver's, not the packaging's — keep the warning co-located with the operation.
