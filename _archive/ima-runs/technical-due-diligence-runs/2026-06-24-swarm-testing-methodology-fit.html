Is Swarm Testing the right methodology for this suite & application?

A methodology-fit due-diligence on adopting Groce-style Swarm Testing (randomized-subset random test generation, Groce/Regehr et al., ISSTA 2012) for the IMA 360 Playwright E2E + API suite and the IMA 360 / CRM application it tests. Evidence-first: the verdict is grounded in what the suite and app actually are.

Verdict — bottom line up front

No. Swarm Testing is the wrong methodology for this suite as constituted, and a poor fit for the application's highest-value test surfaces. It is a randomized-generation fuzzing technique whose three preconditions — an automated generator over a feature set, an automated oracle, and a massive cheap test budget — are each absent or actively hostile here. The three findings that would most change the decision:

  1. There is no automated oracle to swarm against. Every test encodes a hand-written, requirement-specific expectation; the cheap crash/HTTP-200 oracle swarm relies on is blind to IMA's real failure modes (business errors and silent data loss live in the payload, not in a thrown error).
  2. The most valuable operations are destructive, irreversible, and run against shared mutable INT data with no rollback. A randomized mutation swarm would permanently corrupt the environment the entire deterministic suite depends on — "there is no unmerge."
  3. The execution economics are inverted by 3–4 orders of magnitude. A test here is a real Auth0 browser session or live GraphQL round-trip at 60–180 s each on a single shared environment; swarm's diversity payoff requires 10³–10⁶ runs that this suite can never afford.

There is a narrow, legitimate adjacent move — property-based / metamorphic testing on the read-only GraphQL query layer (Opportunity) — but "Swarm Testing" specifically is only a second-order diversity knob you'd reach for after building that generator, not an adoption decision to make today.

Phase 1 — Surface Map: what swarm testing requires vs. what this suite & app are

DEFINITION The bar Swarm Testing must clear

Swarm Testing (Groce, Zhang, Eide, Chen, Regehr — ISSTA 2012) is a variant of random testing. Default random testing builds each test from all available API features (one "config"). Swarm testing instead generates many tests, each from a random subset of features — the insight being that omitting features increases diversity and uncovers states a saturated config suppresses (the canonical example: tests that only push and never pop find deep-stack bugs). Its power is therefore conditional on four things:

  • A randomized test generator over an enumerated feature/operation set.
  • An automated oracle — because inputs aren't known ahead of time, you cannot hand-write an expected value per test; you rely on crash/assert/invariant/differential/metamorphic oracles.
  • A large test budget — typically 10³–10⁶ generated tests; diversity is an asymptotic, statistical benefit.
  • Cheap, side-effect-isolated execution — so many tests run fast and don't corrupt shared state.

REALITY What this suite actually is

A curated, example-based Playwright suite: *.e2e.spec.ts (real Auth0 browser → IGS-360 project) and *.api.spec.ts (live GraphQL → API-tests project), each test carrying a hand-written oracle and a @CI-T#### Zephyr tag. Data is set up deterministically (dataFactory.createTestParty) or by sampling existing real records (sqlDB.* finders via NEWID). The system-under-test is a production CRM whose marquee operations (merge, transfer, delete) mutate shared INT data and are irreversible. Every one of swarm testing's four preconditions is contradicted by this map; the findings below quote the code that establishes each contradiction.

Fit Findings — ranked by strength of mismatch

CRITICAL 1 · No automated oracle — the suite's entire value is per-test, hand-authored expectations

Evidence: src/tests/parties/party.merge.api.spec.ts:71-74, 101-105, 125 — each test asserts a specific documented requirement, not a generic "didn't crash":

// :71  No-data-loss: both legal entities must survive on the consolidated party.
const names = await dataFactoryParties.pollLegalEntityNames(gql, survivorId!);
expect(names).toContain(alpha);
expect(names).toContain(beta);
// :101 CI-7323: same-named legal entities are disambiguated, never silently merged away.
const acmeNames = names.filter((n) => n.startsWith(`Acme ${ts}`));
expect(acmeNames.length).toBeGreaterThanOrEqual(2);
// :125 Survivor is the party whose identity is kept (organizationNamePartyId).
expect(res.data.mergeParties.survivingPartyId).toBe(partyB);

Evidence: src/fixtures/graphql.fixture.ts:60-71 — IMA's failure model is precisely the kind a cheap oracle misses:

// IMA 360 error model: errors surface in data business logic fields
// (e.g., res.data.updateX.failedAccounts[0].errors, res.data.createX.validationResults)
// NOT in res.errors[] — the GraphQL spec errors array is empty on IMA business failures.

Why it matters: Swarm testing must supply its own oracle, and the only oracles cheap enough to run at swarm scale are crash / non-2xx / unhandled-exception. But the IMA app returns HTTP 200 with the failure buried in the payload — and the bug class the suite exists to catch is silent (the "snackbar lies" no-op of CI-7855; the FK regression at party.merge.api.spec.ts:215-245 that pre-fix returned success:false with no thrown error). A swarm run would score those as PASS. The expensive, irreplaceable asset here is the human knowledge encoded in each assertion; swarm testing throws it away and substitutes an oracle that is blind to the application's actual defects.

Severity rationale: Critical — without an oracle that can see business-layer failure, a swarm finds only outright 500s, which the existing suite + App-Insights monitoring already catch. The methodology cannot detect what this suite is for.

Remediation sketch: Not remediable as "swarm testing" — you would first have to build a differential or model oracle for CRM business state (a separate, large program). See Opportunity below for the one surface where a partial oracle exists.

CRITICAL 2 · Destructive, irreversible operations on shared mutable INT data with no rollback

Evidence: src/tests/parties/party.merge.api.spec.ts:11-17 — the suite's own keystone warning:

// ⚠️ Party merge is DESTRUCTIVE and IRREVERSIBLE — there is no unmerge. Every test here
// CREATES its own disposable parties per run (never samples or hardcodes real parties).

Evidence: src/tests/parties/party.merge.e2e.spec.ts:192-202 — data is consumed by the operation:

// Sampled INT pair (SINGLE-USE — the merge consumes it; re-sample before re-running):
//   victim PartyId 36281235 (Alsip MiniMill, LLC) -> survivor PartyId 31411464.
// ... the SP deletes the victim, so the pair can only be merged once.

Why it matters: Swarm testing generates random sequences of operations — exactly merges, transfers, deletes, bulk-updates in arbitrary combinations. Against a shared INT database with no test transaction boundary (tests drive the live resolver; the MSSQL MCP's auto-rollback does not wrap the application's writes), a swarm would permanently destroy real parties, contracts, and opportunities — the same records the deterministic suite samples via sqlDB.*. There is no undo. This also collides head-on with the project's standing destructive-test policy (create-then-delete only; transfer-against-existing-data permitted, deletes must self-create their target).

Severity rationale: Critical (blast-radius multiplier: data-loss path, no rollback). The failure mode isn't a bad test result — it's irreversible corruption of a shared environment used by every other test and by manual QA.

Remediation sketch: Would require a disposable, swarm-only environment seeded and torn down per run (a net-new isolated DB/app stack) — infrastructure this suite does not have and that no single methodology choice provides.

HIGH 3 · Execution economics are inverted — the achievable test budget is 3–4 orders of magnitude too small

Evidence: playwright.config.ts:97-102 — per-test cost and concurrency:

timeout: 60000,                       // 60 s default per test
expect: { timeout: 60_000 },
retries: process.env.CI ? 0 : 0,
workers: process.env.CI ? '70%' : 1,  // single worker locally

Evidence: individual tests bump far higher — party.merge.api.spec.ts:193 test.setTimeout(180_000); party.merge.e2e.spec.ts:331 test.setTimeout(180_000). Evidence: src/utils/auth0.fixture.ts:60-83 — every E2E worker pays a real Okta login round-trip before it can run a test.

Why it matters: Swarm testing's diversity benefit is statistical and asymptotic — it shows up after thousands to millions of generated runs. Here a single "test" is a real browser session or a live GraphQL call at 60–180 s against a shared, rate-sensitive environment. A multi-hour run yields on the order of 10²–10³ executions; swarm needs 10³–10⁶. The omission-diversity that makes swarm beat plain random never has room to pay off, and you would saturate/destabilize INT (and trip search-index lag, the --disable-http2 Google-Places fragility, etc.) long before it did.

Severity rationale: High — even granting an oracle and an isolated environment, the cost-per-test makes the core statistical mechanism inoperable at any budget this suite can spend.

Remediation sketch: Only a fast, in-process oracle target (e.g. a pure backend unit/service layer in the C# repos) could supply the budget — which is a different repo, language, and team, not this Playwright suite.

HIGH 4 · Traceability / governance incompatibility — every test's identity is a Zephyr test case

Evidence: party.merge.api.spec.ts:44 and throughout — tests are tagged to Zephyr and reported by a tag-driven reporter:

test('Verify merging two parties ... retains both ... legal entities ...',
  { tag: ['@api', '@Regression', '@CI-T2315'] }, async ({ gql }) => { ... });

Evidence: playwright.config.ts:93,95 — the Zephyr reporter activates whenever ZEPHYR_CYCLE_ID is set, recording each tagged test as an execution against its CI-T#.

Why it matters: Swarm-generated tests are anonymous and ephemeral by design — they have no stable title, no @CI-T####, no Navigate/Act/Verify steps, and no regression tier. The suite's governance contract (which Robert owns across the CI project) is 1:1 test↔Zephyr-TC traceability with a curated smoke/regression pyramid. Pouring thousands of untraceable, single-use executions into that model doesn't extend it — it bypasses it. There is nothing for /zephyr-reporter, the smoke registry, or the regression overhaul to attach to.

Severity rationale: High — a structural conflict with the suite's reason-for-being (auditable, traceable regression coverage), not a tooling gap that a wrapper could close.

Remediation sketch: Swarm results would have to live in a wholly separate, non-Zephyr "fuzzing" lane with its own triage — which concedes it is not part of this suite's methodology.

MEDIUM 5 · The randomization the suite already has is value-level, not structure-level — it is not "almost swarm testing"

Evidence: package.json:96 ships @faker-js/faker, used in 25 source files (e.g. src/dataFactory/dataFactory.party.ts, dataFactory.account.ts) to randomize field values in create flows; the sqlDB.* finders randomly sample existing rows via NEWID(). Evidence: there is no property-based or fuzzing library in package.json — no fast-check, no jsverify.

Why it matters: It is tempting to think "we already randomize, so swarm is a small step." It isn't. Faker randomizes the inputs to a fixed test structure with a fixed hand-written oracle; NEWID randomizes which real record is the input. Neither randomizes the set of operations a test performs, and neither replaces the oracle. The distance from here to swarm testing is the entire generator-plus-automated-oracle machinery — i.e. essentially all of the cost. Naming this explicitly prevents the decision from being made on a false sense of proximity.

Severity rationale: Medium — not a risk on its own; it corrects the most likely misconception underpinning a "yes."

Remediation sketch: Keep faker/NEWID for what they're good at (input variety within curated tests); don't mistake them for a stepping stone to swarm.

OPPORTUNITY 6 · Where a swarm-adjacent technique would actually pay off — the read-only GraphQL query layer

Evidence: the one surface with the right shape is the pure query layer — typed via codegen, request/response with no destructive side effects. The suite already gestures at invariant oracles: src/fixtures/matchers.fixture.ts:51-103 defines toBeSortedAscending / toBeSortedDescending:

toBeSortedAscending<T>(received: T[], dataType?) { /* asserts ordering invariant */ }

Why it matters / what to do instead: On read-only queries (party/opportunity/contract grids, filters, search, pagination) you can get a cheap, automatic oracle from metamorphic relations — e.g. "a filtered count ≤ the unfiltered count," "sort is idempotent and order-reversible," "page 1 ∪ page 2 ⊆ the unpaged set," "an unknown filter value returns 0, never a 500." That is property-based testing, for which a small fast-check footprint over query variables is the right tool. It is bounded (no shared-state mutation), affordable (queries are cheaper than browser flows), and oracle-bearing.

The key distinction: swarm testing is only a diversity-tuning knob within a random/property-based generator — you reach for feature-subsetting after you have a generator whose features suppress one another. You do not "adopt swarm testing"; you would adopt property-based testing on the query layer first, and only later, if a single saturated generator proved to hide bugs, turn the swarm knob on. So even the legitimate move reads as "pilot property-based + metamorphic tests on the GraphQL query surface," not "swarm-test the suite."

Severity rationale: Opportunity, not a risk — a scoped, reversible experiment worth a spike if randomized testing is the underlying goal.
