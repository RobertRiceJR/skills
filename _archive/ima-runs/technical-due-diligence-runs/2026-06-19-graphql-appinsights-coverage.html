Verdict — the three things an acquirer should price
  1. The #1 user-facing GraphQL failure has zero test coverage. getCurrentUserTeam fails 244×/24h on INT (resolver crash) yet is not referenced by any test — it isn't even codegen'd into the suite. The suite is blind to its largest real GraphQL error. (See F1.)
  2. GraphQL failures are invisible to HTTP-status monitoring. ~271 of the real failures return HTTP 200 with success==false (the error rides in the errors[] body). Anything keying on resultCode >= 400 — dashboards, alerts, naive tests — misses them entirely. (See F5.)
  3. The bulk of the volume is app-side and beyond this suite's reach. A search-index handler throws 976×/24h and Industrial SQL queries time out ~640×/24h; both are background/server paths an E2E suite cannot exercise. Coverage stops at the API boundary — these need server-side tests/alerts. (See F7, F6.)

Post-release (2026-06-20): the updated release fixed F2 (updatePreset) but introduced a masked regression — F11: AssociateAccountsToOpportunity 500s for ULOB COH ("no authorization strategy"), new on release day. The masking drill (F5) confirms nothing is lost — every masked-200 failure has a 1:1 logged exception — so monitor on success==false / the exceptions table, never resultCode.

Dimension 2 — Test coverage & gaps

HIGH F1 — The #1 user-facing GraphQL failure (getCurrentUserTeam) has zero test coverage

Evidence: App Insights 24h: /graphqlgetCurrentUserTeam = 244 failures (HTTP 200, success==false), exception BestOrganizationAgentIdDataLoader…GetBestOrganizationAgentId ArgumentOutOfRange ×206. Suite grep:

$ grep -rln "getCurrentUserTeam" src/
src/generated/graphql.ts        # ← only here? NO — query name absent even from generated
# (no match in src/data, src/tests, src/api, src/pages — and not codegen'd as an operation)

Why it matters: getCurrentUserTeam bootstraps the signed-in user's sales-team context. It is the single largest real GraphQL error on INT, it crashes server-side, and no test in this suite exercises it — the suite cannot regress-detect the app's most frequent GraphQL failure. An acquirer inherits a blind spot on the app's startup path.

Severity rationale: High — a user-context bootstrap failing undetected reaches many users; not Critical (read/bootstrap path, no data loss).

Remediation: add an API test that issues getCurrentUserTeam for the automation user and asserts res.errors.length === 0; codegen the operation so it's typed.

MEDIUM F2 — updatePreset mutation is untested and failing on FE/BE schema drift

Evidence: App Insights: /graphqlupdatePreset 500 ×4; exception HotChocolate.Types.SerializationException: The fields `descriptions`, `count` do not exist on the type `PreferenceDetailModelInput`. Schema has the mutation, no test drives it:

src/generated/graphql.ts:4114:  updatePreset: OrganizationAgentPreferenceModel;
$ grep -rln "updatePreset" src/   →  only src/generated/graphql.ts

Why it matters: the client sends input fields (descriptions, count) that no longer exist on PreferenceDetailModelInput — a front-end/back-end contract drift. No API test asserts the updatePreset input shape, so this drift shipped silently and only surfaces as a prod 500.

Severity rationale: Medium — preset save is a non-critical feature; localized blast radius.

Remediation: an API test that sends a valid UpdatePresetInput and asserts no errors[] — would catch input-type drift at PR time.

MEDIUM F3 — A brittle, abortable fetch is used as a functional waiter (Office365Photo)

Evidence: src/tests/dashboard/pipeline.management.e2e.spec.ts:151

const promise = waiters.getResponsePromise(page, API_ENDPOINTS.OFFICE_365_PHOTO);
await searchComponent.fillSearchTextBox('Inc');
await waiters.waitForResponseSuccess(promise);

Why it matters: Office365Photo is a user-avatar fetch that aborts constantly (230 × 499 in 24h — the 2nd-noisiest op on INT). Gating the party-search assertion (CI-T1258) on it makes the test flaky/false-negative regardless of whether search works, and it isn't the search's completion signal. CI-T1258 is one of the 44 current red survivors.

Severity rationale: Medium — test reliability defect, not an app defect.

Remediation: wait on the actual search/pipeline response (or the rendered rows), not the avatar fetch.

Evidence: the prod-failing upload endpoints map 1:1 to currently-RED suite tests:

Dimension 4 — Error handling & failure modes

HIGH F5 — GraphQL failures are masked as HTTP 200 — invisible to status-code monitoring

Evidence: 24h — getCurrentUserTeam (244), mergeParties/createParty/deleteOpportunity/createCalendarEvent all return resultCode 200 with success==false; the error lives in the GraphQL errors[] body, not the HTTP status. (Verified: requests | where name contains "graphql" and success==false returns 200s.)

Why it matters: any monitor/alert/test that filters on resultCode >= 400 misses the majority of GraphQL failures. The App Insights "Failures" blade only surfaces them because it keys on success. This is the single biggest reason GraphQL errors are easy to miss — the correct filter is success == false + name contains "graphql", never resultCode.

Drill (2026-06-20): the masking loses nothing — every masked-200 GraphQL failure has a 1:1 logged exception in the exceptions table (24h join: getCurrentUserTeam 104/104, mergeParties 6/6, AssociateAccounts 1/1 — reqs == with_exception for every op). HotChocolate catches the resolver exception, logs it, and returns 200 with the error in errors[] (GraphQL spec). The danger is purely the misleading HTTP status, not data loss.

Severity rationale: High — systemic observability gap across the whole GraphQL surface.

Remediation: monitor on success==false or the exceptions table (both 1:1 reliable), never resultCode (now captured in app-insights-diagnose/reference/README.md). Optional app-side hardening: configure HotChocolate to emit a non-200 status on execution errors so plain HTTP monitoring catches them too (QueryGateway config).

HIGH · POST-RELEASE F11 — New regression: AssociateAccountsToOpportunity 500s for ULOB "COH" (masked 200)

Evidence: App Insights — /graphqlAssociateAccountsToOpportunity first failed 2026-06-20 (release day; 4 fails, 0 across the prior 6 days), HTTP 200 / success==false. Underlying logged exception:

System.Net.WebException: {"message":"No strategy found to resolve authorizations for ulobs: COH", "statusCode":500}
  at Igs.MarketIntegrations.ApiClients.AuthorizationClient.SearchAuthorizations(...)
  at Igs.Operations.Core.Services.MarketIntegrations.AuthorizationService.SearchAuthorizationsAsync(...)  AuthorizationService.cs:179
  at Igs.Operations.Service.Web.Controllers.Operations.AuthorizationsController.SearchAuthorizations(...)  AuthorizationsController.cs:59

Why it matters: associating accounts to an opportunity calls the Operations AuthorizationService.SearchAuthorizations for the account's ULOB; the service has no authorization-resolution strategy registered for ULOB COH → downstream 500 → masked as a GraphQL 200. New on the release day = a release regression. This is market/utility authorization, not Auth0/user auth.

Call path / suite reach: driven by src/tests/opportunities/opportunity.record.accounts.e2e.spec.ts:11 (CI-T1301 "Adding party accounts to an opportunity") and opportunities.bulk.actions.e2e.spec.ts (CI-T1253) via waiters.types.ts:2 ASSOCIATE_ACCOUNTS_TO_OPPORTUNITY + dataFactory.account.ts. The suite will catch it — but only when the sampled account's ULOB is COH (data-dependent), so it reads as an intermittent flake unless explicitly seeded.

Severity rationale: High — a new release regression on a core flow (account↔opportunity association), masked so HTTP monitoring won't see it; held below Critical by low volume + single-ULOB scope.

Remediation: register an authorization-resolution strategy for ULOB COH in Igs.Operations AuthorizationService (or handle the missing-strategy case gracefully) — QueryGateway/Operations work; add a CI-T1301 variant seeded with a COH account to regression-detect it.

MEDIUM F8 — 499 client-closed dominates the failure count (~82%), inflating the alarm

Evidence: 24h — 944 of ~1,149 failures are HTTP 499 (POST /graphql 714 + Office365Photo 230); the matching exception is HotChocolate OperationCanceledException ×558. No server fault — the caller closed the socket.

Why it matters: the "1.14k failures" headline is ~82% client-abort noise (navigation/timeout). Triage that doesn't strip 499 first burns on noise and buries the ~205 real failures. (Same class as the EvaluateAll 499 drilled earlier — a 499 is a timeout/abort, not a server error.)

Severity rationale: Medium — distorts triage; no direct user impact.

Remediation: exclude resultCode == "499" in the standard failure query; treat 499 spikes as a latency/timeout signal, separately.

LOW F9 — createCalendarEvent fails with MS Graph "Forbidden" (environment permission)

Evidence: /graphqlcreateCalendarEvent 200/success==false ×3; exception Igs.MicrosoftGraph.Exceptions.MicrosoftGraphCalendarException: Forbidden.

Why it matters: a Microsoft Graph calendar permission denial (downstream), not an app-code defect. Covered by calendar.e2e.spec.ts, but the failure is environmental — worth confirming the INT service principal's Graph calendar scope.

Severity rationale: Low — env/permission, low volume.

Remediation: verify the Graph app registration's Calendars permission in INT; don't chase as a code bug.

Dimension 7 — Performance hotspots (app-side; suite-untestable)

HIGH F6 — Industrial entity SQL queries are timing out (~640×/24h)

Evidence: 24h exceptions — Win32Exception: Execution Timeout Expired at Igs.Crm.Core.Data.Ima.IndustrialEntities.GetImaCustomerCustomerSegments ×504, GetImaOpportunityCustomerSegments ×70, GetBrokersByPartyAccountId ×25, ParallelForEachAsync ×40.

Why it matters: the Industrial customer-segment/broker lookups exceed the SQL command timeout at volume — driving 500s and feeding the 499s (callers give up). This is server-side DB performance; an E2E suite cannot exercise or assert it (no test boundary touches these batch loaders).

Severity rationale: High — a core data path failing ~640×/day; cross-cuts opportunity/party reads.

Remediation: index/plan review on the customer-segment + broker-by-account queries; add a server-side timeout/perf SLO + alert. Out of this suite's reach — flag to the app team.

HIGH F7 — Duplicate-identification index handler throws on an unhandled table (976×/24h)

Evidence: 24h top exception — System.ArgumentOutOfRangeException at Igs.Crm.Ima.Search.Index.ResolverHandlers.DuplicateIdentificationResolverHandler.GetIndexMapper: Table 'Party.OrganizationContactAddresses' was not handled by DuplicateIdentificationResolver (sibling ×2 for Opportunity.IndustrialLargeVolumeTransportOpportunities). Empty operation_Name → fires in the background RAD/indexing path.

Why it matters: the dominant exception by volume. The duplicate-identification search-index mapper hits a table case it doesn't handle and throws — meaning dedup indexing is partially broken (the CI-7764/CI-7826 search-index class). It's background/server-side, so this suite cannot catch it (no E2E or API path covers the indexer). An acquirer should see this as an unmonitored index-integrity defect.

Severity rationale: High — index-integrity defect at high volume; not Critical (no confirmed user-facing data loss yet), app-side.

Remediation: add the missing table case(s) to the index mapper; this is QueryGateway/CRM-app work, not test-suite work — surface to the indexer owners (ties to /elastic-drift).

Streamline — making this diagnosis a repeatable loop

LOW F10 — The diagnosis is manual and trap-laden; no saved query or scheduled pull

What slows it down today (all now captured in app-insights-diagnose/reference/):

  • Sampling trap: count() undercounts (adaptive sampling); must use sum(itemCount) to match the portal — cost me a wrong "419 vs 1.14k" until corrected.
  • 499 noise: 82% of "failures" are client-aborts; the standard query must exclude resultCode=="499".
  • GraphQL is 200-on-error: filter success==false + name contains "graphql", never resultCode.
  • Tooling: system Python lacks azure-cli; az.cmd chokes on the (INT) parens → use the bundled CLI2 python.

Recommended streamlined loop:

  1. Canonical 24h GraphQL-failure query (saved): requests | where name contains "graphql" and success==false and resultCode!="499" | summarize n=sum(itemCount) by name, resultCode | order by n desc.
  2. Auto-map each failing op → suite test (the table above) and flag untested ops (today: getCurrentUserTeam, updatePreset) as coverage gaps.
  3. Run it on a schedule (a /elastic-drift-style nightly, or a cron agent) that diffs the top failing ops against suite coverage + current reds, and posts the delta.

This run already hardened the reference (sampling rule, 499 rule, GraphQL-200 rule, no-browser-table rule) so the next person doesn't re-learn the traps.
