Do we need WITH (NOLOCK) on the SQL finder queries?

Every query in src/utils/sql/queries is a read-only SELECT used to sample or verify test data against the shared IMA 360 CRM & Industrial databases. This run answers the NOLOCK question against the databases' actual concurrency configuration — not from first principles.

The Answer — bottom line up front

No. Do not add WITH (NOLOCK) to these queries — not to any of the 166. It would buy you nothing and cost you correctness. The reasoning is one fact, verified live against all four databases the suite touches:

  1. READ_COMMITTED_SNAPSHOT is ON on the CRM and Industrial databases in both INT and QA (verified via sys.databases, 2026-07-01). Under RCSI, the default READ COMMITTED already serves reads from row versions — your SELECTs take no shared locks, never block a writer, and are never blocked by one. → Finding 1
  2. NOLOCK's only real benefit — avoiding blocking — is therefore already provided, but without NOLOCK's hazards. Adding it is a pure downgrade: same concurrency, plus dirty reads, missing/duplicate rows, and Error 601 scan-aborts. → Finding 2
  3. The finders are exactly where dirty reads hurt most — the merge-safety predicate and HU-eligibility filters depend on precise, committed counts; a dirty read reintroduces the very flakes (CI-7631/7633/7761) those predicates were built to prevent. → Finding 3

The current implementation — single-statement autocommit reads on an RCSI database — is already the correct, lock-light shape. The right action is no code change, plus a one-line note that the suite relies on RCSI. → Finding 5

Phase 1 — Surface Map (how a query actually runs)

PATH Finder string → autocommit read → no lock held

Every finder is a plain SQL string (or a ParameterizedQuery) returned from the queries.* modules, handed to the sqlDB.* facade, and executed by one method:

// src/utils/sql/sql.client.ts:38-50
async executeQuery<T>(query: string | ParameterizedQuery): Promise<T[]> {
    const pool = await this.connect();
    const request = pool.request();
    ...
    const results = await request.query<T>(query.text);   // ← single statement, autocommit
    return results.recordset ?? [];
}

What this tells us about locking:

  • No explicit transaction. No beginTransaction() / pool.transaction() anywhere in the SQL layer (grep-confirmed). Each finder is a single autocommit statement — so cross-statement anomalies (non-repeatable read, phantom) can't even arise; only within-statement consistency matters.
  • No isolation-level override. No SET TRANSACTION ISOLATION LEVEL and no isolationLevel in the connection config (sql.config.ts:11-25). Every statement runs at the server/database default = READ COMMITTED.
  • The database default is not lock-based. Because READ_COMMITTED_SNAPSHOT is ON (Finding 1), READ COMMITTED here means row-versioned reads — statement-level snapshot, no shared (S) locks acquired.
  • Targets are shared non-prod app DBs. Context.getCrmSqlServer() / getIndustrialSqlServer() resolve to INT (default) or QA — the live app writes to these concurrently. Prod is a separate MCP-only server this suite never queries. So the only stake in the NOLOCK question is the quality of test-data selection, never production integrity.

166 finders by file: party 47 · account 48 · opportunity 43 · industrial 14 · contract 6 · enum 6 · communications 2. All are SELECT-only; _shared/fragments.ts holds two helper builders (dateWindow, toIdList), not queries.

Findings (ranked by blast radius of adopting NOLOCK)

HIGH 1 · On a RCSI database, NOLOCK is a pure downgrade — it removes locks that are already not being taken

Evidence — live query, all four databases (2026-07-01):

SELECT name, is_read_committed_snapshot_on, snapshot_isolation_state_desc
FROM sys.databases WHERE name = DB_NAME();

CRM  (INT)        → is_read_committed_snapshot_on = True   snapshot_isolation_state = ON
CRM  (QA)         → is_read_committed_snapshot_on = True   snapshot_isolation_state = ON
Industrial (INT)  → is_read_committed_snapshot_on = True   snapshot_isolation_state = ON
Industrial (QA)   → is_read_committed_snapshot_on = True   snapshot_isolation_state = ON

Why it matters: people reach for NOLOCK for exactly one reason — to stop a reader from taking shared locks that block, or get blocked by, writers. Under READ_COMMITTED_SNAPSHOT, the default isolation already does that: readers are served from the version store and acquire no S-locks. So NOLOCK's entire upside is already delivered by the database configuration — for free, and without side effects. Adding NOLOCK on top changes the concurrency behaviour by zero while opting every finder into READ UNCOMMITTED semantics (dirty reads, allocation-order scan anomalies). You would be trading a guaranteed-consistent statement snapshot for an inconsistent one and getting nothing back.

Reach: all 166 finders, both environments. This is the finding that decides the question for the whole directory.

Severity rationale: High, not Critical — the blast radius is suite-wide test reliability (flaky, non-reproducible data selection), not production data or money. It cannot reach prod: the suite never connects to a prod DB.

Remediation: add nothing. The default path is correct. (If RCSI were off, the honest answer would be "enable RCSI at the DB, don't sprinkle NOLOCK" — but it's already on.)

MEDIUM 2 · NOLOCK would introduce Error 601 scan-aborts on the hot append tables these finders read

Evidence — the log-table finders that read high-churn tables:

// src/utils/sql/queries/queries.industrial.ts:93-100 · 130-135
FROM dbo.[log] l  WHERE l.Exception LIKE '%ProcessErroredDeals%' ...     // getProcessErroredDealsErrors
FROM dbo.[Log] l  WHERE l.Logger IN ('IGS.CIBatch','IGS.DealCapture') ... // getCIBatchRecentErrors

Why it matters: READ UNCOMMITTED scans read pages in allocation order without stability locks. When a concurrent writer causes a page split or row movement mid-scan (routine on a hot, constantly-appended log table), SQL Server can raise Error 601: "Could not continue scan with NOLOCK due to data movement." The query doesn't return bad data — it throws, and the finder fails. The classic "read a busy log table" scenario is the one place a NOLOCK argument sounds plausible, yet it's precisely where 601 and skipped/duplicated rows are most likely. Under RCSI (today) these same reads get a stable, non-blocking snapshot with none of that risk.

Reach: the two Industrial log finders most directly; more broadly any finder scanning a churning table (HU request/response tables, merge history) inherits the same class of intermittent, load-dependent failure.

Severity rationale: Medium — intermittent data-setup aborts under load; a workaround exists (don't add NOLOCK), and it degrades reliability rather than corrupting anything downstream.

Remediation: do not add NOLOCK; the RCSI snapshot already reads these tables cleanly without blocking the batch writers.

MEDIUM 3 · Dirty reads would corrupt the merge-safety and HU-eligibility predicates — reviving the exact flakes they were built to kill

Evidence — the merge-safety predicate depends on precise, committed state:

// src/utils/sql/queries/queries.party.ts:663-670
const MERGE_SAFE_PARTY_PREDICATE = `
    p.RelationshipManagerId = 1
    AND p.IsMerged = 0
    AND p.PartyTypeId = ${PartyType.Organization}
    AND NOT EXISTS (SELECT 1 FROM Party.IneligibleForMergeParties gts WHERE gts.PartyId = p.PartyId)
    AND (SELECT COUNT(*) FROM Party.Accounts acct WHERE acct.PartyId = p.PartyId) <= 3
    AND (SELECT COUNT(*) FROM Opportunity.Opportunities opp WHERE opp.PartyId = p.PartyId) <= 3`;

Why it matters: these finders don't just fetch rows — they encode business-safety invariants. getMergeSafeDuplicatePairs selects only pairs with IsMerged = 0, an exact account/opp count ≤ 3, and no GTS-ineligible row; the HU family (queries.account.ts) selects on NOT EXISTS request rows and COUNT(...) = 0 "no HU yet" conditions. A dirty read of an in-flight, uncommitted merge, account insert, or HU request — data that may then roll back — would let a finder pick a party/opportunity that isn't actually eligible. Result: the merge test then trips the masked server-side error (CI-7761) or the non-atomic MergedOpportunities PK violation (CI-7631/7633) that the predicate's comments say it exists to avoid. NOLOCK would silently reintroduce those flakes, at random, under concurrent load — the hardest kind to reproduce.

Reach: the party merge finders (getMergeSafeDuplicatePairs*, getPartiesForMerge), the HU eligibility/status family in queries.account.ts, and any count/NOT-EXISTS-gated finder — dozens of queries.

Severity rationale: Medium — produces flaky, misleading test runs (a green test that selected invalid data, or a spurious failure), not permanent corruption.

Remediation: keep the default RCSI reads, which see only committed rows as of statement start — exactly the semantic these predicates assume.

LOW 4 · The one place NOLOCK is "traditionally" argued for — the busy log tables — still fails the cost/benefit here

Evidence: getProcessErroredDealsErrors and getCIBatchRecentErrors (queries.industrial.ts:93,130) read append-heavy diagnostic log tables — the textbook "just NOLOCK it" case.

Why it matters: even granting the sympathetic case, the trade is bad. (a) The blocking concern is already gone under RCSI. (b) A dirty read here means reading a logged exception row from a transaction that later rolls back — i.e. an error that never really committed — which actively misleads the very diagnostics these finders feed (Risk Dashboard / deal-scraper triage). (c) Exposure to Error 601 (Finding 2) is highest on exactly these tables. So the "hot log table" intuition points the wrong way here.

Severity rationale: Low — narrow to two diagnostic finders; worst case is a misleading or briefly-failed diagnostic read, not test-data corruption.

Remediation: none; leave as default reads.

LOW 5 · The current design is already the correct lock-light shape — the only gap is that the RCSI dependency is undocumented

Evidence: single-statement autocommit reads (sql.client.ts:38-50), no isolation override (sql.config.ts:11-25), RCSI ON (Finding 1). This combination gives every finder a transactionally-consistent, non-blocking snapshot as of statement start — the best available correctness/concurrency profile for read-only test-data selection, with no hints required.

Why it matters: the suite's clean concurrency behaviour is inherited from a database setting it doesn't control. If a DBA ever disabled READ_COMMITTED_SNAPSHOT on INT/QA, these reads would silently revert to acquiring shared locks and could begin blocking / being blocked by the app — the failure mode people wrongly try to pre-empt with NOLOCK. That dependency is nowhere recorded in the SQL layer.

Severity rationale: Low — latent, contingent on an external config change; no present impact.

Remediation: add a one-line comment at the top of sql.client.ts (or the query README) noting: "Reads rely on READ_COMMITTED_SNAPSHOT = ON on the CRM & Industrial DBs for non-blocking, consistent snapshots — do not add NOLOCK; if RCSI is ever disabled, revisit isolation here, not with table hints." Optionally add a startup/CI assertion querying sys.databases.is_read_committed_snapshot_on.

Recommendation

What to do

1. Add no NOLOCK hints. The proposed change is contraindicated on a RCSI database — all cost, no benefit.

2. Record the dependency. One comment in sql.client.ts (Finding 5) so the next engineer knows the clean concurrency behaviour rides on READ_COMMITTED_SNAPSHOT = ON, and that the correct lever — if it's ever disabled — is isolation configuration, not table hints.

3. If a specific finder is ever slow, that is a scan-volume / indexing issue (these already use TOP N + indexed predicates), not a locking one — NOLOCK would not help a query that is slow for any reason other than lock waits, and under RCSI there are no lock waits to remove.
