Will the volume of the QE Claude ecosystem mis-steer an agent if it lives inside the app repo?

A context-pollution / agent-steering risk audit of the full .claude/ tree (61 skills · 69-node knowledge tree · 82 memory files · two CLAUDE.md layers) as a footprint co-located inside a developer/app repository — not a review of the app source. Question: how badly can the sheer mass of QE context confuse or persuade an agent doing development work, and how do we contain it?

Verdict — the three things that matter most

Bottom line: the answer to "will Claude get confused?" is yes, materially — but only because co-location is the one configuration that breaks the isolation the platform otherwise has by construction (see the sibling run TDD-022, which found the shipped toolkit well-isolated). The danger is not the raw 40,125 lines of markdown — most of it is lazy, read only on demand. The danger is the always-on floor (~23.5K tokens) of QE convention, framed as MUST-follow overrides, sitting on top of a body of confident test-harness assertions an app agent has no way to recognize as test-only. The mitigation is real and ranked below; the cheapest correct one is "don't co-locate."

  1. [CRITICAL] Test-harness facts stated as unconditional app imperatives. "apply to all party-scoped SQL queries", "Google Places broken — do NOT use", "exclude StatusIds 7/8" — under the harness's OVERRIDE any default behavior / MUST follow exactly framing, a dev agent will obey them against app code and silently scope or break real data.
  2. [HIGH] The eager-load floor frames every session as a test session. ~23.5K tokens of QE convention auto-loads before any app task is read — the agent's default interpretation of "fix this" or "add a query" is pulled toward Playwright/Zephyr/test-data shapes.
  3. [HIGH] Globally-visible autonomous writers with no approval gate. ruby-migration ("writes … autonomously — no approval gates"), fix-loop, and the standing memory "never ask 'should I proceed' … skills run start-to-finish" become live edit hazards over app source the moment a broad trigger mis-fires.

Scope & Plan

Target: the .claude/ QE ecosystem as a passive + active context footprint, evaluated against the hypothetical the developer posed — "if the entire Claude knowledge ecosystem that lives in the test framework sat inside a developer repo where the app lives, what is the risk the agent is confused or persuaded wrong by the volume?" This is a forward-looking risk / control-gap audit, not a defect hunt in app source — where no enforcing code exists, findings are framed as control gaps + recommended guardrails.

In scope: what loads eagerly vs. lazily; instruction precedence & the OVERRIDE/MUST framing; the false-authority surface; trigger/description hygiene across 61 skills; autonomous-write failure modes; the deliberate-duplication design; dangling external assumptions; context-window cost.

Out of scope: the app source itself; the distribution/leak-out question (that is TDD-022 — this run is the inverse: harm once the ecosystem is already inside the dev repo).

Run shape: inline (single cohesive target). Measured, not inferred: file/byte/LOC counts via find/wc/du; eager-load bytes summed from the actual frontmatter + the two CLAUDE.md layers + MEMORY.md; every directive finding cites file:line with the verbatim quote.

Relationship to TDD-022 (same day): TDD-022 asked "will the QE platform leak out into a dev repo via sharing?" and found it well-isolated by construction. TDD-023 asks the opposite: "given it is all co-located inside the dev repo, does the volume mis-steer the agent?" The two are complementary, not duplicative.

1 · Architecture & boundaries — the eager-load precedence floor

HIGH Every session opens framed as a QE/test session before any app work is read

Evidence: harness wrapper around both CLAUDE.md layers — system-reminder (claudeMd):

IMPORTANT: These instructions OVERRIDE any default behavior and you MUST
follow them exactly as written.

…wrapping CLAUDE.md:13-21 (the repo's self-description):

## What This Repo Does
End-to-end and API-level test automation for the IMA 360 CRM application.
Tests run against two environments: `int` (default) and `qa`.
... E2E tests — browser-driven via Playwright; `*.e2e.spec.ts` (→ `IGS-360` project)
  • Why it matters: ~23,500 tokens of QE convention load before the agent reads a single app file, and the harness frames it as overriding the agent's defaults. In a test repo that is correct. Co-located in an app repo, it means the agent's prior on "what is this repo / what does 'add a query' mean here" is anchored to Playwright + GraphQL test shapes — the volume itself is the persuasion. A dev asking "write a query for parties" gets pulled toward data/<domain>.queries.ts + the fixture pattern, not the app's actual data layer.
  • Call path / reach: affects 100% of turns. The floor is the single largest lever on interpretation.
  • Severity rationale: High — pervasive bias on every interaction, but it steers interpretation, not an irreversible action by itself (that escalates in Dim 3 / Dim 4).
  • Remediation: a dev-repo CLAUDE.md must describe the app; QE conventions move behind an explicitly test-scoped file the agent loads only for test work. Never let the QE CLAUDE.md be the repo's top-level identity.

2 · Guardrail coverage — what actually verifies the ecosystem doesn't mis-route?

MEDIUM No runtime guard prevents a QE trigger from firing on development work

Evidence: the drift machinery is report-only and QE-scoped — e.g. /sync family (sync-skills, sync-knowledge, sync-memory) audits skill architecture, not trigger-vs-task fit; and the only behavioral guards in the repo are husky pre-commit hooks scoped to the test suite.

  • Why it matters: nothing checks "did a skill fire on a prompt that was actually about app code?" The protections that exist are about the QE ecosystem's internal consistency, not about containing it when it shares a repo with app source. There is no negative test for mis-routing.
  • Call path / reach: the absence is the finding — every mis-route is silent.
  • Severity rationale: Medium — a control gap, not an active fault; raises the cost of every other finding because nothing catches the failure after the fact.
  • Remediation: if co-located, add a lightweight router guard (or namespace, Dim 3) and a "this repo is an app repo, QE skills are opt-in" banner the orchestrators check before acting.

3 · Trust boundary & instruction precedence — the contamination surface

CRITICAL Test-harness facts are written as unconditional app imperatives — High → Critical (data path)

Evidence: CLAUDE.md:160, :161, :169 (the "Known Conventions" table):

| CI filter | `p.RelationshipManagerId = 1 AND p.CreateDate BETWEEN ...`
            — apply to all party-scoped SQL queries |
| Terminal state | Exclude IMA opportunity StatusIds 7 (Won) and 8 (Dead)
            from mutation test data ... |
| Google Places broken | Playwright config has `--disable-http2` in
            `launchOptions` which breaks the Google Places Autocomplete API.
            Do NOT use `pressSequentially` / Google Places in address fields ... |
  • Why it matters: these are test-harness truths — the CI filter exists to scope test data to one RM; "Google Places broken" is broken because the Playwright launch flag disables HTTP/2, which has nothing to do with the app. But they are phrased as flat imperatives ("apply to all party-scoped SQL queries", "Do NOT use Google Places") and sit under the harness's OVERRIDE / MUST follow exactly framing. A co-located dev agent has no signal that these are test-only. Acting on the CI filter would silently scope a production query to RelationshipManagerId = 1 — wrong results for every other RM's parties. Acting on "Google Places broken" would make the agent misdiagnose or rip out a working app integration.
  • Call path / reach: any app task touching party SQL, opportunity status, or address autocomplete — i.e. core CRM data paths. The instruction is unconditional, so the only thing standing between it and a corrupt query is the agent choosing to ignore an OVERRIDE-framed MUST.
  • Severity rationale: High baseline (false authority that requires the agent to act) → Critical via the data/correctness multiplier: a silently RM-scoped query is data-correctness corruption with no rollback signal.
  • Remediation: re-label every test-harness fact as a test-harness fact ("in TESTS, the Playwright --disable-http2 flag breaks Google Places — fill fields manually"; "the CI filter scopes test data, not production queries"). The fix is one clause per row and it neutralizes the whole finding.

HIGH 61 always-on skill descriptions with broad triggers collide with ordinary dev verbs

Evidence: generic-named, broadly-triggered skills in the eager registry — e.g. playwright: "Automate browser interactions, test web pages and work with Playwright tests"; plus /run, /init, /verify, /code-review, /security-review; and IMA orchestrators like ready-for-testing / three-amigos keyed to "test", "review", "ready".

  • Why it matters: 49,213 bytes of trigger phrases are always in the routing surface. A developer saying "run the app", "review this", "init this repo", or "test the change" can route into a QE orchestrator that then executes a Zephyr/Playwright-shaped workflow on app code. The volume of overlapping triggers makes a wrong routing decision more likely, not less.
  • Call path / reach: every prompt is matched against all 61 descriptions; the broadest ones dominate.
  • Severity rationale: High — mis-routing a dev prompt into a QE orchestrator can trigger real writes (compounds with Dim 4).
  • Remediation: namespace QE skills (prefix), tighten descriptions so they don't fire on bare dev verbs, and mark IMA-coupled skills project-scoped so they are absent from a dev repo entirely.

4 · Failure modes — what happens when a skill mis-fires

HIGH Autonomous, globally-visible writers with no approval gate

Evidence: ~/.claude/skills/ruby-migration/skill.md (global, IMA-specific, user-invocable):

... and writes the implemented test body and updates POMs autonomously — no
approval gates. Runs type-check and shows a git diff when done.

…reinforced by the standing memory MEMORY.md:5 → feedback_autonomous_execution.md:

[Autonomous Execution] — never ask "should I proceed" mid-skill; skills run
start-to-finish
  • Why it matters: the "no approval gates / never ask / run start-to-finish" posture is correct and safe inside a QE sandbox — it is what makes the skills pleasant. Over app source, the same posture means a mis-triggered skill (Dim 3) edits files and updates "POMs" with no human checkpoint. fix-loop likewise iterates fixes autonomously. ruby-migration is also global, so it is visible in a dev repo even if the project skills are not copied.
  • Call path / reach: any mis-route into a write-capable skill → unguarded working-tree edits; recoverable via git, but only if the dev notices before committing.
  • Severity rationale: High — autonomous writes to app code without a gate; the blast radius is bounded by git, which keeps it below Critical.
  • Remediation: any write-capable QE skill that could run in a non-QE repo needs an approval gate or a repo-type assertion ("refuse unless this is the QE suite"). Keep the no-gate posture scoped to the QE repo.

MEDIUM Silent global shadowing — the wrong skill body can run invisibly

Evidence: CLAUDE.md:172 (already a documented live failure in this ecosystem):

Global skill shadowing | A global `~/.claude/skills/` copy can silently
shadow an enhanced project `.claude/skills/` skill of the same name —
the project body won't execute ... (identical descriptions mask it: triggering
looks fine, the wrong body runs).
  • Why it matters: this is a real failure mode today. Co-locating the full ecosystem multiplies the name-collision surface between the dev repo's .claude/skills/ and the global tree — and the failure is invisible (triggering "looks fine, the wrong body runs"). An agent confidently runs QE logic the operator did not intend and gets no error.
  • Call path / reach: any name collision between project and global skills.
  • Severity rationale: Medium — silent and confusing, but bounded to skill invocations.
  • Remediation: unique namespacing per repo; the existing ls ~/.claude/skills/<name> check becomes mandatory if the trees co-exist.

5 · Tech debt & maintainability — the duplication amplifier

MEDIUM Deliberate duplication multiplies the salience of every QE assertion

Evidence: ~/.claude/CLAUDE.md:94 (a stated design rule):

**Rule:** Skills intentionally duplicate context from CLAUDE.md rather than
cross-importing. CLAUDE.md may not be in context when a skill runs —
duplication is the cost of reliability.
  • Why it matters: the ecosystem by design repeats the same domain facts across CLAUDE.md, the 69-node knowledge tree, and many of the 61 skills. For reliability inside the QE repo this is the right call. But repetition is exactly what raises a statement's weight for an LLM — the same IMA imperative restated five times is harder for a co-located dev agent to discount than one it sees once. The design that makes the toolkit reliable is the same design that makes its assertions sticky in the wrong repo. It is also a maintenance tax: 40,125 markdown LOC + 82 memory files to keep coherent.
  • Call path / reach: compounds Dim 1 and Dim 3 — duplication is the volume's force multiplier.
  • Severity rationale: Medium — amplifier rather than a standalone fault.
  • Remediation: the duplication rule is fine within the QE boundary; the fix is the boundary itself (Dim 1 / mitigation #1–2), not de-duplication.

6 · Dependency / environment health — dangling external assumptions

MEDIUM Hardcoded QE environment refs become confidently-cited dead references in a dev repo

Evidence: CLAUDE.md:163-166 + global tool table — MCP servers and IDs that exist only in the QE setup:

| Jira project | `CI` — tickets are CI-#### |
| Jira cloud ID | `5fa58df5-1af7-4834-b76b-201a98e74211` |
| ADO project | `IGS` — repos: QueryGateway, IMA360, CRM, DB-Industrial |
MSSQL INT → mssql-int · Zephyr → zephyr-scale · Ruby suite path hardcoded in ruby-migration
  • Why it matters: a co-located agent inherits a registry of confident references — specific MCP servers (mssql-*, zephyr-scale), a Jira cloud ID, CI-T test-case keys, the Ruby suite path — none of which a developer's environment is guaranteed to have. The agent will cite them as fact and may attempt to use tools that aren't connected, producing confidently-wrong instructions and failed tool calls rather than admitting the repo is a different context.
  • Call path / reach: any task the agent tries to enrich with "ticket"/"test"/"DB" context.
  • Severity rationale: Medium — misleading and noisy; failures are loud (tool errors) more often than silent.
  • Remediation: strip QE-specific tool/ID tables from any CLAUDE.md that ships into a dev repo; keep them in the test-scoped file.

7 · Context-window cost (performance, reframed)

LOW Footprint: ~23.5K-token eager floor + 129 MB / 2,241 files on disk

Evidence: measured — du -sm .claude = 129 MB; find .claude -type f = 2,241; eager floor = 7,605 + 20,228 + 16,914 + 49,213 ≈ 93,960 B ≈ 23,500 tokens.

  • Why it matters: the eager floor is a fixed tax on every session's budget (modest against a 1M window, non-trivial against smaller ones). The 129 MB / 2,241 files (mostly HTML run reports, run-logs, plans) are a clone/CI/IDE-index tax with near-zero steering risk but real repo-hygiene cost if dropped into an app repo's tree.
  • Call path / reach: every clone, every session, every IDE index.
  • Severity rationale: Low — cost and hygiene, not correctness.
  • Remediation: if anything co-locates, keep generated artifacts (runs/, run-log/, plans/) out of the dev repo; ship only the skill/knowledge text the agent needs.

Mitigation Strategy — if we proceed (ranked by leverage)

Containment controls, cheapest-correct first
  1. Don't co-locate the full ecosystem (the real answer). Keep the QE ecosystem in the QE repo. If a developer needs QE skills, ship only the portable subset via the qe-essentials installer — never the QE CLAUDE.md, the knowledge tree, or MEMORY. This is precisely the isolation TDD-022 verified the toolkit already has; the risk in this report exists only in the co-located configuration, so not co-locating closes almost all of it.
  2. If co-location is unavoidable, scope the CLAUDE.md. The dev repo's top-level CLAUDE.md must describe the app. Move QE conventions into an explicitly test-scoped file the agent loads only for test work — kills Dim 1 (the precedence floor) and Dim 6 (dangling refs).
  3. Re-label test-harness facts as test-harness facts. One clause per "Known Conventions" row ("in TESTS …", "scopes test data, not production") — directly neutralizes the Critical (Dim 3).
  4. Namespace + tighten triggers; mark IMA-coupled skills project-scoped. So they are absent from a dev repo and don't fire on bare dev verbs — closes Dim 3 (collisions) and Dim 4 (shadowing).
  5. Gate the autonomous writers outside the QE repo. Any write-capable skill that could run in a non-QE repo asserts repo-type or requires approval before editing — keeps the no-gate posture safely scoped (Dim 4).
  6. Adopt the boundary doc TDD-022 already recommended (a SHARE / NEVER-SHARE list) plus an optional export guard, so the discipline is enforced rather than convention.

Net: the developer's instinct is right — co-locating this volume is a genuine agent-steering risk, driven less by raw line count than by an always-on, OVERRIDE-framed QE floor and a body of test-truths an app agent can't distinguish from app-truths. But it is fully containable, and the lowest-cost control (#1) is the one the platform was already designed around.
