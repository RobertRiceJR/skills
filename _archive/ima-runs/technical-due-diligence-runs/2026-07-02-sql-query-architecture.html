Technical Due Diligence · QE · TDD-030

SQL Query Architecture — src/utils/sql

Where the suite's SQL calls go (CRM vs Industrial), what they cost, which entities they touch — and a phased plan to make the layer faster, observable, and single-sourced. Includes live timing from 10 sanity-run histories and both INT plan caches.

Prepared for: Robert Rice (QE) Date: 2026-07-02 Scope: src/utils/sql (21 files, ~5,080 LOC) + all call sites Prior runs: TDD-015…018, TDD-027
Critical — 0
No data-loss, auth, or money-path defects. The layer is read-only by construction.
High — 3
Unbounded hot-path fetches · 12–40s query family at the timeout cliff · three unreconciled SQL surfaces.
Medium — 8
Health-check blind spot, per-call Key Vault + pool churn, plan-cache fragmentation, empty-pick crashes, stale docs.
Low — 7
Typed-only numeric interpolation, magic literals in relocated SQL, vacuous HAVING, misleading connect log.
Verdict — the three findings that matter most

1. The hottest path is the most wasteful. sqlDB.getRandomParty has 43 call sites (15% of all SQL usage) and its query, getCiParties, has no TOP clause — it ships ~10,300 rows / 81k logical reads per execution to pick one row client-side. The INT plan cache shows 67 executions of exactly this query (134s cumulative). Four sibling finders repeat the pattern, one returning 38,299 rows. A one-line TOP fix per query. → H2

2. Five queries sit at the timeout cliff, and the framework is blind to them. The heavy family averages 12–20s with observed maxima of 35–42s against a 60s request timeout, drifting 2–3× run-to-run; the plan cache confirms one at 35.2s average. Outside the sanity spec there is zero per-query timing or error labeling — when one of these times out mid-run, it will present as an anonymous flake. → H1, M5

3. Industrial SQL is fragmented across three surfaces and its finder SQL is unchecked. Two disjoint Industrial catalogs (queries.industrial.ts vs inline SQL in finders/industrial.ts), plus 10 spec/factory files opening raw pools — and none of the 15 inline finder queries (including the CI-7868 seasonal-truth twin) is covered by the SQL sanity health check. Every Industrial call also pays an uncached Key Vault round-trip + fresh connection handshake. → H3, M1, M3

Dimension 1 — Architecture, coupling, boundaries

HIGH · H3 Three unreconciled SQL surfaces; the Industrial catalog is split in two

Evidence: finders/industrial.ts:24-28 inlines its own SQL (SELECT DISTINCT PartyAccountId FROM IMA.ServiceLocations WHERE PartyAccountId IN (${'{'}idList{'}'}) …) while queries/queries.industrial.ts holds 14 other Industrial builders consumed by only 2 e2e specs + the sanity spec — two disjoint catalogs for one database. Meanwhile 10 files bypass the facade entirely: dataFactory.ercot.account.ts:6, deal-sheet-pending-hedges.api.spec.ts:3 (+4 sibling api specs), contract.contacts.e2e.spec.ts:6, contract.record.overview.e2e.spec.ts:7, party-addresses.api.spec.ts:6 (buildSqlConfig raw) — each running inline SQL on a self-opened pool. A third surface, database.fixture.ts:1 → @igs-quality/ima-queries, brings its own pool and teardown.

Why it matters: conventions enforced in the facade (CI filter, constants, fragments, sanity coverage, future instrumentation) silently don't apply to ~10 files; a convention change (e.g. the CI date window) now has three places to miss. The /sql-data skill's reuse-first contract can't see surface B at all.

Call path / reach: surface B files hit Pending.Deals/Hedges, IGS_AppData.IGS.EsiIds, dbo.[Customer Contacts] — entities the facade never touches, so drift there is invisible to every audit keyed on sqlDB.

Severity rationale: High — boundary erosion with compounding cost; no data risk (all read-only).

Remediation sketch: declare queries.industrial.ts + finders/industrial.ts a single catalog (finders import builders); promote recurring inline spec SQL into finders via the existing sql-helper backlog; document surfaces B/C as sanctioned exceptions or migrate them.

LOW · L6 Dead references to a facade member that no longer exists

Evidence: opportunities.api.spec.ts:719,735 — commented blocks call await sqlDB.executeRaw(`…`); no executeRaw exists on the barrel (sql.db.ts:19-27) or any finder module.

Why it matters: uncommenting the "ready-made" verification block fails to compile; misleads the next author about the facade's API.

Remediation sketch: delete the blocks or rewrite against getMergedOpportunityHistory.

Checked and clean: the barrel spread (7 modules, no key collisions — verified export-name union); per-worker pool singleton is sound under Playwright's process-per-worker model (core.ts:9-21); config duplication from TDD-016 is resolved via buildSqlConfig.

Dimension 3 — Security surface & trust boundaries

LOW · L1 Numeric interpolation relies on compile-time types only; two TOP interpolations skip the sibling's sanitizer

Evidence: queries.party.ts:335-336 (WHERE e.PartyId = ${'{'}partyId{'}'} AND e.EventTypeId = ${'{'}eventTypeId{'}'}), queries.industrial.ts:88,145,204, queries.opportunity.ts:529-560 (count lookups), getSuccessfulDealSheetExports(days) at queries.opportunity.ts:812, and finders/industrial.ts:94,112 interpolate ${'{'}topN{'}'} raw while the sibling sampleSlcaContractsWithLegalEntity:267 wraps it in parseInt(). All callers are internal and typed; toIdList and Number(topN) patterns already exist as the fix.

Why it matters: TypeScript types vanish at runtime — a future caller passing request-derived input through an any creates a textbook injection seam on a service account with broad read grants. Cheap to close uniformly.

Remediation sketch: lint-style sweep — every arithmetic interpolation goes through Number()/parseInt() or becomes a bound ParameterizedQuery param (also fixes M4).

LOW · L2 trustServerCertificate: true remains blanket-on (carried from TDD-016)

Evidence: sql.config.ts:21 — now single-homed and documented ("required for the internal self-signed DB certificates on INT/QA"). Carried as accepted risk; revisit only if a CA becomes available.

Checked and clean: every user-string input is bound via ParameterizedQuery (TDD-017 F1 remediation verified in party/account/opportunity/communications modules); credentials flow Key Vault → config, never literals; the RAD proc runner whitelists proc names via a union type (finders/industrial.ts:289) and int-sanitizes TVP values; the LIKE-wildcard caveat on getPartiesWithNoteContainingWord is documented at queries.party.ts:442-444.

Dimension 4 — Error handling & failure modes

MEDIUM · M2 Inconsistent empty-result contract — bare pick() crashes with a context-free FakerError

Evidence: finders/core.ts:30 — export const pick = <T>(records: T[]): T => faker.helpers.arrayElement(records);. Verified live: faker.helpers.arrayElement([]) throws "Cannot get value from empty dataset." Roughly half the finders guard (if (records.length === 0) throw new Error('No merge-safe duplicate party pair found…'), parties.ts:92) — but the other ~60 don't (getRandomParty parties.ts:8-12, getActiveParty, getRandomContract contracts.ts:6-10, most of opportunities.ts). getTaskById (parties.ts:180-184) returns records[0] as a non-nullable type — undefined propagates.

Why it matters: the single most common SQL failure in this suite is "pool ran dry" (M6 proves it happens). Guarded finders fail with an actionable message; unguarded ones fail with a faker error naming no finder, no query, no environment — triage cost on every occurrence.

Remediation sketch: make pick take a context label — pick(records, 'getRandomParty') — and throw the friendly message centrally; fix getTaskById's type or add a throw.

LOW · L5 "Connected to SQL" is logged before the connection exists

Evidence: sql.client.ts:26-29 — this.poolConnectPromise = this.pool.connect().then(() => this.pool); logger.info('Connected to SQL'); — the log fires synchronously after starting the connect; a refused connection logs "Connected" then the error.

Remediation sketch: move the log into the .then(); include server/database.

Checked and clean: Industrial finders correctly close pools in finally (finders/industrial.ts:31-33 et al.); connect() resets the memoized promise on failure so a transient outage doesn't poison the worker (sql.client.ts:31-35); toIdList([]) returns 'NULL' so empty IN-lists are valid SQL, and the id-list filters short-circuit on empty input.

Dimension 6 — Dependency health & version risk

MEDIUM · M3 (dependency half) azure-secrets-helper@0.0.11 performs a full credential-chain + Key Vault fetch on every call — no cache

Evidence: package dist source, azure-credential-helper.js:171-175: async function getServiceAccount(secretName = 'service-account') {'{'} const secret = await getSecret(secretName); … {'}'} — no memo, no module-level cache (grep for cache/Map: 0 hits). buildSqlConfig (sql.config.ts:12) calls it unconditionally, and getIndustrialPool (sql.industrial.config.ts:16-19) calls buildSqlConfig per invocation.

Why it matters: every Industrial finder call = credential-chain resolution (AzureCliCredential first) + HTTPS Key Vault round-trip + fresh TCP/TLS/NTLM pool handshake, then teardown. The CRM side fixed exactly this in TDD-015 (core.ts:9-13 comment: the old pattern "re-fetched the Key Vault secret each time"); Industrial kept it by explicit decision (TDD-016 F4 "intentionally not memoized yet") — that decision is now the layer's dominant fixed cost, multiplied by the HU finders that open two Industrial pools sequentially per call (accounts.ts:59-62).

Severity rationale: Medium — pure latency/reliability tax (Key Vault throttling is a real transient-failure source), no correctness risk.

Remediation sketch: memoize the secret in buildSqlConfig (module-level promise, mirrors core.ts) — one edit serves both DBs and all bypassing files; optionally memoize the Industrial pool per worker with a closeIndustrialClient() twin.

Checked and clean: mssql loaded via createRequire to dodge the ESM/CJS boundary — functional, duplicated (L7); @azure/keyvault-secrets/@azure/identity ride inside the helper package, not direct deps of this repo. No known-vulnerable versions in the SQL path.

Dimension 7 — Performance hotspots

HIGH · H1 A five-query family runs 12–20s average, 35–42s max — against a 60s timeout, with 2–3× run variance

Evidence: history.json (10 runs): getPartiesWithMultipleSalesRepActivities 20.2s avg / 40.3s max; getPartiesWithMultipleEventTypes 19.3s / 38.0s; getPartyEmailHistoryByPartyNumber 18.1s / 35.2s for 2 rows; getPartiesWithUserProfiles 12.0s / 41.9s; getCiContractWithUlobLinkedAccounts 14.1s. Plan cache corroborates server-side: the EventNotes contains-word shape at 35.2s avg / 1.32M logical reads, getRenewalNonRenewalMergePairs at 15.97M logical reads per execution.

Why it matters: a query whose max is 42s under a quiet INT is one noisy-neighbor episode (cf. CI-7868's 17h Choice_OpsEng burst) away from the 60s wall. Because failures are unlabeled (M5), the result reads as a random E2E flake, not "known-heavy query timed out."

Call path / reach: the email-history and multi-activity finders back party filter/history E2E tests; the merge-pair family backs the merge suite — high-traffic areas.

Severity rationale: High — active flake-risk on core suites; measured, not modeled.

Remediation sketch: per-query rewrites: EXISTS instead of triple-LEFT-JOIN aggregates; resolve PartyId first then seek Contacts/Contacts_arc by PartyId; DISTINCT+EXISTS for L4; keep the documented 3.6s→16s merge-predicate tradeoff as-is (already a reasoned decision).

HIGH · H2 Unbounded candidate fetches on the hottest path — thousands of rows shipped to pick one

Evidence: queries.party.ts:11-16 — getCiParties has no TOP: SELECT p.PartyId, p.PartyNumber, p.PartyName FROM Party.Parties p WHERE p.RelationshipManagerId = 1 AND ${'{'}dateWindow(){'}'}. History: 10,302 rows avg. Plan cache: 67 executions, 81,345 reads each, 134s cumulative — the most-executed suite query on the server. Consumed by getRandomParty (43 call sites) and sampleParties (parties.ts:246-252, which shuffles the full set client-side). Same shape: getPartiesWithNotes (6,702 rows, :19-26), getPartiesWithCallOutcomes (5,629 rows, :29-35 — also missing the date window), getPartiesWithMultipleSalesRepActivities (1,240 rows, no TOP), getIndustrialContractTermStatus (38,299 rows, queries.industrial.ts:55-75).

Why it matters: every one of the 43 getRandomParty call sites pays ~2s server time + a 10k-row transfer + client materialization to select one random party. Randomness does not require the full set — TOP 200 … ORDER BY NEWID() preserves the sampling contract (the file's own siblings already do this).

Severity rationale: High — largest aggregate cost in the layer, trivially fixable, zero behavioral change.

Remediation sketch: add TOP 200 … ORDER BY NEWID() to the five unbounded queries (larger TOP for sampleParties' max batch); done.

MEDIUM · M3 Industrial pays pool-per-call + Key Vault-per-call; HU finders stack two of them after a CRM query

Evidence: sql.industrial.config.ts:16-19 — const config = await buildSqlConfig(…); return new sql.ConnectionPool(config).connect(); per call, and every finder closes in finally. getProspectCiElectricAccountWithNoHu (accounts.ts:55-64) chains: CRM query → filterAccountIdsWithServiceLocations (pool #1 open/close) → filterAccountIdsWithoutStagedHu (pool #2 open/close). 17 facade call sites + 10 bypass files pay the same fixed cost. MEDIUM · M4 Interpolated literals fragment the plan cache; the one parameterized query proves the fix

Evidence: CRM plan cache shows 4 separate entries for the getCompatibleOpportunityByULOB shape (ulobId/excludePartyId inlined at queries.account.ts:137-152) and 2 for getCiPartyWithNoOpportunitiesExcluding — one compile + one cache slot per distinct literal.
