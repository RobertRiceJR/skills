Verdict — the three findings that most change whether this gate is trustworthy

1. The deterministic checks false-positive (F-SEC1, High). The greps are comment- and string-blind. Run #1's S2 fired on two // comment lines mirroring the old Ruby assertion — zero real violations — yet the band table mechanically dropped PASS→WARN and a human had to override it. A gate that cries wolf on its first run stops being read.

2. The reuse model has a subset/superset blind spot (F-D1, High). A spec block that is a strict prefix of an existing composite is neither REUSE nor EXTEND → forced NEW, and "never disturb existing members" forbids the clean fix. Run #1 produced openCreateAccounts() duplicating 3 lines of openNewAccountForm() — and DUP can't see it. The headline feature accretes near-duplicate composites every run.

3. The validator is itself unvalidated (F-A1 + F-T1, High). The check regexes are duplicated across the rubric and skill.md (two SSOTs that can drift), and there is no calibration corpus — so "any run gets the same answer" is true but "the same answer is correct" was never tested. Run #1 was the first exercise and immediately found a false positive.

Phase 1 — Surface Map

Entry points (the judge's "API"). Phase 5a (static score) → 5b (haiku narrative) → 5c (log to both). Inputs: the final spec + POM files and the Phase-4 type-check exit code. Outputs: a PASS/WARN/FAIL band, an appended reference/grades.jsonl line, and the populated runs.<name>.jsonl entry (quality/well/poorly/improve).

Traced data flow. Five greps + one boolean → grade-band table → quality. The band is asserted authoritative ("Only this table sets the band … fully reproducible", skill.md:211). The haiku output is explicitly non-authoritative and is parsed into the same logs.

Change-radius / blast surface. (a) The band feeds two persisted logs and the user-facing Output Summary — a wrong band propagates to the trend file. (b) The check definitions exist in two files (rubric §A and skill.md §5a) — the highest-coupling hazard. (c) The reuse decision (Phase 2) is what the DUP check later audits — and they disagree about what "duplicate" means.

Dimension 1 — Architecture, coupling, boundaries

HIGH F-A1 — Check definitions are duplicated across the rubric and the skill body (two SSOTs that can silently drift)

Evidence: conformance-rubric.md:17 and skill.md:204 state the same regex independently:

rubric §A:  grep the spec for  page\.(getByTestId|getByRole|getByText|getByLabel|...|locator)\(
skill 5a:   S1 spec: Grep      page\.(getByTestId|getByRole|getByText|getByLabel|...|locator)\(

Why it matters: the whole value of a gate is a trustworthy, single definition. Any fix to a pattern (e.g. the comment-stripping F-SEC1 needs) must be made in both places or the documented rubric and the executed check diverge — the exact "duplicated-thing drift" the repo's own bug-patterns #19 / CLAUDE.md warns against.

Call path / reach: rubric §A is cited by skill.md:197 as the source of truth, yet skill.md:204–209 re-lists the patterns instead of referencing them.

Severity rationale: High — credibility/maintainability of a gate; drift produces silently-wrong grades.

Remediation: keep the regexes only in rubric §A; skill.md Phase 5a says "run the §A checks" with no inline patterns.

NOTE Band-authority tension (folded into F-SEC1 / F-E1)

The rubric asserts the band is authoritative and reproducible (skill.md:211; rubric:37), but run #1's band had to be manually overridden ("mechanically WARN ⚠️, substantively PASS ✅"). Authority a human must correct on run #1 is not authority. Root cause is F-SEC1; recovery gap is F-E1.

Dimension 2 — Test coverage, quality & gaps (of the judge itself)

HIGH F-T1 — The validator has no calibration corpus; "reproducible" is conflated with "correct"

Evidence: conformance-rubric.md:12 — "Each is a single reproducible grep / read — any run gets the same answer." There is no fixture set of known-good / known-bad spec+POM pairs the rubric is checked against.

Why it matters: deterministic ≠ accurate. A check can return the same wrong answer every time. Run #1 was the judge's first exercise and immediately surfaced a false positive (F-SEC1) — there was no mechanism that would have caught it earlier.

Call path / reach: the band feeds grades.jsonl (the stated trend-analysis source) and the run-log quality; an uncalibrated check pollutes both.

Severity rationale: High — an unmeasured gate gives false confidence precisely where the user wants a guardrail.

Remediation: add a tiny calibration corpus (2–3 existing verify-method POMs that must grade PASS, 1 crafted violator that must WARN/FAIL) and a "rubric self-test" runnable standalone or under /sync.

MEDIUM F-T2 — P1's over-strictness stayed unexercised even by a "passing" run

Evidence: the run #1 target CreateAccountsPage has zero expect(; a prior sweep of src/pages/ found 25+ POMs with legitimate await expect(this.x).toBeVisible() in verifyLoaded-style helpers (e.g. create.party.page.ts:58, dashboard.page.ts:69). P1 (rubric:19, grep POM for \bexpect\( → 0) would false-FAIL every one of them.

Why it matters: the planned validation could not trip P1, so a known latent false-positive survives a green run — it will fire on the next verify-method POM.

Severity rationale: Medium — latent, but certain to fire; same defect class as F-SEC1.

Remediation: drop P1 from the deterministic set (visibility-vs-business expect isn't reliably greppable); move "no business assertions in POM" to the haiku qualitative dimension.

Dimension 3 — Trust boundaries & input handling

HIGH F-SEC1 — The greps trust raw file text: they count comments and strings as violations

Evidence: conformance-rubric.md:18 defines S2 as grep the spec for expect\(\s*page\. → 0 matches. Run #1's two hits were both comment text mirroring the Ruby assertion:

accounts.create.e2e.spec.ts:97   // `expect(page.accounts).to all(have_attributes(status: 'Ready'))`
accounts.create.e2e.spec.ts:152  // `expect(page.accounts).to all(have_attributes(status: 'Ready'))`

Zero real TypeScript assertions on raw page. locators. The grade-band table (rubric:34) then drops PASS→WARN on these phantom hits.

Why it matters: this is the gate's credibility, which is its entire value. A line-oriented, comment-blind matcher over-reports; the first false WARN trains the reader to ignore the grade — and an ignored gate is a dead gate. It bit on run #1.

Call path / reach: applies to S1, S2, P1, P2 — every text-grep check. P1 has the same exposure via doc-comment expect( examples.

Severity rationale: High — the top finding; it directly defeats the deterministic gate's purpose.

Remediation: strip // line-comments and /* */ block comments (ideally string literals too) before counting; on any non-zero count, print the offending lines so a phantom is visible at a glance.

MEDIUM F-SEC2 — Haiku output is logged with no schema validation; a non-schema field already crept in

Evidence: conformance-rubric.md:69 fixes the grades.jsonl keys; run #1 appended a non-schema note field "to capture the S2 false-positive." The judge persists subagent/operator output without enforcing the contract.

Why it matters: grades.jsonl exists for trend analysis; unvalidated key drift erodes that trend over runs (the same way ad-hoc columns rot a ledger).

Severity rationale: Medium — slow-burn data-quality erosion of the stated purpose.

Remediation: validate the haiku JSON against the §C schema before append; either reject extra keys or formally add an optional note to the schema.

Dimension 4 — Error handling & failure modes

MEDIUM F-E1 — Haiku failure is handled; a wrong static check is not

Evidence: skill.md:219 gracefully degrades the narrative ("If the subagent fails or returns junk, proceed with — "). But skill.md:211 computes the band purely from grep counts with no recovery path when a count is itself wrong — the only fallback that existed on run #1 was a human override.

Why it matters: the asymmetry is backwards — the authoritative half (the band) has no safety net while the non-authoritative half (haiku) does.

Severity rationale: Medium — depends on F-SEC1 to trigger, but it's the missing guardrail that let the bad band stand.

Remediation: after F-SEC1's comment-strip, add a band sanity-gate: any failing check must echo its raw (post-strip) matches; a zero-real-match "failure" cannot lower the band.

LOW F-E2 — The haiku field-budget contract is unenforced

Evidence: conformance-rubric.md:57 caps well at ≤15 words; run #1's subagent returned ~30 and the operator trimmed by hand. The prompt states the cap; nothing enforces it.

Why it matters: log-field bloat and a manual cleanup step on every run.

Severity rationale: Low — hygiene only; band unaffected.

Remediation: truncate/normalize the haiku fields to budget before logging, or relax the cap to what the model reliably hits.

Dimension 5 — Tech debt & maintainability (the rule-set the judge enforces)

HIGH F-D1 — Reuse model blind spot: a strict prefix of an existing composite is forced to NEW, and the fix is forbidden

Evidence: the method match key is "the full ordered sequence" (skill.md:118–120); EXTEND is only "differs by a value" (skill.md:113,122); and "Never disturb existing members" (skill.md:147,263). Run #1 result — verbatim 3-line overlap:

openNewAccountForm() // POM:95-100   flyoutCreate → summaryCreateAccount → waitForURL → newAccountAction
openCreateAccounts() // POM:138-142  flyoutCreate → summaryCreateAccount → waitForURL        (NEW)

Why it matters: the skill's headline feature is reuse, yet a subset/superset relationship — the most common composite overlap — slips through as NEW, and the constraints actively forbid refactoring openNewAccountForm to call openCreateAccounts. DUP sees two different bodies → "unique." Across many runs this accretes near-duplicate composites that the gate reports as clean.

Call path / reach: Phase-2 reconciliation feeds the Phase-5 DUP check; they disagree on "duplicate," so the debt is invisible to the grade.

Severity rationale: High — systematic DRY-debt accretion in the feature the skill exists to provide.

Remediation: add a COMPOSE disposition — when a move-block is a strict prefix/subset of an existing composite, extract the shared prefix and have both call it; grant a scoped exception to "never disturb existing members" for this case. Teach DUP to flag prefix-overlap, not just identity.

MEDIUM F-D2 — Internal contradiction: who owns the waiter wiring

Evidence: Purpose says the spec is "left owning … the waiter wiring" (skill.md:29–30), but the disposition table says "Move the whole triad" into a POM method (skill.md:101). Run #1 resolved it (action-triggered waiters → POM saveAccounts() at POM:167–171; navigation/load waiters → spec) but the doc never states that split.

Why it matters: two runs can split waiters differently → inconsistent POMs, and no deterministic check detects either choice.

Severity rationale: Medium — ambiguity that drives inconsistent refactors.

Remediation: state the split explicitly — action-triggered waiters → POM method (waiter pattern); navigation/page-load waiters → spec. Fix the Purpose line to match.

MEDIUM F-D3 — No rule for "same intent, different locator mechanism" → soft twins DUP can't catch

Evidence: selectStagingRow (getByRole('row').filter({hasText}), POM:120–122) vs selectDuplicateStagingRow (getByRole('row',{name: /Duplicate.*n/}), POM:174–176) are behaviorally adjacent. The match key is the selector string (skill.md:117), so different selectors = "different" = NEW, and DUP treats them as unique.

Why it matters: a second, softer class of duplication accretes invisibly — same user intent, two methods.

Severity rationale: Medium — narrower than F-D1 but the same blind-spot family.

Remediation: default such cases to NEW but surface them as a possible-duplicate in the haiku Reuse dimension, so a human sees the adjacency.

LOW F-D4 — The pom-variable example contradicts the "match local convention" rule

Evidence: skill.md:160 hardcodes const pom = new XxxPage(page); the style rule at skill.md:42–43 says match surrounding siblings. Run #1 correctly used createAccountsPage to match the sibling test — against the literal example.

Why it matters: minor friction; the example and the rule point opposite ways.

Severity rationale: Low — cosmetic.

Remediation: note in skill.md:160 that the variable name yields to local sibling convention.

Dimension 6 — Dependency health & version risk

LOW F-DEP1 — The judge inherits ripgrep's line-orientation and Explore's excerpt-reading

Evidence: the checks are Grep (ripgrep) calls — line-oriented, which is the proximate cause of F-SEC1's comment-blindness; a real fix needs a comment-strip pre-pass or a small tokenizer, not a richer regex. The narrative uses Agent subagent_type:"Explore" (skill.md:216), which "reads excerpts" and could under-read a longer POM.

Why it matters: the toolchain shapes what's checkable; pretending greps are AST-aware is how F-SEC1 shipped.

Severity rationale: Low — haiku is non-authoritative; the grep limit is captured under F-SEC1.

Remediation: implement the comment-strip as an explicit pre-pass; instruct the subagent to read the full (small) files, or use a general-purpose haiku agent if Explore under-reads.

Dimension 7 — Performance hotspots

No material findings — one haiku subagent plus ~5 greps and one type-check per run. Cost is negligible and bounded; the static checks are O(file size). Traced: no loops, no fan-out, no repeated reads.

Bottom line. The judge's architecture is right — static score + non-authoritative haiku, logged, no auto-edit. The defects are in the implementation of "deterministic": text-greps that can't tell code from comments, a band with no recovery when a check is wrong, a reuse model that can't see prefix-overlap, and a validator with no fixtures of its own. Fixes 1–4 convert it from "passed its first run by being overridden" to a gate you can trust unattended.
