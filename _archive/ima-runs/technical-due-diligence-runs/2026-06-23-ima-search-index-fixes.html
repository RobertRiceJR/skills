These five fixes are real and the diagnoses are largely sound — but three concerns dominate whether they can be picked up as written this sprint:

  1. CI-7863 is the highest-volume failure and the only fix with no agreed direction. ≥250 dead-letters/30 days, firing nightly and during business hours.   2. Three of the five fixes do not line up with main.   3. CI-7864 is mis-scoped and its blast radius is wider than the ticket states. The indexing story — one transport, two write paths, one failure mode

Every ima-* index is populated by two paths that do not always agree, and all five fixes sit on this backbone:

  • Bulk reindex (the "nightly rebuild" the opportunity/party fixes touch): a Control-M job → Ima*ReindexingService → GetIdsAsync(), membership = the C&I-gated SQL.
  • Incremental RAD CDC (the "build seam" the accounts fixes touch): a SQL Rad_TR_* trigger batches changed PKs → Service Broker → RAD host → RabbitMQ topic exchange → IMA_RAD_CrmChangeQueue (500 party IDs/message) → table gate → resolver maps leaf keys to root IDs (party → opportunity / account IDs) → GetEntitiesByIdsAsync(ids) hydrates the document → ES.

The shared failure mode this report is really about: a hydration/resolver step runs a slow or unbounded query → it exceeds the EF Core default 30-second command timeout (no override on these paths) → the exception propagates through DataChangeResponderService, which has no try/catch around its Task.WhenAll across index handlers → the RAD consumer faults → the message is nacked, the dead-letter exchange retries once, the second attempt fails identically, and the batch is dropped. Net effect: the ES document goes stale for up to 24 hours (until the next nightly bulk reindex heals it); the SQL system-of-record is untouched. This is data staleness, not data loss — but because one slow query can fault the whole batch's Task.WhenAll, a single bad party can stale several indices at once.

One root cause, five shapes — and a failure handler that amplifies it

All five are the same class: an unbounded or unscoped query under nightly-rebuild density, in five flavors — row-cap truncation (CI-7868), a synchronous blocking long-pole (CI-7869), cross-join / self-join expansion (CI-7862), a distributed linked-server query (CI-7863), and a no-date-filter full-history scan (CI-7864). What turns each slow query into a dropped batch across multiple indices is the handler gap flagged in TDD-003 and re-confirmed here: DataChangeResponderService wraps its per-index work in a Task.WhenAll with no try/catch (:58-96), so one unhandled timeout faults the whole message → nack → DLX-retry-once → identical failure → dropped. Even after all five query fixes ship, that handler still converts any future slow query into the same multi-index staleness — worth a separate hardening ticket (isolate per-index failures; structured nack/backoff).

  • CI-7828 (LVT poison message, HIGH) — same index (ima-opportunities) and same fault→nack→stale mode as CI-7862/7863; the resolver lacks an LVT branch → default throw. A sibling opportunity-indexing defect on this transport (confirmed in TDD-003).
  • CI-7829 (rep-rename not reindexed, MED) — globalsearch, not in this set, but the same accretion-gap class (a table added to the data fetch but not the resolver/AppliesTo). Watch for it when adding the new segment lookup (CI-7863) or the activity cutoff (CI-7864).
  • CI-7764 (RAD consumer deadlock, High) — distinct but adjacent: a write-side MERGE … HOLDLOCK deadlock on the Industrial UPSERT end of the same RAD/RabbitMQ backbone. Same transport, different layer; not overlapping with these read/hydration timeouts.
  • TDD-003 (2026-06-16) — the prior indexer review: 2 confirmed (CI-7828, CI-7829), 3 refuted, review-only. TDD-008 extends that baseline; it does not re-litigate the refuted three.

What these fixes do NOT address

BPL flooding / queue saturation (out of scope, by design). A bulk party-load operation fires the Rad_TR_* triggers and turns one logical change into a flood of per-id RAD messages, which the consumer then processes at ~5,000-way concurrency (100 receivers × prefetch 50) — a throughput/concurrency problem. These five are per-message query fixes: they lower the per-message failure rate but change neither the message volume a bulk op generates, the consumer concurrency, nor the DLX-retry-once behavior. That belongs to the CI-7764 fix family and the trigger/Control-M batching design.
