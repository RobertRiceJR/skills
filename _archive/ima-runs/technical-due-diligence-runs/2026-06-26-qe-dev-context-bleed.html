The QE → Developer Context-Bleed Boundary

A risk-mitigation review of sharing QE Claude skills with developers without Claude absorbing QE context and "getting confused about what a developer needs to do." Prompted by direct junior-developer feedback after testing the platform in a developer repo.

Prepared for: QE platform owner + dev stakeholders Date: 2026-06-26 Scope: claude-skills platform × IMA QE passive surface
0 Critical 2 High 3 Medium 1 Low
Active context (skills)
Opt-in, trigger-gated. Only fires on a matching prompt. Lower bleed risk — if triggers are scoped and the skill is portable. This is what the toolkit ships.
Passive context
Auto-loaded, always in the window: CLAUDE.md, the knowledge tree, MEMORY.md. Absorbed without being asked — this is where the junior dev's fear actually lives.
Blast radius
GLOBAL ~/.claude/skills/ = visible in every repo/session. PROJECT .claude/skills/ = scoped to one repo. The install target decides the reach.
Verdict — the three things that matter most

Bottom line: the platform is already well-isolated by construction — the bleed the developer fears is not what the toolkit ships. The shared toolkit is 9/9 scope: portable (domain-agnostic), and the installers copy only qe-essentials/skills/* — never the QE CLAUDE.md, the 17,878-line knowledge tree, or MEMORY. The risk is real but narrow, and it lives in the gaps below, not in the toolkit.

  1. Isolation is convention, not enforced (High). Nothing in the platform prevents the QE passive surface from being copied or loaded into a developer's context — no "what never leaves the QE repo" boundary, no guard. The discipline holds today only because no one has broken it.
  2. init-qe-repo is the real "Claude gets confused" vector (High). It is globally invocable, fires on broad triggers like "bootstrap testing for this repo," and writes a CLAUDE.md + knowledge tree + husky + settings into whatever repo it's pointed at — turning a developer's app repo into a QE-shaped repo.
  3. Global-only install = maximal blast radius (Medium). The toolkit lands in ~/.claude/skills/, visible in every repo and session, with no project-scoped install lever. The 9 skills are generic so the harm is mild — but the lever to contain them is missing.

1 · Context isolation & boundaries

HIGH Isolation is convention, not enforced — no "never leaves the QE repo" boundary

Evidence: qe-essentials/install-toolkit.ps1:33-44 (copies only the skills dir) — and the absence of any export-guard anywhere in the platform.

Get-ChildItem -Path $skillsDir -Directory | ForEach-Object {
    $name = $_.Name
    $dest = Join-Path $target $name
    ...
    Copy-Item -Recurse -Force -Path $_.FullName -Destination $dest

Why it matters: The installer is correctly narrow — it copies qe-essentials/skills/* and nothing else, so it never ships CLAUDE.md, the knowledge tree, or MEMORY. That is the good news. The gap: this is the only thing keeping QE passive context out of a dev's session, and it holds by accident of what the script happens to copy. Nothing states what must never be shared, and nothing prevents a well-meaning person from cp -r-ing the knowledge tree into a dev repo, pointing Claude at the IMA repo as cwd while "helping" on dev work, or pasting QE conventions into a global ~/.claude/CLAUDE.md (which would then load in every repo).

Call path / reach: The junior dev's exact fear realizes through this gap, not through the toolkit.

Severity rationale: High — it's the controlling boundary for the entire concern, and it's undocumented and unenforced.

Remediation: Add a one-page playbook/context-boundary.md — an explicit SHARE / NEVER-SHARE list — and state the rule in the toolkit README. Optionally a guard (below).

SOLID The scope-discipline control works — credit where due

Evidence: all 9 toolkit skills scope: portable (grep, 9/9); playbook/metadata-schema.md + placement-rubric.md; promote-skill/SKILL.md gate.

$ grep -rh "scope:" qe-essentials/skills/*/SKILL.md | sort | uniq -c
      9   scope: portable

Why it matters: The scope facet (portable vs project) is a real, load-bearing isolation control: promote-skill refuses to put a project-coupled skill on the shared shelf, so the toolkit cannot accumulate QE-domain skills by design. This is why the verdict is reassuring rather than alarming — the contamination most people would fear (QE orchestrators, tickets, test cases riding along) is structurally excluded.

Maintainability note: the control is only as good as its enforcement — see Dimension 5.

2 · Install blast radius & trust boundary

HIGH init-qe-repo is global, broadly-triggered, and writes QE substrate into any target repo

Evidence: ~/.claude/skills/init-qe-repo/SKILL.md:12-16, :7-9, :63-66, :53-61 (lives in the GLOBAL tree — confirmed by directory listing).

metadata:
  kind: orchestrator
  scope: portable          # globally available, invocable in ANY repo
  ...
  Use when someone says "set up a QE repo", "scaffold the QE ecosystem here",
  "init qe repo", "bootstrap testing for this repo", or "stand up the substrate".

### Step 3 — Stamp repo-root conventions
- <repo>/CLAUDE.md from CLAUDE.md.template (fill project/slug/repo/framework).
- <repo>/TODO.md from TODO.md.template.

Why it matters: This is the single most likely path to "Claude got confused about what I need as a developer." It's in the global tree, so it's offered in every repo; its triggers ("bootstrap testing for this repo") are exactly what a developer might say when they just want a test or two. When it fires it writes a CLAUDE.md, a knowledge tree, husky hooks, and settings.json into the dev's repo (Steps 2–4) and installs the toolkit globally (Step 1, :47-51) — converting an app repo into a QE-shaped repo.

Mitigants already present: it stamps fresh, templated substrate filled from the target repo's own discovery — it does not copy IMA's actual CLAUDE.md or knowledge content — and it is "idempotent-ish," refusing to overwrite a customized CLAUDE.md (:100-101). So it imposes QE shape, not IMA content. That keeps it out of Critical.

Severity rationale: High — broad trigger + structural writes into a dev repo from the always-present global tree. Recoverable via git (no data loss), so not Critical.

Remediation: (1) a loud confirm-gate that lists every file it will write before writing; (2) re-scope the trigger phrasing so "add a test" / "write a test" does not match (reserve it for explicit "stand up a QE repo"); (3) consider making it project-tree-only rather than global.

MEDIUM Toolkit installs to the GLOBAL tree only — no project-scoped option

Evidence: qe-essentials/install-toolkit.ps1:~20 (and install-toolkit.sh: TARGET="$HOME/.claude/skills").

$target = Join-Path $HOME '.claude/skills'   # hardcoded global; no per-repo target

Why it matters: Global install is the maximal blast radius — every installed skill is offered in every repo and session, including a developer's pure app repos. A developer who wants exactly one tool in exactly one repo has no supported path; it's all-or-nothing-everywhere. The 9 skills are generic, so the harm is mild, but the lever to contain reach is simply absent.

Severity rationale: Medium — degrades the bleed posture but the payload is domain-agnostic.

Remediation: add a -Project <path> / --project mode that copies into <repo>/.claude/skills/ instead of $HOME; recommend project-scope for anything that could mis-trigger, reserve global for the truly universal (e.g. todo, session-review).

3 · Passive-context surface (the real bleed surface)

MEDIUM The QE passive surface is large and domain-saturated — outsized steering if it ever loads

Evidence: IMA CLAUDE.md (234 lines) — the "Known Conventions" table; knowledge tree _shared/knowledge/ = 69 md files / 17,878 lines; MEMORY.md index = 111 lines.

| Jira project      | `CI` — tickets are CI-####
| Zephyr project    | `CI` — test cases are CI-T####, cycles are CI-R####
| CI filter         | p.RelationshipManagerId = 1 AND p.CreateDate BETWEEN ...
| Waiter Pattern    | the promise MUST be created before the triggering action

Why it matters: This is the literal substance of "Claude assumes QE work" — every line is Jira CI / Zephyr / MSSQL / IMA-domain. It is auto-loaded and, at ~18k lines for the tree alone, it would dominate the context window and the model's sense of "the task" if pulled into a dev session. The platform does not ship it — but its sheer size and saturation mean any accidental inclusion (wrong cwd, a copied file, a global CLAUDE.md) has outsized steering effect. The bleed risk is a function of this surface area.

Severity rationale: Medium — not distributed by the platform, but high-impact-if-loaded and currently unfenced.

Remediation: keep it strictly project-local — never lift QE conventions into a global ~/.claude/CLAUDE.md; name it in the NEVER-SHARE list (Dim 1 remediation); when assisting a dev, run Claude with the dev's repo as cwd, never the IMA repo.

4 · Trigger / description hygiene

LOW A few generic toolkit triggers can fire unprompted in a dev session

Evidence: toolkit skill descriptions — e.g. todo ("add to the todo"), session-review ("are we good?", "before we move on"), condense-claude-md.

Why it matters: These fire on generic phrasing a developer might use naturally, so Claude may offer a QE-flavored skill action unprompted. But all three are genuinely generic engineering tools (a task list, an end-of-phase review, a CLAUDE.md condenser) — firing them in a dev repo is mostly helpful, not contaminating. This is hygiene, not harm.

Severity rationale: Low — generic tools, graceful, no QE-domain assumption baked in.

Remediation: accept (they're universal), or tighten descriptions if a dev reports noise.

MEDIUM Two toolkit skills assume QE-shaped repo artifacts

Evidence: qe-essentials/skills/sync-skills/SKILL.md (expects _shared/skill-architecture.md routing doc); condense-claude-md targets a CLAUDE.md.

Why it matters: In a dev repo with no routing doc / no CLAUDE.md, these skills have nothing to act on. sync-skills degrades gracefully (it falls back to a disk-only audit), so the failure mode is benign — but it's a sign these were authored against a QE-shaped repo and merely tolerate a generic one. They're portable in license, not fully in spirit.

Severity rationale: Medium — they degrade rather than mislead, but they assume context a dev repo lacks.

Remediation: confirm each toolkit skill degrades cleanly with no QE artifacts present; note the assumption in their descriptions; the scout (Dim 5) should flag this when auditing portability.

5 · The scope-discipline control — maintainability

MEDIUM Nothing automatically re-checks that the toolkit stays domain-agnostic over time

Evidence: promote-skill/SKILL.md (the gate runs only at promotion time, on one skill); no recurring audit of the already-installed toolkit.

Why it matters: The isolation guarantee ("the shelf holds only portable skills") is enforced once, at the moment of promotion. There's no periodic check that a previously-portable skill hasn't drifted to reference a project artifact, or that someone didn't hand-drop a coupled skill into qe-essentials/skills/. Over months, the control can erode silently — the exact failure mode that lets a QE-coupled skill end up on a developer's machine.

Severity rationale: Medium — compounding maintainability tax on the control that the whole verdict rests on.

Remediation: have skill-scout (or a CI check) re-run the portability test across the whole toolkit on a cadence, flagging any shelf skill that no longer passes scope: portable.

6 · Failure modes of accidental bleed

LOW→INFO When bleed does happen, the failure is mostly graceful and recoverable

Evidence: init-qe-repo/SKILL.md:100-101 (idempotent-ish, won't overwrite customized CLAUDE.md); sync-skills degrade-to-disk-only path.

Why it matters: The blast of a realized bleed is bounded: init-qe-repo writes are git-tracked and reversible and it won't clobber a customized CLAUDE.md; mis-triggered toolkit skills are read-mostly and degrade rather than corrupt. There is no money/auth/data-loss path in this surface — which is why nothing here is Critical. The residual harm is wasted developer attention and confusion, not damage — real, but recoverable.

Severity rationale: Informational — characterizes the ceiling on harm.

Remediation: none beyond the above; this is the reassuring backstop.

Mitigation strategy — what to do

Share / Never-share — the boundary to write down

SHARE freely (domain-agnostic, safe for any developer): the qe-essentials toolkit (9 portable skills), the playbook/, skill-starter, promote-skill, skill-scout. These carry no QE domain, tickets, or test cases.

NEVER share / never global-ize (QE passive surface): the project CLAUDE.md, the knowledge tree (_shared/knowledge/), MEMORY, any ticket.md / Zephyr test-case artifacts, and the domain orchestrators (three-amigos, ready-for-*, prod-support, etc. — all scope: project).

DO The five guardrails, in priority order

  • 1. Write the boundary down (closes H1) — a playbook/context-boundary.md with the SHARE / NEVER-SHARE lists above, echoed in the toolkit README. Cheapest, highest-leverage fix.
  • 2. Gate init-qe-repo (closes H2) — loud confirm listing every file it will write; re-scope its triggers so "write/add a test" does not match; reserve it for explicit "stand up a QE repo."
  • 3. Add a project-scoped install (closes M1) — install-toolkit -Project <path> drops into one repo's .claude/skills/; reserve global for the truly universal.
  • 4. Keep dev repos' CLAUDE.md their own — never inherit QE's; when helping a dev, run Claude with the dev's repo as cwd, never the IMA repo.
  • 5. Make the scout audit the shelf (closes Dim 5) — re-run the portability test across the whole toolkit on a cadence so the isolation guarantee can't erode silently.

Net: the developer's instinct was right to ask, but the platform's architecture already answers most of it. Two documented guardrails (#1 and #2) close the bulk of the residual risk; the rest is hardening.
