Three long-running sessions on Industrial INT — why, from where, and is this normal?

A SQL slowdown investigation on mssql-industrial-int. Three sessions (405, 383, 340) have run 6–8 hours. This report answers, evidence-first from the live DMVs: why they are slow, who/what triggered them, and whether they normally take this long.

Verdict — the three things that matter

None of the three queries is blocked and none has a lock problem (blocking_session_id = 0, waiting-tasks empty). They are slow because the instance is CPU-starved and the two DemandForecasting queries scan expensive views. Only one of the three is a genuine risk to the business:

  1. The nightly billing-flag proc (session 405) is 100×+ slower and now overlaps its own next run. IGS.SetIsManualBilledOnContractTerm historically finishes in ~1–2 minutes; it has run 8.3 hours today, holds an open transaction on dbo.ContractTerms, and its empty CATCH silently swallows any failure — the 07-01 run left no completion record at all. This is a money-path (billing) job flying blind. Critical
  2. A 128-request Igs.Rad IMAAccountFeed storm is saturating the box. 128 of ~144 active requests come from one feeder (INT-QUEUE-03); 105 tasks are queued runnable on 24 cores. This is the noisy neighbor starving billing and ETL — the proximate cause of all three slowdowns. High
  3. The two 6-hour extracts are full SELECT * scans of heavy views. SSIS pulls entire DemandForecasting views (CTEs, window functions, per-row correlated TOP 1 subqueries, non-sargable date predicates) with no filter — 76–81M logical reads each, on serial plans. High

1 · Why are they taking so long?

Not blocking, not locks — CPU starvation + heavy scans. Every session is runnable (waiting for a CPU quantum), not suspended on a lock. The tell is the ratio of CPU to wall-clock: session 405 burned 95 min of CPU across 8.3 hr (81% of its life not on a CPU); the two extracts used ~4 min of CPU across 6 hr (99% waiting). The instance has 24 schedulers but 105 runnable tasks queued (avg load factor 29) — roughly 4–5 requests deep per core. On top of that, the two DemandForecasting objects are views, so each SELECT * materialises the full view computation (76–81M logical reads) on a serial (DOP 1) plan. Slow work, running on a starved box.

2 · Where did they come from / who triggered them?

Two different scheduled batch systems, none of them a person. Session 405 is the nightly billing determination job — Choice_OpsEng firing sqlcmd from the app server INTAPP06 at 00:00 (an external scheduler — Control-M / Windows Task, not SQL Agent, which would name itself differently). Sessions 383 and 340 are one SSIS ETL package on the supply SSIS host INT-SPLYSSIS-01 (service account INT_SPLYSSIS_SVC, OLEDB) that kicked off two data-flow extracts 23 seconds apart at 02:18. They are unrelated jobs that happen to collide on the same instance — and both are being crushed by a third, unrelated workload (the RAD feed).

3 · Is there historical usage — do they normally take this long?

No. The billing proc is wildly abnormal; the extracts have no clean baseline but are clearly starved, not inherently 6-hour jobs. The proc self-logs each successful completion to IGS.StoredProcedureAndFunctionExecutionLog, and that history is unambiguous:

  1. 2026-06-19 → 06-26: completes at 00:01–00:02 every night — i.e. ~1–2 minutes after its midnight start. This is normal.
  2. 2026-06-27 onward: completions jump to 10:47, 10:42, 17:42, 16:15 — hours-long runs. A step-change regression begins on 06-27.
  3. 2026-07-01: no completion row at all — the run did not finish (or errored into the swallowing CATCH).
  4. 2026-07-02: a completion at 05:08 and session 405 still running since 00:00 — runs are now overlapping (today's midnight run launched while the prior day's was still going).

The two SSIS extracts leave no completed-run history on this instance (only lightweight SSIS metadata-validation probes are cached — 1–26 logical reads, created at the 02:17:46 connect). SSIS run durations would live in the SSISDB catalog on INT-SPLYSSIS-01. But the same CPU-to-wall ratio (1%) proves they are starved, not doing 6 hours of real work.

Follow-up — RAD burst TRIAGE, novelty & 06-27 traceability

What this adds · snapshot @ 08:55 CT (≈40 min after the first)

Continued diligence chased the RAD burst with the /rabbitmq HEALTH/TRIAGE flow and pulled the historical trend. The storm broke on its own. Between 08:18 and 08:55 the Igs.Rad load on INT-QUEUE-03 fell from 128 requests to 1, the account-feed queue drained to 0 pending, and all three long-running sessions (405 / 383 / 340) completed and disappeared from dm_exec_requests. That is direct causal proof of the noisy-neighbor thesis: kill the CPU pressure and the starved jobs finish. It also reframes two things — the RAD storm is not new, and it is not a volume event.

This is a transient concurrency burst, not a stuck consumer.

TRACEABILITY The 06-27 onset is real but unpinned — because the traceability isn't there

The regressed proc: IGS.SetIsManualBilledOnContractTerm (DB IGS_AppData, schema IGS) — created 2017-04-06, last modified 2020-04-28. The proc code did not change on 06-27; this is not a deploy. What you have to trace it:

  • Invocation: IGSDOMAIN\Choice_OpsEng via sqlcmd from INTAPP06 at 00:00 daily. Program name SQLCMD (not SQLAgent…) → an external scheduler (Control-M / Windows Task), not a SQL Agent job. The schedule definition and its logs live on INTAPP06 / Control-M — check there, not msdb.
  • Completion audit: only the self-log IGS.StoredProcedureAndFunctionExecutionLog — and it is fragile. The 13 completion rows visible at 08:18 were gone by 08:56 (server INT-OPSSQL-02, no HADR; total log rows today = 2,153,322). The log takes 2M+ rows/day and is actively trimmed, so this daily proc's handful of entries are needles that get swept. No start time, no duration, no row count is ever recorded.
  • No plan/duration history: Query Store is OFF on IGS_AppData, and the proc recompiles each run (SELECT INTO temp tables), so dm_exec_procedure_stats holds nothing. There is no stored plan or duration to compare 06-26 vs 06-27.
  • Dependent objects: every table the proc reads (IGS.Accounts, Contract.*, dbo.Contract/ContractTerms, and Industrial.IMA.ServiceLocations/ContractDetails/…) shows modify_date of today 08:51 — i.e. a schema-touch/maintenance op runs regularly across them, but sys.objects keeps only the latest date, so it can't confirm or rule out a 06-27 change.

Net: the 06-27 step-change is documented (the 08:18 completion trail: ~1–2 min through 06-26, then 10:47 / 10:42 / 17:42 / 16:15, missing 07-01, overlap on 07-02) but its root cause cannot be pinned from current telemetry — QS off, proc unchanged, RAD volume flat, log volatile, no plan history. Candidates that remain open: a statistics/plan regression on the heavy cross-DB joins, gradual base-table growth crossing a plan tipping point, or a shift in RAD burst timing (not volume). Closing that gap is itself a recommendation (R6).

CONTEXT The instance at 08:18 CT

  • Everyone else is a rounding error: next-busiest sources are INT-QUEUE-02 (7), the two SSIS extracts (2, INT-SPLYSSIS-01), and the billing SQLCMD (1, INTAPP06).

Findings — ranked by blast radius

CRITICAL F1 · Nightly billing-flag proc degraded 100×+ and now overlapping — failures silently swallowed

Evidence — historical baseline vs. now (IGS.StoredProcedureAndFunctionExecutionLog, completion timestamps):

Evidence — the swallowing CATCH (proc body, currently executing in session 405):

update dbo.contractterms
set IsManualBilled = 1
where contractid in (select contractid from #iscomplexbilledcontractids)
...
BEGIN TRY
  DECLARE @... NVARCHAR(1000) = 'INSERT INTO IGS.StoredProcedureAndFunctionExecutionLog ...'
  EXEC sp_executesql @...
END TRY
BEGIN CATCH        -- empty: any failure is discarded, no log, no rethrow
END CATCH

Why it matters: this proc writes a billing determination — it sets IsManualBilled = 1 on dbo.ContractTerms for complex-billed IN-% Choice contracts. A run that overruns or dies means those flags are stale or unset for the day, and because the completion log is the only observability and it lives behind an empty CATCH, a failure looks identical to "never ran." The 07-01 gap is exactly that signature. Session 405 currently holds open_transaction_count = 2 while starved, so its final UPDATE dbo.ContractTerms can also become a blocker for other billing writers.

Severity rationale: High → Critical (money/billing path + silent failure + no rollback visibility).

Remediation sketch: add duration + start/row-count logging (not just an end timestamp) and make the CATCH log/alert instead of swallow; add a run-guard (skip/abort if the prior instance is still running) to stop overlap; medium-term, filter the work set and index the join (see F4) so it returns to ~minutes.

Why it matters: 105 runnable tasks on 24 cores means every CPU-hungry batch on this instance is queued 4–5 deep behind other work. The RAD feed is the overwhelming source of that pressure, and it is the proximate cause of all three slowdowns — session 405's SOS_SCHEDULER_YIELD (couldn't get CPU) and 383/340's PAGELATCH_EX (tempdb allocation contention from mass concurrent spills) are both symptoms of this one storm. The three long-runners are victims, not the disease.

Call path / reach: RAD, billing, and DemandForecasting ETL are three unrelated workloads co-tenanted on one 2016 Standard instance with no Resource Governor isolation — any burst in one starves the others.

Severity rationale: High — broad blast radius across every workload on the instance; contained to INT (not prod), so not bumped to Critical.

Remediation sketch: find out why 126 IMAAccountFeed executions launched in one 12-second window (a reindex backfill? a stuck queue consumer fanning out?) — cross-reference /rabbitmq TRIAGE. Longer term, throttle the feed's consumer concurrency and/or isolate it with Resource Governor.

HIGH F3 · Full SELECT * extracts of heavy view-on-view definitions (76–81M logical reads, serial)

Evidence — both DemandForecasting objects are views, not tables (sys.objects.type_desc = 'VIEW'), and the SSIS extract runs an unfiltered SELECT of every column. The definitions contain per-row correlated subqueries and non-sargable predicates:

-- ContractVolumes: a per-row correlated TOP 1 subquery in the SELECT list
,ISNULL((SELECT TOP 1 '[Component: ' + cdrdt.Description + ', Index: ' + cdrd.IndexName + ']'
          FROM IMA.ContractDetailRateDetails cdrd
          INNER JOIN IMA.ContractDetailRateDetailTypes cdrdt ON ...
          WHERE cdrd.ContractDetailId = cd.Id AND (cdrdt.Code = 'GasCapacityVariable' OR ...)), (...)) as IndexText
-- ...and self-joins on ContractDetails via BETWEEN date ranges (secondary/tertiary details)
LEFT JOIN IMA.ContractDetails secondaryDetails ON secondaryDetails.termid = cd.termid
     AND cd.fromdate BETWEEN secondaryDetails.FromDate AND secondaryDetails.ToDate AND secondaryDetails.typeid = 2

The …ByMarketArea view stacks a GROUP BY CTE (MonthlyContractAccountVolumesByMarketArea, aggregating SUM(...) across service-location components) and a market-area zip-code CTE on top of the same base joins — a second, heavier layer that reads even more (81.4M vs 76.2M logical reads).

Why it matters: pulling an entire computed view with no WHERE forces SQL Server to materialise the whole thing every run. On a quiet box this might be tolerable; on the saturated box (F2) it stretches to 6 hours. The SSIS side is also draining rows slowly (near-zero physical reads on 340 — data is cached — yet still 6 hr), pointing at a slow OLEDB consumer compounding the problem.

Severity rationale: High — a core ETL feed that can't reliably complete; INT-only keeps it off Critical.

Remediation sketch: push a date/LOB filter into the extract instead of SELECT * (the view already self-limits to one year, but the extract can scope to the delta it needs); review the SSIS data-flow buffer/commit size so the source isn't throttled by a slow destination; consider a materialised/indexed intermediate for the market-area aggregation.

MEDIUM F4 · All three heavy queries run serial (DOP = 1) despite tens of millions of reads

Evidence: dm_exec_requests.dop = 1 for sessions 405, 383, and 340 (from the live request snapshot). Each is doing 58–81M logical reads on a single scheduler.

Why it matters: million-read scans are exactly the workload parallelism exists for; a serial plan multiplies wall-clock and makes each query far more sensitive to the CPU contention in F2. This is consistent with a low max degree of parallelism / high cost threshold for parallelism at the instance level, or plans cheap-enough-per-row that the optimiser never parallelises them.

Severity rationale: Medium — amplifies F1–F3 but is not itself the trigger; needs a config/plan review, not an emergency change.

Remediation sketch: check instance MAXDOP / cost-threshold vs. this hardware (24 cores); confirm whether the serial plans are chosen or forced before changing anything.

MEDIUM F5 · Non-sargable date predicates in the views defeat index seeks

Evidence (both view WHERE clauses):

WHERE cd.typeid = 1
  AND cd.FromDate IS NOT NULL AND cd.ToDate IS NOT NULL
  AND DAY(cd.FromDate) = 1                              -- function on column
  AND cd.ToDate = EOMONTH(cd.FromDate)                  -- function on column
  AND cd.FromDate > DATEADD(year, -1, getdate())        -- ok, but paired with the above
  AND cd.IsUnwind = 0

Why it matters: wrapping cd.FromDate in DAY() and comparing cd.ToDate to EOMONTH(cd.FromDate) makes those predicates non-sargable — the optimiser can't seek an index on FromDate/ToDate and falls back to scanning ContractDetails, feeding the large logical-read counts in F3.

Severity rationale: Medium — a real scan driver, but bounded by the 1-year filter.

Remediation sketch: where possible, express the "first-of-month / full-month" test with sargable range boundaries or a computed persisted column that can be indexed.

MEDIUM F6 · The proc's only failure signal is an empty CATCH — no error surface, no duration

Evidence: see the BEGIN CATCH … END CATCH block quoted in F1. The log row is written inside the TRY, only on success, and records just Name, ExecutionDate, ExecutedBy — no start time, no duration, no row count.

Why it matters: operationally you cannot tell "slow" from "failed" from "never scheduled" — precisely the ambiguity around the missing 07-01 row. A billing job deserves better than write-only-on-success logging behind a swallowing CATCH.

Severity rationale: Medium — observability gap on a money path; separate from the perf regression itself.

Remediation sketch: log start + end + duration + affected-row count; in CATCH, capture ERROR_MESSAGE() and re-raise or alert.

LOW F7 · Wide SELECT * with in-view string parsing, and a CTE duplicated across both views

Evidence: the extracts select ~35 columns including several that parse numeric volumes out of text at read time:

,NULLIF(LTRIM(RTRIM((CASE WHEN cd.TTLExpectedBillingMonthVolume LIKE '(%)'
     THEN REPLACE(REPLACE(cd.TTLExpectedBillingMonthVolume,'(','-'),')','') ELSE ... END))),'-')

The UnwoundContractDetails CTE is copy-pasted verbatim into both views, so the same window-function computation is maintained (and executed) twice.

Why it matters: maintainability tax and wasted per-row work; storing volumes as numerics upstream and factoring the shared CTE would shrink both the code and the read cost. Low because it compounds F3 rather than causing it.

Remediation sketch: normalise the volume columns to numeric at write time; extract the shared CTE into one reusable view.

LOW F8 · SQL Server 2016 (13.0.4466), Standard Edition

Evidence: the proc's scripting header and dm_os_sys_info confirm build 13.0.4466 (SQL Server 2016). Standard Edition caps usable memory and lacks some parallelism/columnstore refinements that would help these scan-heavy views.

Why it matters: SQL Server 2016 mainstream support has ended and extended support is closing; version and edition ceilings quietly limit the remediation options for F3/F4. Low — a background risk, not today's cause.

Remediation sketch: factor the edition/version ceiling into any capacity or upgrade planning for this instance.

Bottom line

Where it landed · what's left

The incident self-resolved by 08:55 CT — the RAD account-feed burst drained (queue to 0 / 100 consumers), and sessions 405, 383, and 340 all completed once CPU freed up. Nothing needed to be killed. That confirms the mechanism: these three jobs were victims of transient noisy-neighbor CPU starvation, not broken queries.

But the durable risks remain and will recur on the next burst: the billing proc can still run 8+ hours, overlap itself, and hide failures (F1/F6); the account-feed still has no isolation (F2); the SSIS extracts and views are still heavy and serial (F3–F5). The 06-27 regression root cause stays unpinned because the traceability to diagnose it doesn't exist yet — see R6. Priorities: R1 + R6 (make the billing proc guarded, logged, and diagnosable), then R2 (isolate the account-feed), then the durable perf work (R3–R5).
