Caveats — the honest frame

  • Unit demotion is inference — FLAGGED rows are candidates to confirm, not coverage.
  • E2E-depth, not API-depth: DFR resolve, email template CRUD, updateContractFriendlyName, deleteOpportunity are covered only through the browser via waiters — no API spec asserts res.data.<mutation>.errors. A lower-tier "API-depth gap" for those user-facing mutations.
  • Integration-only paths (Outlook/MS-Graph, Exchange, Choice360/EDSA outbound) enumerated only where they surface in CRM's SDL; pure webhook/outbound paths are not enumerated.
  • Elastic-index admin mutations (7 ops, all @authorize) are out of product-domain scope — noted, not ranked.
  • Normalization is heuristic — every gap row prints its raw SDL line for human verification.
  • Since last scan: the runs.jsonl baseline was a party-only scan — scope differs, no meaningful delta. First whole-suite-minus-party run.

Highest-leverage fix

Cover mergeAccounts first. It is the only merge operation in the suite with zero coverage (party and opportunity merge are both tested), it is a data-loss-class mutation cascading across accounts, opportunities, contracts, and legal-entities, and a companies.merge.spec.ts-style fresh-data-per-run harness already exists to clone. Then close the four @authorize gaps (commissionable / bulk-update-opportunities / bulk task+note) — one negative authz-denied test each verifies the gate.
