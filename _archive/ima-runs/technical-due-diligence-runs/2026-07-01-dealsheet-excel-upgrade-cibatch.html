Dimension 1 — Architecture, Coupling, Boundaries

HIGH Fix deployed but not live: wrong execution host + service-account COM-launch gap (deterministic-by-box)

Evidence: README.md:305-320 (Run 8 resolution)

| box | Excel COM as dev_cibatch_svc | MAXIFS / monthly margins |
| -02 (where the scraper actually runs) | ✅ works | ❌ Excel too old → #NAME? → zero margins (the original bug) |
| -07 (the 2024-upgraded box)          | ❌ 0x80070005 | ✅ works |
...the live INT-ProcessDeals job's Host is INT-BATCH-02, not -07 ... the repoint to -07
lived in an uncommitted workspace (WSSTATE_EndUserWorks) and never reached the running job.

Why it matters: the entire Excel-2024 upgrade (which fixes the MAXIFS/modern-formula capture) is validated only on -07, but production deals are scraped on -02 (old Excel). So the money-path defect (Dim 4) is live while the fix sits un-wired. Worse, -07 can't run it yet — dev_cibatch_svc hits 0x80070005 launching Excel COM (5/5 probe failures vs eheineking succeeding on the same box). An outbound integration whose "done" state was mis-read three times (host, then DCOM-ACL, then per-user Office — all superseded; the real gap is account-on-box config).

Severity rationale: High (outbound-integration delivery on the money path; the validated fix is not the one running).

Remediation sketch: ITOPS configures dev_cibatch_svc's Office/COM on -07 to match -02, then move INT-ProcessDeals Host to -07 and verify the checked-in job actually deployed (the last repoint silently didn't).

MEDIUM Both probes hand-duplicate ExcelTools.ExcelContext's open semantics — drift risk vs the real thing

Evidence: ProbeScraper.cs:71-80 (and FormulaProbe.cs:142-147)

wb = app.Workbooks.Open(wbPath, ReadOnly: true, Notify: false,
    CorruptLoad: Excel.XlCorruptLoad.xlRepairFile, IgnoreReadOnlyRecommended: true, UpdateLinks: 0);
app.Calculation = Excel.XlCalculation.xlCalculationAutomatic;
app.CalculateFullRebuild();

Why it matters: the probes' fidelity depends on manually mirroring ExcelContext's arguments (comments say "mirror ExcelTools.ExcelContext.Open"). If the real ExcelContext changes (e.g. the CI-7806 move to a cached read, or a different CorruptLoad), the probes silently stop representing production and could give false comfort. The CorruptLoad: xlRepairFile choice is itself flagged in the log as a suspect for the MAXIFS token issue (README.md:254).

Remediation sketch: if these probes are kept, have them #include/reference the real ExcelContext rather than re-declaring its open args; otherwise mark them frozen-to-a-point-in-time.

Dimension 3 — Security Surface & Trust Boundaries

LOW Probe build enables "Trust access to the VBA project object model" and leaves it on

Evidence: build-probe-kit.ps1:34-36; README.md:95

$sec = 'HKCU:\Software\Microsoft\Office\16.0\Excel\Security'
Set-ItemProperty -Path $sec -Name AccessVBOM -Value 1
# README: "create mode required enabling ... AccessVBOM=1 ... left enabled."

Why it matters: AccessVBOM=1 lets code inject/modify VBA projects — a real macro-security relaxation, set on the dev machine and (per the run-log) on the server for the running user, and left enabled. Low because it's HKCU on non-prod investigation boxes, not the prod scraper — but it should be reverted, and never scripted on a prod host. (Related, non-blocking: README.md:123 documents unprotecting a deal sheet's VBA project with a team-known password — correctly kept out of git.)

Remediation sketch: reset AccessVBOM=0 after probing; never set it as part of a prod deploy.

Otherwise no material findings — the scripts use integrated Windows auth (no secrets in-file); connection targets are hostnames, not credentials.

Dimension 4 — Error Handling & Failure Modes

CRITICAL Scrape recalc diverges from the authored sheet; a #NAME? is silently stored as 0 in a Pending margin column

Evidence: README.md:216-221, 242-247 (Run 7 root cause) + live Pending.MonthlyPriceDetails shape

MonthlyMargin ×12:  authored 0.00097138  →  server recalc 0 (dates intact)
Hedges!AF5 = IFERROR(AE5 +  × (AC5−AB5), 0)
Hedges!I4  = ... _xlfn.MAXIFS(Main!$D$24:$D$29, Main!$R$34:$R$39,"<>"&0)  → #NAME? on the scrape box
→ AF5's IFERROR(…,0) turns #NAME? into 0 → MonthlyMargin emits 0 for all months
→ Pending.MonthlyPriceDetails.Margin = 0   (decimal(19,8), confirmed live 2026-07-01)

Why it matters: the scraper re-evaluates formulas on open (xlCalculationAutomatic + CalculateFullRebuild) and captures the recalculated value, which is not the value on the approved deal sheet. Two failure shapes, both money-path: (a) the hedge energy/margin split shifted on recalc (market-dependent input re-evaluating at scrape time — README.md:238); (b) MonthlyMargin zeroed because a MAXIFS #NAME? is swallowed by the sheet's own IFERROR(…,0) and written to the DB as a real 0. Verified as a fresh insert, not a stale update (README.md:200-203). This is silent financial-data corruption reaching the system of record with zero error surfaced — the textbook argument for CI-7806's "read cached values instead of recalculating."

Call path / reach: deal sheet → ExcelContext recalc → mapping.xml (From="=MonthlyMargin|…") → PendingDeal.MonthlyMarginCurve → Pending.MonthlyPriceDetails.Margin (README.md:196-198). Every LFA deal with these UDFs.

Severity rationale: Critical — money computed wrong + silently persisted, no rollback, no error. (The scraper source that recalcs is in DealDashboard, unaudited here; evidence is the run-log's investigation + live DB.)

Remediation sketch: CI-7806 — scrape cached values (not a live recalc) for authored/approved figures; and remove the IFERROR(…,0) mask in Hedges!AF so a #NAME? surfaces as an error, not a zero.

MEDIUM Scraper exit code is 0 even on total per-deal failure — errors swallowed into the "errored deals" flow

Evidence: README.md:59

"Exit code is 0 even on total failure — per-deal errors are swallowed into the
'errored deals' flow. Don't use the exit code as a success signal; read the error output."

Why it matters: any monitoring or Control-M success check keyed on exit code will read a fully-failed run as success. Combined with the #NAME?→0 masking above, the pipeline has two independent "failure looks like success" layers. (Control-M does watch NOTOK/failcount, but a per-deal swallow never trips it.)

Remediation sketch: non-zero exit (or an explicit failed-deal count signal) when any deal lands in Errors\Failed.

LOW Probe COM cleanup leaks orphan EXCEL.EXE (v1); acknowledged

Evidence: FormulaProbe.cs:74-83; README.md:96

try { app.Quit(); } catch ... Marshal.FinalReleaseComObject(app); GC.Collect(); ...
// README: "the probe leaves orphan EXCEL.EXE processes behind ... the runner kills them."

Why it matters: orphaned hidden Excel processes wedge later automation (the same class that bit runs 2/8). Probe-only (ProbeScraper.cs:124-133 does the double-GC dance better); the real exe shut clean in run 3.

Remediation sketch: release every intermediate RCW (worksheets/cells), not just the app — or accept it as a throwaway.

Dimension 5 — Tech Debt & Maintainability

MEDIUM Hard-coded absolute dev paths in the deploy/build scripts

Evidence: copy-local-to-server.ps1:13-14; build-probe-kit.ps1:16-17

[string]$SourceDir = 'C:\Development\DealDashboard\IGS.DealCapture\IGS.CIBatch\bin\Local'
[string]$LibsDir   = 'C:\Development\DealDashboard\IGS.DealCapture\libs\Office'

Why it matters: the scripts only work from one machine layout (C:\Development\…); a different clone path silently breaks them. They're params (overridable) but the defaults encode one operator's box. Fine for a personal investigation, a tax if this tooling is shared.

Remediation sketch: derive from $PSScriptRoot / repo-relative discovery, or a small config.

MEDIUM Two probes with a duplicated, drifting test matrix

Evidence: FormulaProbe.cs:30-44 vs build-probe-kit.ps1:42-61

Why it matters: the 12-row formula matrix is authored twice (C# array + PS array), already diverging (the PS version adds volatile/side-effect rows). Two sources of the same truth → they'll drift; a reader can't tell which is canonical. Low-stakes (throwaway), but it's exactly the "duplicated fact" smell.

Remediation sketch: pick one probe (ProbeScraper is the closer-to-real one) and retire the other, or share the matrix from one file.

LOW Console logs written UTF-16; Compress-Archive drops empty dirs (folder-tree bug)

Evidence: README.md:82 (UTF-16); copy-local-to-server.ps1:69-82 + README.md:50-54

Why it matters: Tee-Object defaults to UTF-16 → grep tools mis-read the logs. Compress-Archive silently omits empty directories, which caused the run-1 DirectoryNotFoundException; the script now recreates the tree by hand (a workaround for a packaging gotcha). Both documented, both papered-over rather than fixed at the source.

Dimension 6 — Dependency Health & Version Risk

LOW Vendored Office 2013 PIAs + x86 + .NET Framework — investigated and correctly settled

Evidence: README.md:24-28

Why it matters: normally "2013 PIAs in 2024" is a red flag, but the log empirically settled it: Office LTSC 2024 registers the same Microsoft.Office.Interop.Excel 15.0.0.0 build in the GAC; MS froze the Excel PIA at the 2013 build; with EmbedInteropTypes=true binding is by COM GUID, not assembly version. One real gap noted (typelib 1.9 vs PIA-from-1.8 → post-2013 API members invisible at compile time) with a mitigation. This is a model of a dependency risk chased to ground — leave the DLLs alone.

Dimension 7 — Performance Hotspots

LOW Scraper opens Excel ~3× per deal (documented, not measured here)

Evidence: README.md:176

Why it matters: "the exe re-opens each output copy to break external links and save (three Excel opens total)." Per-deal Excel process churn is the perf shape of COM scraping; at the 2-min Control-M cycle with the 30-min kill it's tolerable, but it's the scaling ceiling if deal volume grows. No in-folder code to optimize (scraper is elsewhere); flagged for the real audit.
