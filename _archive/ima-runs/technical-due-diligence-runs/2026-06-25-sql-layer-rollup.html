Layer verdict — the three things to fix first across the whole layer

No Critical findings: this layer doesn’t lose or corrupt data, and it computes the right things. It is careful, well-commented work. The overhaul should be a consolidation, not a rewrite — the same good ideas are applied inconsistently, and three of those inconsistencies dominate the risk:

  1. Parameterize the queries (TDD-017, High). String params are interpolated into live-DB SQL with no binding barrier — the only finding in the layer with an external-risk shape. The fix is mechanical, and the layer already does it correctly one folder over (sql.industrial.config.ts uses .input()).
  2. Fix the connection lifecycle (TDD-015, High). The CRM facade news a new pool per finder call and never closes it; the Industrial path closes correctly in a finally. Make the CRM client a memoized per-worker singleton — one change that also removes the per-call Key Vault fetch.
  3. Extract the shared fragments & constants (TDD-017 + 016 + 018). The CI filter (~80 copies), the terminal-state gate (two drifting forms), the magic IDs, and the duplicated connection config are the compounding tax that makes every other change risky. The layer proves it knows how (MERGE_SAFE_PARTY_PREDICATE) — apply that discipline to the fragments that repeat most.

Underneath those: the type layer is a claim, not a check (hand-written, partial, with one type that declares fields its query never returns) — add generation or a runtime assertion so the cast stops being an act of faith.

Suggested overhaul sequence

Order the work so the safety nets land before the refactor

  • 1 · Parameterize string inputs (TDD-017 F1) — close the injection surface + the latent quote-breakage first; it’s isolated and high-value.
  • 2 · Singleton the connection (TDD-015 F1, F8 + TDD-016 F4) — one memoized CRM pool + one Industrial pool per worker; removes the leak and the per-call secret fetch in one move.
  • 3 · Add a type/schema guard (TDD-018 F1) — generation or a dev/CI shape assertion before moving code, so the refactor can’t silently break a result shape.
  • 4 · Extract shared fragments & constants (TDD-017 F2/F3/F4/F5 + TDD-016 F2) — CI filter, terminal-state gate, magic-ID registry, connection-config builder.
  • 5 · Re-draw boundaries (TDD-015 F5/F6 + TDD-016 F1 + TDD-017 F10 + TDD-018 F3) — split the god-facade into per-domain finder modules behind a stable sqlDB barrel; move the Industrial finders out of “config”; consolidate the types. Do this last, on top of the guard from step 3.

Cross-cutting themes — the same good idea, applied unevenly

Four patterns the overhaul can standardize once and reuse everywhere

  • Parameter binding is bimodal. sql.industrial.config.ts binds scalars with .input() (correct); queries/* interpolate strings (TDD-017 F1) and IN-lists (TDD-017 F9, TDD-016 F6). Pick binding, everywhere.
  • Pool lifecycle is bimodal. Industrial closes in a finally (TDD-016 F4); CRM never closes (TDD-015 F1). Pick a shared singleton, everywhere.
  • Predicate extraction is bimodal. MERGE_SAFE_PARTY_PREDICATE proves the team extracts shared fragments (TDD-017); the CI filter and status gate don’t (TDD-017 F2/F3). Extract the high-frequency fragments too.
  • Type safety is advisory, not enforced. The GraphQL side is generated (codegen-int); the SQL side is hand-written, partial, and in one case wrong (TDD-018 F1/F2/F3). Generate or assert.
