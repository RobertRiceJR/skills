Feature-Flag / Environment-Gating Strategy for the Playwright Suite

Pricing the risk in how the suite decides what runs where — and the lightest correct way to add a feature/environment gate so INT-only work (communications) stops polluting the regression signal.

Prepared for: Robert Rice / QE (suite owner) Date: 2026-06-24 Scope: TDD-012 · test-selection & env-gating subsystem
Critical · 0
No data-loss / auth / money-path gating defect.
High · 3
Signal pollution, a phantom gate, and unenforced capability debt.
Medium · 4
Missing axis, silent-forever skips, flag-source trust, floating PW pin.
Low · 4
Coupling, fail-open lookup, idiom drift, and the perf upside.
Verdict — the three things that matter

The suite has no environment/feature axis in test selection at all. Env (NODE_ENV) only changes where a test points (baseURL/SQL), never which tests run. So npm run regression (= -g @Regression) sweeps in every INT-only communications test on every pass. Robert's instinct — a tiny conditional test.skip fed by a lightweight flag source — is the right shape. The risk isn't the mechanism; it's doing it without a single audited source, which trades today's noise for tomorrow's false confidence.

  1. [High] Signal pollution is structural, not incidental — ~72 live, INT-only communications tests carry @Regression with zero guard; they run (and time out) on every regression run.
  2. [High] The suite's only documented env-gate is a phantom — "the CI-T2422/2423 callers skip on QA" is a comment with no skip code; the test runs unconditionally (and is a repeat-offender failure in INT). Comments-as-gates have already failed here once.
  3. [Medium] Skips are silent-forever — a skip reads as "fine" indefinitely. Built on a scattered/env-only source, a feature flag becomes permanent invisibility. The fix is one committed, reviewable flag file + an audit line, not the skip itself.

Phase 1 — Surface Map (how the suite picks tests today)

MAP Three selection axes — none of them is environment

Tracing the real entry points (npm run regression → run-staged.ps1 → npx playwright test):

  • Axis 1 — Project (filename suffix). playwright.config.ts:127-172: IGS-360 matches *.e2e.spec.ts, API-tests matches *.api.spec.ts, SQL-tests = src/tests/sql, setup only in storageState mode. Routing is suffix-based.
  • Axis 2 — Tag grep. run-staged.ps1:77-84: -Suite regression → -g @Regression; smoke → -g @Smoke; api → --project API-tests. npm run regression = -g @Regression, all workers, then --last-failed rounds.
  • Axis 3 — Path / title. ad-hoc -Spec / -Grep.
  • Environment is NOT a selection axis. context.ts:8 static env = process.env.NODE_ENV || 'int' drives only getBaseURL()/getBaseAPI()/SQL-server (context.ts:9-74). run-staged.ps1:133 sets $env:NODE_ENV. Nothing reads env to decide which tests execute.

Existing conditional-execution machinery (the precedent to build on):

  • The one real conditional skip: src/tests/setup/login-unprivileged.setup.ts:13,15 — setup.skip(!USE_UNPRIVILEGED, '…'). A boolean env-flag → skip(condition, reason). This is exactly the idiom to generalize.
  • The boolean env-flag parser idiom: auth-config.ts:22-46 — USE_STORAGE_STATE, TOKEN_REFRESH_ENABLED, TOKEN_BROKER_ENABLED each parse process.env.X into a boolean with a default. A clean, tested, ~3-line-per-flag template.
  • Hand-disable idiom: test.skip(true, reason) ×2 (data-exhausted) and test.fixme(true, reason) (pervasive — 30 in communications.e2e.spec.ts alone). Both are unconditional.

Change-radius: Context is imported by the config process and by fixtures and many specs — high blast radius. auth-config.ts is a low-radius, single-purpose flag module — the correct sibling location for new flags.

Dimension 1 — Architecture, coupling, boundaries

MEDIUM F4 · Capability-availability has no declared home in the selection model

Evidence — communications.e2e.spec.ts:410, 492, 651, 752, 913, 1149

// Note: ImaCommunication schema is INT-only — run with NODE_ENV=int

Why it matters: the config cleanly separates routing (suffix) from scoping (tag) from targeting (env→URL), but the suite's real intent — "this behavior only exists in INT" — has nowhere structural to live, so it leaked into prose. There is no boundary object for "capability availability per environment." Every new INT-only feature repeats the leak.

Severity rationale: Medium — a design gap that compounds with each feature, but no current correctness/security impact.

Remediation sketch: introduce a single declarative capability source (per-env flag file) + a one-line guard idiom; that is the missing boundary.

LOW F8 · Don't gate via the Context env singleton — use a sibling module

Evidence — context.ts:7-74 (baseURL + baseAPI + 2 SQL servers + exchange URL all switch on this.env)

export class Context {
  static env: string = process.env.NODE_ENV || 'int';
  static getBaseURL(): string { switch (this.env) { /* int|qa|prod */ } }
  // …getBaseAPI / getCrmSqlServer / getIndustrialSqlServer / getExchangeURL
}

Why it matters: Context is already an overloaded env singleton imported by the config process and widely across specs. Hanging feature flags off it raises its blast radius and couples "which tests run" to "where they point." auth-config.ts is the proven low-radius home for boolean flags.

Severity rationale: Low — a placement choice; either works, but the sibling keeps blast radius contained.

Remediation sketch: new src/utils/feature-flags.ts alongside auth-config.ts; it may read Context.env but should not live inside Context.

Dimension 3 — Security surface & trust boundaries

MEDIUM F6 · The flag source is the trust boundary — and a flag must never silently disable an authz test

Evidence — auth-config.ts:22 (env-only flag, no record) & opportunity.bulk-transfer.e2e.spec.ts:12-43 (an authz test, CI-T2157, currently a shell)

const storageStateFlag = process.env.AUTH_STORAGE_STATE;   // flips from any shell, leaves no trace
…
test.fixme('Verify user without BulkTransferOpportunitiesViaTemplate is redirected …', …)  // access-control case

Why it matters: if flags live only in process.env, what's switched off is invisible and un-reviewed — anyone can flip a gate per-shell with no PR record. A committed JSON makes the gate state a reviewable artifact (the trust-boundary-correct choice). Separately: a feature flag that skips an authz/permission test (e.g. the unprivileged-redirect CI-T2157) would hide an access-control regression behind a green check — flags must not gate security cases without a louder, separate marker.

Severity rationale: Medium — no secret exposure, but an un-audited gate over authz tests is a latent access-control blind spot.

Remediation sketch: committed per-env JSON as the canonical source (env var only as a one-off override); never list an authz test under a routine feature flag.

Dimension 4 — Error handling & failure modes

MEDIUM F5 · Skip-based gating is silent-forever → flag-off becomes permanent false confidence

Evidence — Playwright semantics + run-staged.ps1:141-150 (the cascade counts only failedTests; skipped ≠ failed)

function Get-FailedCount { … return @($j.failedTests).Count }   // skipped tests are invisible to the cascade/triage signal

Why it matters: Playwright reports a skip as non-failing; the staged cascade and /triage-failures only ever see failedTests. Once communications=false, those ~72 tests are green-by-omission with no alarm when the feature ships and the flag should flip back. The failure mode of the solution is permanent dormancy — the same way 30 test.fixme shells have sat in one file.

Severity rationale: Medium — the cure can quietly become the disease without an audit/report loop.

Remediation sketch: the flag JSON is the one audit surface + a periodic "flagged-off" report; consider a guard test that fails if a flag has been off past a review date.

LOW F9 · A flag lookup should fail closed on an unknown name

Why it matters: if featureEnabled('comunications') (typo) silently returns false (or true), the gate misfires invisibly — the test silently skips (or runs) against intent. Mirror the strictness of Context.getBaseURL(), which throws on an unknown env (context.ts:18).

Severity rationale: Low — a small helper-design choice with outsized debugging cost if wrong.

Remediation sketch: featureEnabled(name) throws if name isn't a declared key in the flag file.

Dimension 6 — Dependency health & version risk

MEDIUM F7 · @playwright/test is pinned to "latest" — a floating contract under the gate you're building

Evidence — package.json dependency

"@playwright/test": "latest"   // floating — every install can ingest a breaking change silently

Why it matters: the gate idiom rests on test.skip(condition, reason) (stable for years) and optionally tags / --grep-invert (also stable), so the design adds zero new dependencies — good. But a test framework pinned to latest means the very API your gate stands on is non-deterministic build-to-build (this repo has already been bitten by a version trap — the graphql-codegen double-emit). The risk is pre-existing, not introduced here, but it widens as more behavior (gating) leans on the framework.

Severity rationale: Medium — floating major-version on the core test dependency; no known active break, but unbounded.

Remediation sketch: pin @playwright/test to an explicit version. Reject any third-party feature-flag SDK (LaunchDarkly etc.) — network dependency + secret + overkill for ~5 booleans.

Dimension 7 — Performance hotspots

UPSIDE · LOW F11 · Gating reclaims the biggest wall-clock sink — and short-circuits before browser setup

Evidence — playwright.config.ts:97-98,111-112 (60s test + 30s action/nav timeouts) × ~72 comms E2E × every staged round

timeout: 60000, expect:{timeout:60_000}, actionTimeout:30_000, navigationTimeout:30_000
// ~72 INT-only comms browser tests (auth+nav), several timing out at 30–60s, on EVERY @Regression round

Why it matters: this is the "resources and time" Robert called out, quantified. A describe-level test.skip(condition, reason) is evaluated before the test body and per-test fixtures, so a gated file never spins up the browser/auth for those workers — the reclaimed time is direct, and it multiplies across the staged cascade's rounds. The flag isn't only signal hygiene; it's the cheapest available performance win.

Severity rationale: Low — an upside, not a defect. Noted because it's a primary motivation.

Remediation sketch: put the guard at the top of the describe (or the file's top-level describe), not inside individual tests, so the short-circuit happens before fixture setup.

Recommendation — the lightest correct approach

Build this (≈25 lines, zero new deps)

Robert's instinct is correct: native test.skip(condition, reason) fed by a lightweight source. Make the source a single committed per-env JSON, and generalize the proven auth-config.ts boolean idiom into one tiny helper. Two flag axes fall out naturally: env-capability ("INT-only" — schema absent in QA) and feature-WIP ("not ready for the regression signal yet"). The JSON expresses both per env.

STEP 1 A committed, reviewable flag file — config/feature-flags.json

{
  "int": { "communications": false, "summaryPanelActivity": true },
  "qa":  { "communications": false, "bulkTransferSlimTemplate": false }
}

One file, per-env, flips in a PR. This is the single audit surface (answers F3b/F5/F6). false = gated off.

STEP 2 A ~20-line helper mirroring auth-config.ts — src/utils/feature-flags.ts

import flags from '../../config/feature-flags.json';
import { Context } from './context';
import type { Test } from '@playwright/test';

type FlagName = keyof (typeof flags)['int'];

export function featureEnabled(name: FlagName): boolean {
  const envFlags = (flags as Record<string, Record<string, boolean>>)[Context.env] ?? {};
  if (!(name in envFlags)) throw new Error(`Unknown feature flag '${name}' for env '${Context.env}'`); // fail-closed (F9)
  const override = process.env[`FEATURE_${String(name).toUpperCase()}`];  // one-off shell override
  if (override === '0' || override === 'false') return false;
  if (override === '1' || override === 'true') return true;
  return envFlags[name];
}

/** First line of a gated describe: skips the whole block, before fixtures, with an honest reason. */
export function skipUnlessFeature(test: typeof Test, name: FlagName): void {
  test.skip(!featureEnabled(name), `feature flag '${String(name)}' is off for ${Context.env}`);
}

STEP 3 One line at the top of each gated file — replaces the comment

test.describe('Communications — email templates', { tag: ['@Regression'] }, () => {
  skipUnlessFeature(test, 'communications');     // ← was: // Note: ImaCommunication is INT-only
  // …the 72 tests; all skip cleanly with a reason when the flag is off
});

Then delete that file's "INT-only" comment (retires F3). First concrete adopter beyond comms: turn F2's phantom into a real test.skip(Context.env === 'qa', …).
