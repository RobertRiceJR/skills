Confirmed bug A — LVT opportunity change is a poison message HIGH

Where: OpportunityResolverHandler.GetIndexMapper · Index: ima-opportunities · Trigger: a RAD change on the Opportunity.IndustrialLargeVolumeTransportOpportunities (LVT) table.

Why we believe it's a bug

1. The trigger fires. The LVT table is a real, RAD-published, index-depended-on table — declared in Tables.cs:119-120 and included in SynchronizationDependencies (Tables.cs:25-30), so DependedOnByAnIndex() returns true and the RAD consumer does not early-return it. It's also listed in ImaOpportunityDocument.AppliesTo (:280), so the message passes the AppliesTo gate and reaches the resolver.

2. The mechanism is real. GetIndexMapper (OpportunityResolverHandler.cs:121-311) has branches for the LDC, FrequencyRegulation, and Solar sub-tables but none for LVT → it falls through to the default throw new ArgumentOutOfRangeException (:313-316). Classic accretion gap: LVT was added to Tables + AppliesTo + the data fetch, but not to the resolver switch.

3. No mitigation — and the blast radius is large. UpdateIndexesForMessagesAsync (DataChangeResponderService.cs:58-96) builds the per-index work and await Task.WhenAll(tasks) with no try/catch anywhere in the chain (incl. IndexService.IndexForIdsAsync). So the throw faults the whole operation → the RAD consumer's Handle sees an unhandled exception → the queue message is nacked and redelivered → poison-message churn (repeated failures until dead-lettered).

Impact: any edit that writes the LVT sub-table never indexes (LVT opportunities silently stop updating in search), and the failing message churns the consumer / fills the dead-letter queue (the same class the CuttlefishReport dead-letter monitor watches). Latent until an LVT opp is edited — likely why it hasn't been noticed — but high-impact and an unhandled crash on a normal data path when it does.

Fix direction: add the missing IndustrialLargeVolumeTransportOpportunities case to GetIndexMapper (mirror the LDC/Solar/FreqReg branches — the data-fetch side already handles LVT at GetEntitiesByIdsAsync). Optionally wrap per-index resolution so one bad branch can't poison the batch.

Confirmed bug D — sales-rep rename doesn't refresh global search MED

Where: ImaGlobalSearchDocument.AppliesTo vs GlobalSearchResolverHandler · Index: ima-globalsearch · Trigger: a name change on Agent.OrganizationAgents.

Why we believe it's a bug

1. The trigger fires. OrganizationAgent changes do produce RAD messages — confirmed because the Party, Opportunity, Account, Map, and dup-id resolvers all list OrganizationAgents in their AppliesTo and reindex correctly on it.

2. The mechanism is real. GlobalSearchResolverHandler.MapToIndexActions has a handler branch for Agent.OrganizationAgents (:209-215), but ImaGlobalSearchDocument.AppliesTo (:215-251) lists Agent.OrganizationGroups and not Agent.OrganizationAgents. The AppliesTo gate (ResolverHandlerExtensions.cs:18-21) drops the message before routing, so that handler branch is dead for global search.

3. No compensating path. No automatic ES reindex on rep rename was found — only the manual admin reindex mutation / console tool refreshes it. (The DB-side RepopulateSalesRepNames proc corrects names in SQL; it does not re-index global search.)

Impact: the denormalized contractSalesReps / opportunitySalesReps fields in ima-globalsearch go stale when a rep is renamed — global searches by the new name miss those parties, and the rep-boost ranking (GlobalSearchClient.cs:503-513) stops boosting renamed reps — until a manual full reindex. Data is stale, not lost; scoped to renamed reps' parties.

Fix direction: add Tables.Crm.Agent.OrganizationAgents to ImaGlobalSearchDocument.AppliesTo (the resolver branch already exists and would then activate).

Investigated & cleared (not bugs)

B — dup-id "Aliases throws": REFUTED. The reverse of the suspicion is true — DuplicateIdentificationResolverHandler does have a Party.Aliases branch (:156-163); the document's AppliesToTables (:56-70) simply doesn't list Aliases, so the message is filtered out and never reaches the switch. No ArgumentOutOfRangeException. (There may be a minor freshness gap — alias changes don't trigger a dup-id reindex — but that's a product-intent question, not a crash.)

C — global-search query-string MM exposure: REFUTED (dead code). FetchQueryStringGlobalSearchResultsAsync does skip the isMassMarket filter, but a repo-wide grep finds no live caller — the two GraphQL resolvers use the filtered methods. It's an unused method on the interface; no active exposure. Worth removing/fixing defensively, but not a live defect.

E — maps lat/lng id collapses entities: REFUTED (by design). The map doc is keyed by coordinate on purpose (one pin per location); collocated parties/accounts are merged into the pin's Parties[] / PartyAccounts[] arrays (ImaMapDataProvider.cs:121-237), not overwritten. The commented-out address fields are the intentional knob that collapses sub-unit precision. No data loss.

Recommended actions

  • File bug A (HIGH) — missing LVT resolver branch / poison-message risk. Add the branch; consider isolating per-index resolution failures.
  • File bug D (MED) — add OrganizationAgents to global-search AppliesTo so rep renames reindex.
  • No action on B, C, E (verified non-issues). Optional cleanup: remove the dead FetchQueryStringGlobalSearchResultsAsync (C) and decide product intent on dup-id alias freshness (B).
