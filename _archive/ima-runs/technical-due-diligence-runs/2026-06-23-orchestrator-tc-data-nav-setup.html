Test-Data & Navigation Setup — TC-Creation Slice of the Orchestrators

Adversarial review of how /three-amigos → /ready-for-testing (and the /sql-data path they invoke) set up the data and navigation a test case needs. Thesis under test: the pipeline authors elaborate, over-precise bespoke SQL where reusing an existing finder (sqlDB.getRandomParty() and its ~120 siblings) would be more than enough.

Verdict — the three findings that most change the process

The thesis holds, and it has one root cause. The reuse surface the tests actually consume — the sqlDB.* façade in sql.db.ts (~120 named finders, 167 call-sites across 31 spec files) — is structurally invisible to the skill that is supposed to drive reuse. So the pipeline matches against a different, staler catalog and composes bespoke SQL when that misses, even when a perfectly good finder exists.

1. [High · F1] The reuse façade is not in /sql-data's match path. Its decision step ranks sql-library.md (markdown) then raw queries.*.ts strings — sql.db.ts appears once, as a passing footnote. The skill literally cannot recommend getRandomParty() because it never looks there.

2. [High · F5] Over-precision manufactures the user gate you're trying to avoid. The heavyweight 6-option "no INT data" stop fires on a zero-row return, and its own cause-list ranks "intent string was too specific" first. The more bespoke the query, the more often the human gate fires. Reuse-first is the way to fewer gates, not more.

3. [High · F3] Bespoke queries skip the column-name safety net. queries.*.ts are validated by sql.queries.spec.ts; a Path-B per-ticket query is not, so it re-rolls the documented #1 failure mode (column mismatch → silent 0 rows) every single ticket.

Bottom line: a deterministic fix exists and needs no new user gate. Make the sqlDB.* manifest the primary match source, insert a "minimum-sufficient-shape" rule ahead of compose, and re-anchor the upstream precondition authors to finder names. A single minimal one-line confirm is the only gate, and only when reuse genuinely can't be auto-decided. See Recommendations.

Dimension 1 — Architecture, Coupling & Boundaries

HIGH F1 — The sqlDB.* reuse façade is not in /sql-data's match path

Evidence · sql-data/skill.md:54–92, 239  ·  sql.db.ts:839–949

/sql-data Step 2 ("Load the Relevant Query Source — check sources in priority order, stop at the first match") lists exactly two sources:

### Primary source — sql-library.md
### Secondary source — TypeScript query files
   src/utils/sql/queries/queries.party.ts        ← Party domain
   src/utils/sql/queries/queries.opportunity.ts  ← Opportunity domain

The façade that the tests actually call — src/utils/sql/sql.db.ts, exporting sqlDB with ~120 ready-to-call finders — is named exactly once in the whole skill, as a passing pointer at line 239: "…TypeScript sqlDB.* helper table." It is not a match source. So the skill resolves an intent against sql-library.md (the markdown catalog) and the raw query strings, but never against the callable finder layer.

Why it matters: this is the root cause of the whole thesis. The skill cannot answer "is there already a finder good enough for this?" because it doesn't look at the finder library. getRandomParty() is invisible to the very decision meant to prefer it. Every other finding downstream is a symptom of this one.

Severity rationale: High — it governs reuse for every data precondition on every ticket; the fix is the highest-leverage single change.

Remediation sketch: make a mechanically-generated sqlDB.* manifest the primary match source in Step 2 (see R1).

MEDIUM F2 — Three "query library" tiers, no single source of truth, silent drift

Evidence · sql-data/skill.md:58–80  ·  sql-library.md:1–4  ·  sql.db.ts:839–949

There are three overlapping representations of "the queries": sql-library.md (markdown catalog, header says Last ingested: 2026-06-09), queries/*.ts (raw SQL strings, validated by the sanity spec), and sql.db.ts (the sqlDB.* façade adding pick() + return types). The skill's own promotion rule acknowledges only two of them:

**Promotion rule:** When a new query is added to the TypeScript files, add it to
sql-library.md … — TypeScript is a subset of the library, not a parallel catalog.

The façade is a third tier with no stated relationship. Intent→query resolution must implicitly cross all three, and they drift independently (the markdown is dated; the TS is live and grows per-ticket).

Why it matters: the skill matches against the layer most likely to be stale (markdown) and least likely to be callable, then falls through to composing new — exactly the wrong default. Reviewers can't tell which tier is authoritative.

Severity rationale: Medium — a maintainability/coupling tax that compounds; not itself a runtime break.

Remediation sketch: declare sql.db.ts the SSOT for "what's reusable" and derive the manifest from it; demote sql-library.md to human-readable annotation (see R1).

Dimension 2 — Test Coverage, Quality & Gaps

HIGH F3 — Bespoke (Path-B) SQL bypasses the column-name sanity net

Evidence · sql-data/skill.md:67–69, 104–122  ·  sql-library.md:30–60

The skill itself states the safety guarantee that only library queries enjoy:

If a TypeScript implementation exists, use that function body exactly — it has been
run through the SQL sanity spec (src/tests/sql/sql.queries.spec.ts) so column names
and joins are confirmed working.

And the catalog documents ~30 column-name traps whose failure mode is silent: sql-library.md:36 notes Party.Accounts.IsActive does not exist; :62 warns mismatches are "the #1 cause of silent zero-row returns." A Path-B query composed fresh per ticket is not in the sanity spec, so it carries the full risk of that #1 failure mode. Reusing a sqlDB.* finder is correct-by-construction; composing bespoke re-rolls the dice every ticket.

Why it matters: the over-precise bespoke query is both more work and less safe than the reuse it displaces — it trades a verified finder for an unverified one. And when it returns 0 rows from a typo, the cost lands on the human (F5).

Severity rationale: High — it re-introduces a documented, high-frequency failure mode on a path that runs every ticket.

Remediation sketch: route to a verified finder first; only compose when no finder's shape can satisfy the assertion, and require a backlog entry when you do (R2 + R5).

MEDIUM F4 — No "minimum-sufficient-shape" rule → preconditions over-constrain vs. the assertion

Evidence · three-amigos/skill.md:412–418  ·  ready-for-testing/skill.md:620–650  ·  queries.party.ts (loose↔precise spectrum)

Both authors derive the precondition from the domain description, never from "what does the key assertion actually read?". three-amigos Phase 3 writes a free-text precondition ("party with at least one linked account"); rft Step 5.0 turns it into {entity type} that is {state/shape constraints}. Nothing in either skill says drop any constraint the assertion does not depend on. Yet the library already encodes the full loose↔precise spectrum — getRandomParty() (sql.db.ts:59) at the loose end, getCertifiedParty() / getPartyWithOpenRenewalTaskAndRenewalOpp() at the precise end — so a happy-path TC that only needs "a party" has a one-call answer that the process never steers it toward.

Why it matters: over-constraint is what turns "reuse getRandomParty" into "compose a 5-predicate bespoke query." It shrinks the candidate pool (more zero-row stops, F5) and manufactures uniqueness where the test needs none.

Severity rationale: Medium — drives F1/F3/F5 but is itself an authoring-discipline gap, fixable by a rule.

Remediation sketch: add a "minimum shape from the assertion" rule to Step 5.0 and to /sql-data Path A′ — a constraint is admitted only if the key assertion reads it (R2 + R3).

Dimension 4 — Error Handling & Failure Modes

HIGH F5 — Over-precision manufactures the very user gate the owner wants to avoid

Evidence · ready-for-testing/skill.md:675–704  ·  sql-data/skill.md:190–211

On any zero-row return, rft Phase 5 halts the whole phase and renders a heavyweight human gate — and ranks the causes itself:

If `Findings` contains "no data available…": **STOP. Do not dispatch remaining
agents or proceed to the gate.**
   ⛔ No INT data — precondition: {intent string}
   Possible causes:
     [ ] Intent string was too specific — state constraint rules out all entities  <-- cause #1
     …
   What would you like to do?  (a)…(b)…(c)…(d)…(e)   <-- 6-option human gate

So the current design creates the human interruption the owner is trying to escape: the more bespoke/precise the query, the more often it returns 0 rows, the more often this stop fires. The user gate isn't a safety valve being over-used — it's a direct downstream consequence of not reusing.

Why it matters: this is the bridge between the thesis and the owner's stated goal. "Prefer reuse" and "fewer user gates" are the same lever pulled from two ends. Reuse-first lowers gate frequency without removing the safety net for the genuinely-rare cases.

Severity rationale: High — it's the user-facing cost of the architectural gap, on the per-ticket happy path.

Remediation sketch: reuse-first eliminates most zero-row stops; replace the 6-option block for the common case with a single one-line reuse-vs-compose confirm (R6).

MEDIUM F6 — Navigation has the same reuse-blindness as data — and no safety net at all

Evidence · tc-template-3-playwright.md:43–48  ·  site-navigator.ts:27–196

The shell template points at the data façade but leaves navigation as free-text prose:

// Precondition: {entity shape} — use sqlDB.{method}() or sampled ID {id}
// Step 1: {UI action — e.g., navigate to party record}   <-- prose, no helper named

Meanwhile authFixture exposes 13 ready, fixture-injected methods — partyFullscreen(partyId, tab), opportunityFullscreen(), summaryPanel(partyId, tab), duplicateCheck(), contracts() … — and each one bakes in the correct URL shape and a load-assertion (e.g. partyFullscreen calls verifyPartyFullscreenLoaded(), site-navigator.ts:151–158). Because the orchestrator describes nav in prose, shell-writers hand-roll page.goto(...) + an ad-hoc wait, re-deriving exactly what the navigator already encodes — and skipping the free load-assert. The grep confirms the navigator is the established norm (used across 20+ specs), so the shells diverge from the house pattern.

Why it matters: same waste as the data side, with strictly worse downside — a hand-rolled goto without the verify*Loaded() gate is a classic flaky-wait source, and there's no sql.queries.spec-style net to catch it.

Severity rationale: Medium — reliability tax on E2E shells; contained to the nav step.

Remediation sketch: the Step-1 nav comment should name a authFixture method, mirroring the data line (R4).

Dimension 5 — Tech Debt & Maintainability

MEDIUM F7 — The promotion backlog exists but is dead — the "compose-once, reuse-forever" loop never closes

Evidence · _shared/sql-helper-backlog.md (whole file — 0 entries)  ·  sql-data/skill.md:215–232

sql-helper-backlog.md is explicitly wired as the place bespoke shapes get queued for promotion into sqlDB.* — its header reads "Populated by /code-intelligence Step 1C.7 and /sql-data on new query composition." The table is empty; the only rows are the commented-out format example. So even when Path B composes a genuinely new query, it is never promoted to a finder. Duplication accretes silently and the loop that would converge the suite toward reuse never runs.

Why it matters: without the closing loop, every bespoke query is permanent debt. The library's growth (good) and the bespoke sprawl (bad) are indistinguishable because nothing forces promotion.

Severity rationale: Medium — compounding debt; the mechanism is built but inert.

Remediation sketch: make the Path-B exit require a backlog row + a suggested sqlDB.* name; surface the backlog in /sync (R5).

LOW F8 — The API shell template imports from a removed alias

Evidence · tc-template-3-playwright.md:98

The API reference shell scaffolds: import { myMutation } from 'queries/myMutation';. Per the project's own record (CLAUDE.md), the queries/* alias was removed in the feature-first data migration — GraphQL ops now resolve via data/* (data/<domain>/<domain>.queries.ts). Any shell scaffolded verbatim from this template carries a broken import that npm run type-check will reject.

Why it matters: minor, but it's a defect in the scaffolding output found inside the reviewed slice — every API shell author hits it.

Severity rationale: Low — localized, caught immediately by type-check; one-line fix.

Remediation sketch: change the template import to import { myMutation } from 'data/<domain>/<domain>.queries';.

Recommendations — Fine-Tuning Toward Reuse-First

Design stance

The owner's preference order — deterministic fix first, user gates only as a last resort — is achievable. R1–R5 are fully deterministic and add no user gate; they remove the conditions that make today's gate fire. R6 is the single minimal gate, reserved for the genuinely-undecidable case. Implement in order; R1 is the keystone and unblocks the rest.

R1 · DETERMINISTIC · KEYSTONE Make a mechanically-generated sqlDB.* manifest the primary match source

Add a tiny build step (sibling to the existing check-data pre-commit guard) that parses sql.db.ts exports + each finder's JSDoc one-liner into a flat sqlDB-manifest.md: name → returns → params → one-line shape. Zero human upkeep — it regenerates from the source of truth. Then rewrite /sql-data Step 2 so the priority is: (1) sqlDB.* manifest → (2) sql-library.md (annotation) → (3) compose. This fixes F1 and demotes F2's drift (the façade becomes the SSOT; the markdown becomes commentary).

R2 · DETERMINISTIC Insert Path A′ — "reuse-first / minimum-sufficient-shape" — ahead of compose

Between today's Path A (exact match) and Path B (compose), add Path A′ with an explicit ladder: (a) exact finder; (b) the loosest finder whose returned shape satisfies the TC's key assertion — dropping every constraint the assertion does not read; (c) a parameterized finder (getPartyWithNoteContainingWord(word)-style); only then (d) compose bespoke. Encode the admission test verbatim: "a precondition constraint is allowed into the query only if the key assertion depends on it." Fixes F4, sharply reduces F3.

R3 · DETERMINISTIC Re-anchor the upstream precondition authors to finder names

three-amigos Phase 3 should suggest a callable sqlDB.* name (resolved against the R1 manifest), not a sql-library.md ID — so the suggestion is directly usable downstream. rft Phase 5 Step 5.0 should build the intent from the assertion's minimum shape and match it against the manifest before any Agent dispatch — i.e. resolve reuse in-orchestrator and only dispatch /sql-data for true misses. Kills over-precision at the source (F4) and cuts dispatch volume.

R4 · DETERMINISTIC Navigation parity — name a authFixture method in the shell template

Change the shell template's Step-1 line from prose to a helper call, mirroring the data line:

// Precondition: {entity shape} — use sqlDB.{method}() or sampled ID {id}
// Step 1: navigate — authFixture.{method}(...)  e.g. authFixture.partyFullscreen(partyId)

And add a one-line rule: prefer a authFixture method over page.goto — it carries the correct URL shape and a built-in load-assert. Fixes F6; same reuse-first principle, free flakiness reduction.

R5 · DETERMINISTIC Close the loop — make Path B require a backlog entry

When Path B genuinely composes new SQL, make the skill's exit require appending to sql-helper-backlog.md with a suggested sqlDB.* name (the hook already exists — it's just never fired). Surface the backlog count in /sync so bespoke sprawl is visible and convergent. Fixes F7; turns each bespoke query into a one-time event.

R6 · MINIMAL GATE · LAST RESORT One-line reuse-vs-compose confirm — only when undecidable

For the rare case where the assertion needs a shape no finder covers, replace the heavyweight 6-option no-data block (for this case only) with a single confirm: "Closest finder is getX() + one extra predicate {p}. Reuse getX() (good enough) / compose bespoke?" This is the only place a human is asked, and only because R1–R3 could not auto-decide. The full 6-option stop remains for the true zero-data case (no candidate at all). Honors the owner's "gates as last resort" while keeping the safety net.

Sequencing & effort: R1 is the unlock (≈ a small script + a Step-2 rewrite). R2/R3 are skill-text edits gated on R1's manifest. R4 and R8-fix (the F8 alias) are one-line template edits, do them immediately. R5 is a skill-exit rule + a /sync check. None requires touching sql.db.ts itself — the reuse library is already strong; the fix is teaching the process to look at it.
