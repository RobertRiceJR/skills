Verdict — the case, made

review-pr → KEEP-but-watch (or fold into /sync-skills). It is the only thing that validates skill.md governance — phase maps, tool names, ToolSearch loads, delegation soundness — none of which the built-in /code-review or /spec-review touch. A skill costs nothing when idle, and the gap is real now that you are actively reshaping the skill tree. The honest caveat: 0 runs. Strongest path — keep it as the PR-time governance gate, OR migrate its checks into /sync-skills (already a monthly cadence item) and retire the standalone skill.

pr-qa → RETIRE-candidate. Its unique angle (AC-driven pre-merge live-app validation → merge recommendation) is genuine, but it overlaps the built-in /verify and the /ready-for-playwright stage, has 0 runs, and its _shared/pr-qa-runs/ output folder was never created — i.e. it has never run here. Unless you expect to do AC-driven pre-merge validation routinely, /verify covers the ad-hoc need.

The bar: orchestrators + tools

principle what earns a place

  • Orchestrator — drives a lifecycle (three-amigos, ready-for-*). Keep.
  • Tool — reusable, called by orchestrators or run often (sql-data, code-intelligence, the reporters). Keep.
  • Maintenance/cadence — periodic health/drift, now wired into the DSU cadence (sync-knowledge, sync-skills, roast-me). Keep, scheduled.
  • One-off / standalone — not in a lifecycle, not delegated-to, often 0 runs. Must justify, or retire. ← review-pr and pr-qa live here.

review-pr

KEEP-WATCH the only skill.md governance gate

Purpose: internal PR code-standards review aimed at skill.md + spec/POM/shared-util changes — validates tool names against the MCP table, ToolSearch loads, phase-map integrity, delegation chains, permission safety. Free first pass before a paid deep review.

  • Usage: 0 ledger runs. Standalone — positioned as the pre-filter step in the skill-maintenance workflow; no orchestrator delegates to it.
  • Overlap: low. Built-in /code-review = code-correctness on a diff; /spec-review = single-spec test quality; neither validates skill.md governance. That governance layer is unique to review-pr.
  • Strongest keep: it is the sole guard against malformed skill.md (tool typos, dangling ToolSearch, phase-map breaks) — and you are actively churning the skill tree right now.
  • Strongest retire: 0 runs; if skill edits are rare or you eyeball them, the gate sees no traffic.
  • Recommendation: KEEP as-is for now, OR fold its skill.md checks into /sync-skills (already monthly cadence) and retire the standalone — consolidates governance into one scheduled auditor.

pr-qa

RETIRE-CANDIDATE unique angle, never used, overlaps /verify

Purpose: exploratory live-app PR validation via playwright-cli against INT — derive a test plan from ACs + PR description, execute in-browser, screenshot, emit a READY / NEEDS-FIXES / BLOCKED merge recommendation + a report file.

  • Usage: 0 ledger runs; its _shared/pr-qa-runs/ output folder does not exist — strong signal it has never run in this repo.
  • Overlap: medium-high. Built-in /verify already "runs the app and observes behavior" for a change; /ready-for-playwright owns AC-driven test execution (against Zephyr TCs). pr-qa wraps playwright-cli (which you have directly) and delegates to /health-check.
  • Strongest keep: it is the only AC-driven pre-merge live-app gate producing a merge recommendation — early INT signal before formal TCs exist.
  • Strongest retire: 0 runs, never produced output, and /verify + playwright-cli cover the ad-hoc need without a bespoke skill.
  • Recommendation: RETIRE unless you commit to AC-driven pre-merge validation as a habit. If retired: archive (it's a full skill) per the qe-dashboard precedent.
