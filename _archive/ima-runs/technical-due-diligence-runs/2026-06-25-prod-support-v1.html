Prod-Support v1 — Triage Operating Model Upgrade

Verdict — the three things a new owner should know before trusting v1

The upgrade is well-designed and the safety boundary is real — the autonomous poller is structurally walled off from production (G4), and today's run proved the dedup capability works on a live incident. But the verdict is “ship, with eyes open”: the most valuable new capability is unproven, and the same dedup that wins headless can also fail silently there.

  1. [High] The headline data-first upgrade has never run. Today's smoke test exercised only the known fast lane; Phases 4–6 and the change-correlation / data-state evidence layer (the whole reason for v1) have zero end-to-end validation.
  2. [High] Autonomous dedup can silently drop a novel bug. The broad signature search, run headless by the poller, can match an unrelated open ticket → return deduped → never file a genuinely new bug, and the headless agent has no tool to flag the miss.
  3. [Medium] Prod reads can leak customer identifiers into git. The new “lead with prod” step pulls live account/party data into RCA/KB files that are committed to the branch, with no warning against persisting prod PII.

Scope & Plan

Target: the prod-support skill subsystem — the interactive skill (.claude/skills/prod-support/skill.md), its evidence-layer playbook (reference/evidence-layer.md), the knowledge base (knowledge-base/), the shared bug-template SSOT (_shared/templates/bug-ticket.example.md), and the autonomous path (prod-support-poller/skill.md + .claude/agents/prod-support-headless.md). Out of scope: the rest of the repo; the elastic-drift scripts the playbook calls are assessed only as dependencies, not audited internally.

Run shape: inline (one cohesive subsystem, ~10 files). All seven dimensions in scope. Note on artifact type: this subsystem is instruction + an agent tool-surface, not compiled code — so “implementation” here means the skill prose that an LLM executes and the headless agent's enforced tools: list. Findings cite file:line and quote the governing text. This is an upgrade review, not a greenfield audit — framing leads with what changed and what it fixed, then prices the residual risk.

Phase 1 — Surface Map

Two entry points drive everything:

  • Interactive — /prod-support [CISD-XXXX] → the 0–9 phase flow in skill.md. Touches: Azure App Insights (az), mcp__mssql-{int,qa,prod} + industrial, the elastic-drift scripts (ES / RabbitMQ / Control-M), ADO MCP + ConfigCat (change correlation), the Atlassian MCP (CISD load, dedup, ticket, comment), and the git-committed knowledge-base/.
  • Autonomous — cisd-prod-support-poll (cron, every 20 min, weekdays) → prod-support-poll → one prod-support-headless subagent per CISD ticket, running skill.md Phases 0–7 only on a deliberately restricted tool surface.

Change-radius map. The highest-blast-radius node is the bug-template SSOT (consumed by /prod-support, /bug-report, and the headless path — one edit reaches three skills). Next is skill.md's phase numbering, referenced by the poller, the headless agent, and the architecture doc. The headless agent's tools: line is the trust boundary — it is the one place where “the poller cannot touch prod” is actually enforced rather than merely stated.

The Upgrade — what changed & the risks it addresses

Branch, not a line Known-vs-novel fast lane on a one-line Signature

v1 renumbered the flow to 0–9 and added a classification gate: build a matchable Signature, match it against the KB, and branch — known incidents skip Phases 4–6 straight to dedup+ticket; only novel ones get the full investigation.

skill.md:267 — "Build the **one-line Signature** from Phases 1–2 and decide whether this is a *known* incident or a *novel* one."

Risk addressed: the common (recurring) case no longer pays a full re-investigation, and the Signature gives every report/KB entry a stable join key.

Data-first Investigation now leads with change + data state, not source

Phase 5 was restructured so a novel investigation opens with change correlation (what shipped/ran in the window) and data-state inspection (SQL SOR, ES projection drift, RabbitMQ DLQ) before reading source — all read-only reuse of existing tooling via reference/evidence-layer.md.

skill.md:318 — "**Novel branch only.** Investigate **data-first** … inspecting recent change and live data *before* reading source."

Risk addressed: the code-first blind spot for a data-first platform — the bad rows / drifted projection / dead-letter backlog where the bugs actually live.

Trust boundary Autonomous poller structurally walled off from prod (G4)

The settling pass elevated prod SQL to a first-class evidence source for the interactive runner, while keeping it interactive-only. The headless agent's tool surface is the enforcement: Jira-read + CI-create + INT/QA MSSQL only — no prod MCP, no addComment/editJiraIssue.

prod-support-headless.md:8 — tools: … mcp__claude_ai_Atlassian__getJiraIssue, …createJiraIssue, mcp__mssql-int__execute_sql, mcp__mssql-qa__execute_sql (no prod, no comment/edit)

Risk addressed: an unattended cron agent can never write to production data or comment on a customer-facing ticket — the boundary is a missing capability, not a policy a tired prompt might ignore.

Dedup File-once: search open CI bugs before creating

Phase 7 now searches open CI bugs for the signature before filing — comment/increment if a match exists, file only if none. This closes the cross-shift duplicate-filing failure mode.

skill.md:541 — jql: 'project = CI AND issuetype = Bug AND statusCategory != Done AND text ~ "{operation | exception | key symptom}" …'

Risk addressed: two responders (or the poller + a human) working the same exception no longer create twin tickets. (This is the capability the proof-of-value case study below exercised live — and it is also the source of finding F2.)

Proof of Value — live smoke test, 2026-06-25

Case study — the dedup upgrade earned its keep on the first real run

/prod-support was invoked with no target, so it self-sourced an incident from live App Insights. Phase 1 surfaced a burst of 9× System.ComponentModel.Win32Exception "The wait operation timed out" (QA, 16:09–16:11 UTC). Phase 3 built the signature and — applying the “confirm root cause, not just symptom” discipline — drilled the stack to IndustrialEntities.GetImaOpportunityCustomerSegments / GetImaCustomerCustomerSegments, an exact match to a known KB asset. Fast lane → Phase 7.

The win: the Phase 7 dedup search found two open tickets already covering the signature — CI-7863 (opportunity variant, Pull Requested) and CI-7871 (broker variant). No duplicate was filed. Without this step, the run — or the autonomous poller — would have opened a near-duplicate of an in-flight ticket.

The bonus: the same dedup pass surfaced a genuine gap — the GetImaCustomerCustomerSegments variant (the highest-volume of the family) has no dedicated ticket and may sit outside CI-7863's fix scope. That was flagged as a comment on CI-7863 before its PR merges. A read-only triage produced an actionable finding for the owning dev, filed zero noise, and updated the KB — exactly the intended loop.

Honest scope of the proof: this validated the known / dedup path end-to-end. It did not exercise the novel branch or the data-first evidence layer (see F1) — the incident was a known crash, so Phases 4–6 and Steps 5.1/5.2 were correctly skipped and therefore remain unproven.

Dimension 1 — Architecture, coupling, boundaries

MEDIUM F4 · Phase 5 is path-coupled to another skill's scripts

Evidence · skill.md:334 (+ reference/evidence-layer.md)
**ES** projection drift (`elastic-drift/scripts/es-drift.mjs test → count → diff`), **RabbitMQ DLQ**
(`elastic-drift/scripts/rabbitmq-health.mjs`)

Why it matters: the data-state layer hard-references scripts owned by the elastic-drift skill by path. If that skill renames, moves, or changes the CLI of those scripts, /prod-support Phase 5 silently points at nothing — and there is no test or guard that would catch the broken reference.

Reach · Phase 5.1/5.2 (novel branch) only; degrades the evidence layer, doesn't crash the flow.
Remediation · A shared manifest of evidence-tool commands, or a thin wrapper the playbook owns, so the contract is explicit rather than a cross-skill path literal.

MEDIUM F5 · KB keys by asset-family; Jira keys by method — grain mismatch

Evidence · knowledge-base/00-index.md (asset IndustrialEntities-CustomerSegments-SqlTimeout) vs CI-7863/CI-7871 (one ticket per method)

Why it matters: the KB groups the timeout family under one asset, but the dev team filed one ticket per method. Today that mismatch produced real “is this already covered?” ambiguity — two of three methods were ticketed, one (GetImaCustomerCustomerSegments) was not, and only a careful read caught it. The dedup step's correctness depends on the responder reconciling these two grains by hand.

Reach · Phase 7 dedup accuracy on any multi-method/multi-symptom asset family.
Remediation · Record the filed CI keys per-method inside the asset file's Signature lines (done for this asset today) so dedup can map family→tickets without re-deriving.

Dimension 2 — Test coverage & gaps

HIGH F1 · The headline upgrade — the novel / data-first branch — has never executed

Evidence · skill.md:301, 318, 419 (Phases 4, 5, 6 all "Novel branch only") · today's only live run took the known fast lane
301: **Novel branch only** (skipped on the known fast lane). Parse the stack trace …
318: **Novel branch only.** Investigate **data-first** …
419: **Novel branch only.** The RCA *is* the knowledge-base entry …

Why it matters: the entire justification for v1 was the data-first evidence layer (change correlation + SQL/ES/DLQ before source). It is the most complex new machinery and it has zero end-to-end validation — no run has built a signature that missed the KB, entered Phase 5, and executed Steps 5.1/5.2. A skill is only as good as its first real invocation; this branch's first invocation has not happened. The evidence-layer commands themselves (es-drift, rabbitmq-health, controlm-health, ADO build-window correlation) are unverified as written.

Reach · Every genuinely novel incident — i.e. exactly the cases the skill exists to handle well.
Remediation · Run the second smoke-test leg: a behavioral incident with no KB match, confirm 5.1/5.2 fire before source, and spot-check one command per evidence-layer reuse row. (Already tracked as the open TODO item.)

Dimension 3 — Security surface & trust boundaries

MEDIUM F3 · Prod reads can persist customer identifiers into the git-committed KB

Evidence · skill.md:329 (lead with prod) + skill.md:466 (commit the RCA)
329: **SQL** SOR — **lead with prod for a prod incident** (`mcp__mssql-prod__execute_sql` / …)
466: Commit the RCA to the current branch so it's linkable from the CI ticket.

Why it matters: v1 rightly tells the interactive responder to diff prod state first — but the RCA/KB files that capture that evidence are committed to the branch (and the KB is full-text in the repo). Account numbers, party names, and other live customer identifiers pulled from prod can land in version control with no prompt to redact or to reference IDs abstractly. This is a confidentiality exposure into an internal-but-broadly-readable surface, not a breach, hence Medium — but it compounds silently across incidents.

Reach · Every prod-incident RCA written under the new “lead with prod” guidance.
Remediation · One line in Phase 6: “reference entity IDs, not customer-identifying values; never paste raw prod PII into a committed RCA.” Pairs with the existing data-exposure close-out prompt.

On the boundary itself — no material finding: traced the headless agent's tools: list (prod-support-headless.md:8) and the poller preamble — the autonomous path genuinely cannot reach prod data, comment on CISD, or move a sprint. The boundary is enforced by absent capability, which is the correct design. The interactive prod MCPs are documented SELECT-only + auto-rollback; that safety rests on an external invariant (the MCP config) the skill cannot itself enforce — acceptable, but worth noting.

Dimension 4 — Error handling & failure modes

HIGH F2 · Autonomous dedup can silently drop a genuinely novel bug

Evidence · prod-support-poller/skill.md:105 + prod-support-headless.md:35-36 + the broad match in skill.md:541
poller:105 — Phase 7.1 dedup: search open CI bugs for the signature. On a match,
DO NOT file a twin (no editJiraIssue / addComment — G1/G5) → return status `deduped` + the existing CI key.
skill.md:541 — … text ~ "{operation | exception | key symptom}" …

Why it matters: the dedup win has a mirror failure mode in the unattended path. The match is a broad Jira full-text search on operation/exception/symptom. If a new incident's signature happens to match an unrelated open bug (common exception types — NullReferenceException, a shared operation name — are exactly the broad case), the headless agent returns deduped and files nothing — and because it deliberately lacks addComment/editJiraIssue, it cannot even leave a “possible duplicate, please confirm” note. A real novel prod bug is then silently absent from the backlog, recorded only as a one-line deduped in the run log. The “confirm root cause, not just symptom” discipline that prevents this (skill.md:287) is prose guidance an autonomous run has no mechanism to enforce.

Reach · The cron poller, on any incident whose signature collides with an open ticket — unattended, so a miss is invisible until someone audits the run log.
Severity rationale · High — a silently dropped novel production bug in an automated path is the failure an owner least wants, even though no data is corrupted.
Remediation · Tighten the headless dedup to require a stronger match (exception type AND key frame, not free-text OR), and on a match have headless emit a distinct deduped-uncertain that the orchestrator surfaces for human confirmation rather than closing.

Dimension 5 — Tech debt & maintainability

MEDIUM F6 · Phase numbers are an un-enforced cross-document contract

Evidence · the 0a→0–9 renumber rippled into ≥5 docs; the poller carried a stale "Phases 0a–5" until reconciled (poller:103/105 now correct), and the session-review caught two consumers (skill-architecture.md:555, bug-report/skill.md) missed on the first pass.

Why it matters: the skill's phase numbering is referenced by prose in the poller, the headless agent, and the architecture doc — none of it checked by any tooling. The renumber this session silently staled every one of those references; only a manual sweep (and a follow-up session-review sweep) found them all. Each future renumber pays the same tax, and a missed reference reads as correct until someone executes it. This is the duplicated-fact-drift class (bug-patterns #19) applied to phase identifiers.

Reach · Any future re-sequencing of the flow; the autonomous poller is the most dangerous consumer because a stale phase range there changes what runs unattended.
Remediation · Refer to phases by stable name (“the dedup phase”) rather than number in cross-skill docs, or add a cheap grep check to /sync-skills that flags Phase \d references to prod-support outside the skill itself.

LOW F8 · The required Signature is applied even to lightweight bug-report runs

Evidence · bug-ticket.example.md:27
27: - **Signature** — **required.** The matchable one-liner that links this report to the KB, the fast lane, and dedup.

Why it matters: the Signature is load-bearing for prod-support's fast lane, but the field is required on the shared template, so a quick /bug-report for a one-off test failure must now also produce a fingerprint it will never match against. Minor friction, and the upside (uniformity) likely justifies it — flagged for the record, not for action.

Remediation · Accept as-is, or mark Signature “required for prod-support; recommended for /bug-report”.

Dimension 6 — Dependency health & version risk

MEDIUM F7 · The Atlassian MCP is a single point of failure for three phases — and it failed today

Evidence · prod-support-headless.md:8 (Atlassian tools in the surface) · observed live: the connector's token lapsed mid-session, and Phases 2 / 7 / 8 could not run until it was re-authenticated.
8: tools: … mcp__claude_ai_Atlassian__getJiraIssue, …searchJiraIssuesUsingJql, …createJiraIssue, …

Why it matters: CISD load (Phase 2), dedup + ticket (Phase 7), and the CISD comment (Phase 8) all hard-depend on the claude_ai_Atlassian connector. Interactively a lapse is obvious — it blocked this very run. In the cron poller it is more dangerous: an expired OAuth token would make every headless run fail its filing step, and the failure surfaces only as failed/parked entries no one is watching in real time. There is no pre-flight auth check before the poller fans out.

Reach · Every interactive and autonomous run that reaches Phase 2 or beyond.
Remediation · Add a one-call Atlassian auth pre-flight to the poller's Phase 1 (mirroring /health-check); on failure, log a single run-level "AUTH-FAILED" line and skip the run rather than parking every ticket.

Dimension 7 — Performance hotspots

LOW F8b · Phase 0 re-reads the full KB + 44 patterns on every run

Evidence · skill.md:84
84: `.claude/skills/_shared/knowledge/reference/bug-patterns.md` (the 44 numbered patterns) and note the
    asset files' `**Signature:**` lines — load them into context …

Why it matters: Phase 0 loads the index, routing files, the 44-entry bug-patterns catalog, and matching asset files into context before any triage. For the interactive case this is negligible. For the cron poller (every 20 min, up to 20 tickets/run, one subagent each) it is repeated context cost paid per ticket. Not a correctness issue; a tokens/latency tax that scales with poller volume.

Remediation · Have the poller pre-load the KB index/signatures once per run and pass a distilled signature list into each subagent, rather than each subagent re-reading the full catalog.
