Verdict — bottom line up front

Two direct answers. (1) Yes — four devDependencies are provably dead and can be removed today, plus one duplicate declaration. (2) No — you cannot "install only dependencies" in the test pipeline: the test runner (@playwright/test) and codegen (@graphql-codegen/cli) are devDependencies, so npm ci --omit=dev would delete the very tools CI runs. In a test-automation repo there is no "production runtime" — the dev/prod split does not map to "needed-in-CI." The real levers are narrower and better:

  1. Dashboard pipeline → npm ci --omit=dev — its runtime graph is only dotenv + Node built-ins; it never runs Playwright/codegen. This is the one pipeline where "install only prod deps" is real, and it also stops the wasted better-sqlite3 native build it triggers today.
  2. Reclassify promptfoo → optionalDependencies and run the test pipeline with npm ci --omit=optional — that lets you drop the blunt --ignore-scripts and shrinks the CI download (promptfoo is the only native/local-only dep).
  3. Delete the 4 dead devDeps (c8, cross-env, @azure/microsoft-playwright-testing, duplicate @azure/identity) — smaller tree, faster restore everywhere.

Findings — Medium

MEDIUM M1 · Phantom (undeclared) dependency: @igs-quality/environments

Evidence — src/fixtures/rabbitmq.fixture.ts:1; the package appears nowhere in package.json. knip reports it as an "Unlisted dependency."

src/fixtures/rabbitmq.fixture.ts:1
import type { Environment } from '@igs-quality/environments';

Why it matters: the fixture compiles only because some other @igs-quality/* package drags @igs-quality/environments in transitively. The day that transitive provider drops or renames it, tsc --noEmit (and the CI type-check / codegen-compile gate) breaks with no change to this repo. It's type-only, so runtime is unaffected — the blast radius is a surprise red build on an unrelated dependency bump.

Remediation: declare it explicitly (devDependencies is fine since it's type-only) at the version currently resolved in the lockfile.

MEDIUM M2 · Dashboard pipeline compiles a native module it never uses

Evidence — .azpl/dashboard-pipeline.yml:74-79 runs npm ci without --ignore-scripts (unlike the two test legs). The install therefore runs postinstall builds, and better-sqlite3 is a build-from-source native module.

dashboard-pipeline.yml:77-78 + npm ls better-sqlite3
# dashboard-pipeline.yml — no --ignore-scripts, so scripts run
npm install -g npm@11
npm ci --cache $(Pipeline.Workspace)/.npm

# provenance: better-sqlite3 comes ONLY from promptfoo
test-suite-ima-360-playwright
`-- promptfoo@0.120.27
  +-- better-sqlite3@12.11.1
  `-- drizzle-orm@0.45.2 -> better-sqlite3 (deduped)

Why it matters: the dashboard job only fetches Zephyr/Jira/ADO data and deploys — it never touches Playwright, codegen, or promptfoo. It nonetheless installs the whole tree and compiles better-sqlite3 from source on every nightly run. It succeeds today only because ubuntu-latest ships a C toolchain; it is wasted minutes now and a hard failure the day the image or the prebuilt binary changes.

Remediation: switch this leg to npm ci --omit=dev — verified safe below — which skips promptfoo (a devDep) entirely, so no native build, and installs only the prod tree the script needs.

why --omit=dev is safe here — fetch-data.ts runtime graph
// scripts/dashboard/fetch-data.ts — runtime imports
import dotenv from 'dotenv';            // ← prod dependency ✓
import { resolveSecret } from './secrets.ts';      // local → node:child_process
import { scanSpec } from '../../src/utils/spec-tags.ts'; // local, no imports
// + node:fs / node:path / node:url — built-ins. No devDep at runtime.

MEDIUM M3 · Documented DB-verify fixture is unwired — @igs-quality/ima-queries is dead in the executed suite

Evidence — src/fixtures/database.fixture.ts:1 is the only importer of the package; a search of src/tests/** for database.fixture / partyQueries / ima-queries returns zero specs. knip (which follows the spec→fixture graph) reports @igs-quality/ima-queries as an unused dependency.

src/fixtures/database.fixture.ts:1 (imported by no spec)
import { PartyQueries, destroyDbConnection } from '@igs-quality/ima-queries';

Why it matters: two readings, both worth a decision. Either (a) it's dead code — the suite standardized on the in-repo src/utils/sql layer (the subject of TDD-015–018, TDD-030) and this external-package fixture is a superseded parallel path; or (b) it's a coverage signal — CLAUDE.md mandates database.fixture.ts for "DB verification … never write inline SQL," yet nothing executes it. Given the mature sqlDB.* finder layer, (a) is the likely truth. Not a blind delete: removing a documented fixture has blast radius if any out-of-tree consumer uses it.

Remediation: confirm DB verification is fully served by src/utils/sql, then retire database.fixture.ts + drop @igs-quality/ima-queries and update CLAUDE.md; or wire the fixture into at least one spec if it's meant to be live.

MEDIUM M4 · --ignore-scripts is a blunt, global workaround for one local-only dependency

Evidence — .azpl/azure-pipelines.yml:61-76 (and the mirror in playwright.yml) skips all install scripts to avoid building better-sqlite3, which (M2) comes solely from promptfoo, a local-only LLM-judge tool CI never imports.

azure-pipelines.yml:69-75 (comment abridged)
# --ignore-scripts: skip native postinstall builds. The only native module is
# better-sqlite3 (transitive via promptfoo, a local-only LLM-judge tool) — this
# pool can't fetch its prebuilt binary and the container has no compiler.
npm install -g npm@11
npm ci --ignore-scripts --cache $(Pipeline.Workspace)/.npm

Why it matters: --ignore-scripts is a sledgehammer — it silently disables every package's lifecycle scripts (incl. husky's prepare, and any future dependency that legitimately needs a postinstall). It works today, but it hides the fact that the real problem is a single misfiled dependency, and it will silently swallow a genuinely-needed build step later.

Remediation: move promptfoo from devDependencies to optionalDependencies; install in CI with npm ci --omit=optional --ignore-scripts. That excludes promptfoo + better-sqlite3 precisely (no ~76MB download, drops 717 packages) and keeps promptfoo installed for local judge work by default.

Correction (2026-07-06, verified via npm ci --dry-run): the original draft of this finding said --ignore-scripts could be dropped. It cannot. A dry-run showed keytar (node-gyp rebuild) and @azure/msal-node-* install scripts survive --omit=optional — they come from @igs-quality/azure-secrets-helper → @azure/identity-broker → @azure/msal-node-extensions, a production path. The compiler-less CI container cannot build keytar, and the original comment's claim that "the only native module is better-sqlite3" was incomplete. --ignore-scripts must stay.

Findings — Low

LOW L1 · Four dead devDependencies inflate every install

Evidence — verified unused (details in the Q1 table): c8 (no script/config), cross-env (no script), @azure/microsoft-playwright-testing (zero imports, superseded by @azure/playwright), and @azure/identity declared twice (package.json:73 dev & :100 prod).

Why it matters: pure footprint/maintainability tax — every npm ci resolves and unpacks packages no code path uses, and the duplicate declaration is a standing source of confusion (which bucket is authoritative?). No functional risk.

Remediation: remove all four (keep @azure/identity as a single dependencies entry). This is the direct, safe answer to "can we eliminate dependencies."

LOW L2 · knip ignoreDependencies drift

Evidence — package.json:132-139. knip's own run emits two config hints and one false positive:

  • @graphql-codegen/typescript-operations is flagged unused but is a live codegen plugin (codegen.int.ts:12) → add to ignoreDependencies to silence the false positive.
  • @graphql-codegen/typescript-resolvers is in ignoreDependencies but is not a declared dependency at all → remove the stale entry (knip hint: "Remove from ignoreDependencies").
  • lint-staged is now auto-detected via the lint-staged config block → remove from ignoreDependencies (knip hint).

Why it matters: stale ignore-lists erode trust in npm run dead-code — the tool you'd rely on for exactly this kind of audit. Low blast radius, quick fix.

LOW L3 · Structural: npm ci --omit=dev cannot work for the test pipeline (the Q2 answer)

Evidence — the test legs invoke devDependencies directly: npx playwright test → @playwright/test (dev, L81); npm run codegen-<env> → @graphql-codegen/cli + typescript + typescript-operations (dev); playwright.service.config.ts:2 imports @azure/playwright (dev, L75).

Why it matters: this is not a defect — it's the reason the naive "install only dependencies" optimization fails and must be recorded so nobody attempts it. A test-automation repo has no production runtime; the split between dependencies and devDependencies here is essentially arbitrary, and CI needs a mix of both. Do not "fix" this by moving the runner/codegen into dependencies so --omit=dev "works" — that misrepresents the manifest and fights every tool's expectations. The correct levers are M2 (dashboard omit-dev), M4 (promptfoo→optional), and L1 (delete dead).
