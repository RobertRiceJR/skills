Verdict — what an acquirer should price in

Grade: weak coverage, well-dressed. The two CI-7735 automated tests are clean, traceable, and honestly commented — but they verify almost nothing about the feature CI-7735 actually shipped (exporting hedge financials to a deal-sheet tab). They assert that generateDealSheetUri returns a non-empty string and no errors. That is a mutation-did-not-500 smoke guard, not feature coverage. The correctness of the exported hedge data rests entirely on two manual TCs that were Not Executed at ticket close.

  1. [Critical] A financial-export feature has zero executed correctness coverage. Automated tests assert only a truthy URI; the two manual TCs (CI-T2485/CI-T2486) that check the actual hedge values were marked "Not Executed" at close. Net executed verification of the hedge numbers a salesperson prices off of = none.
  2. [High] The tests are tautological — "hedged" is asserted nowhere. Drop in a non-hedged opportunity and both tests still pass. They are byte-for-byte equivalent in assertion to the older non-hedged CI-7733 test. The thing the ticket exists to prove is the one thing untested.
  3. [High] No failure-mode coverage at all. Happy path only — no invalid input, no missing-hedge path, no detection of a decayed fixture. A green suite here is not evidence the path is healthy.

Fair-stance caveat: the team correctly identified (Bug Pattern #30) that the WIP-vs-Live template choice and tab content are not observable via the API response, and routed content checks to manual TCs. The criticism is not "bad tests" — it is that the chosen layer cannot cover the feature's value, and the layer that can was not executed. The residual automated value is a thin regression tripwire.

Dimension 1 — Architecture, Coupling, Boundaries

MEDIUM Test owns a hand-written copy of the schema contract, and it has already drifted

Evidence: deal-sheet-wip.api.spec.ts:7-12  vs  src/generated/graphql.ts:8390

// spec — hand-written, used as the generic on every gql.request below
type GenerateDealSheetUriResult = {
  generateDealSheetUri: { dealSheetUri: string | null };
};

// generated/graphql.ts:8390 — the real schema contract
export type GenerateDealSheetUriMutation = { __typename?: 'MutationRoot',
  generateDealSheetUri: { __typename: 'GenerateDealSheetUriPayload', dealSheetUri: string } };

Why it matters: The codegen'd GenerateDealSheetUriMutation / GenerateDealSheetUriPayload types exist (the file even imports GenerateDealSheetUriInput from the same module) but the response type is re-declared by hand. The generic gql.request<GenerateDealSheetUriResult> is therefore an unchecked cast — TypeScript validates the test against a local fiction, not the schema. The hand-written copy already disagrees with the schema (string | null vs non-null string), so a future schema change to this payload will not fail type-check or any test here.

Call path / reach: Local to this file, but it is the boundary where the test's view of the API contract diverges from the generated source of truth — the repo standard (CLAUDE.md: "use generated types; run codegen before authoring") is bypassed.

Severity rationale: Medium — coupling/maintainability risk that defeats the type-safety the codegen pipeline is meant to provide; not a runtime break today.

Remediation: Delete the inline type; type the call as gql.request<GenerateDealSheetUriMutation> from generated/graphql.

Dimension 2 — Test Coverage, Quality & Gaps

CRITICAL A financial-export feature ships with zero executed correctness coverage

Evidence: deal-sheet-wip.api.spec.ts:55-57 (the entire assertion surface)  ·  ticket.md Section 9 (manual TCs Not Executed)

expect(res.raw.ok()).toBeTruthy();
expect(res.errors).toHaveLength(0);
expect(res.data.generateDealSheetUri.dealSheetUri).toBeTruthy();

Why it matters: CI-7735 exports hedge financials — FixedRate, Value, Margin, CommodityCost — into a new IMA-Hedges tab on the deal sheet a salesperson prices off of. The automated tests verify none of it: not the values, not the tab's existence, not even that the export ran the new code. Per the ticket's own close-out (Section 9): "✅ Pass 2 / ⏸ Manual (not executed) 2", and CI-T2485 / CI-T2486 — the only TCs that inspect the actual hedge data — are recorded "Not Executed." So at ticket close, the number of executed checks confirming the hedge numbers are correct is zero (automated or manual).

Call path / reach: The hedge values flow DB-Industrial GetDealSheetData (7th result set) → C# export template → .xlsm. The API response (dealSheetUri) is a desktop-launcher path and never carries the data (Bug Pattern #30), so the automated layer is structurally incapable of catching a wrong rate/margin in the export.

Severity rationale: High → Critical (money path). Per the skill's blast-radius multiplier — a deal-sheet hedge export is commercial/pricing data with no rollback once a rep quotes off it; "no executed coverage on a money path" is the Critical bar.

Remediation: Treat the API tests as regression smoke (they are fine for that) and execute CI-T2485/CI-T2486 before relying on CI-7735; better, add a DB-fixture assertion that the exported values match IMA.Hedges for the fixture, or a backend/file-level check that opens the generated .xlsm.

HIGH Tautological tests — "hedged" is asserted nowhere; identical to the non-hedged sibling

Evidence: deal-sheet-wip.api.spec.ts:43-58 / 60-75 (CI-7735 hedged)  vs  :94-112 (CI-7733 non-hedged)

// CI-T2488 (hedged WIP)            // CI-T2413 (NON-hedged WIP, sibling)
expect(res.raw.ok()).toBeTruthy();   expect(res.raw.ok()).toBeTruthy();
expect(res.errors).toHaveLength(0);  expect(res.errors).toHaveLength(0);
expect(…dealSheetUri).toBeTruthy();  expect(…dealSheetUri).toBeTruthy();
// only difference: the fixture IDs. The "hedged" property is never read or asserted.

Why it matters: The ticket's reason to exist (per Section 1 / TC-4) is "a confirmed hedged opportunity doesn't break the mutation." But nothing in the test confirms the opportunity is hedged, and the assertions are byte-for-byte identical to the pre-existing non-hedged CI-7733 test. Swap HEDGED_ELECTRIC_SAMPLE for any valid ELECTRIC opportunity and both tests still pass. The distinguishing variable is unobserved → the test cannot fail for the reason it was written.

Call path / reach: Self-contained, but it means CI-T2487/CI-T2488 add ~no signal over the existing CI-T2413 coverage — three tests, one effective assertion.

Severity rationale: High — a test that passes regardless of the condition it claims to exercise is false confidence on a money-adjacent feature; just under the Critical (which is the total absence of executed correctness coverage, finding 1).

Remediation: Either assert a runtime hedge precondition (DB count > 0 for the fixture's ContractTermId) so the "hedged" framing is real, or fold these into the regression smoke set and stop labeling them as hedge-feature coverage.

MEDIUM Fixture has no runtime self-verification — silently degrades to a false positive when it decays

Evidence: deal-sheet-wip.api.spec.ts:22-34 (fixture + "re-verify" comment)  ·  :55-57 (no hedge assertion)

// Sampled INT 2026-06-24. Re-verify via Industrial query if hedge expires:
//   …WHERE sl.PartyAccountId=60026695 AND h.EndDate >= GETDATE()
const HEDGED_ELECTRIC_SAMPLE = { partyId: 33706605,
  opportunityIds: [79010179], partyAccountIds: [60026695] };

Why it matters: The comment correctly flags that the hedge can expire and a human must re-verify — but nothing in the test enforces it. The knowledge base already shows this data is volatile (ticket.md: contract 114333 "previously noted with 9 hedges" now has none; "hedge data sparse in INT — only ~5 contract terms carry hedges"). When this fixture decays the same way, the test keeps passing (it only checks for a URI), so the decay is invisible — the worst failure mode for a fixture-backed test.

Call path / reach: Couples the test's validity to a specific, perishable INT row across two databases (CRM ↔ Industrial join). Maintenance burden is real and recurring.

Severity rationale: Medium — degraded confidence with a manual workaround (re-sample), not a break; compounds with finding 2.

Remediation: Assert the precondition in-test (or in a setup step) via the database.fixture — fail loudly if the fixture has 0 active hedges, converting silent decay into a clear red.

Dimension 3 — Security Surface & Trust Boundaries

MEDIUM Test encodes "no authorization control" as the expected contract

Evidence: deal-sheet-wip.api.spec.ts:82-86 (rationale)  ·  :107-111 (assertion)

// The generateDealSheetUri resolver carries no @authorize directive — gating is
// frontend-only … an authenticated caller WITHOUT the permission can still request a WIP deal sheet.
// No transport/authorization error → the mutation is NOT gated on the permission.
expect(res.raw.ok()).toBeTruthy();
expect(res.errors).toHaveLength(0);

Why it matters: This test is valuable as documentation — it surfaces a real production trust-boundary fact: generateDealSheetUri has no backend @authorize, so the "Load Following Deal Sheet" permission is enforced in the UI only. But the test asserts the absence of the control as the pass condition. Two consequences for an acquirer: (1) there is no test anywhere proving the permission is enforced on any path — the gating is untested by construction; (2) if backend gating is ever added (the correct hardening), this test flips to a false failure and reads as a regression. The underlying production posture — an authenticated user can drive a WIP export they shouldn't see — is "acceptable as designed (ticket B-2)" but is a standing API-authz gap, not a test defect.

Call path / reach: Documents the resolver's missing directive; no other test covers the permission, so the authz story for this mutation begins and ends with "it isn't enforced."

Severity rationale: Medium — security-relevant trust-boundary gap, mitigated by frontend gating and explicit sign-off; not bypass of a control that exists, so below High.

Remediation: Keep the documenting assertion but add a comment tying it to ticket B-2, and track the backend-authz decision; if gating lands, convert this to a positive authz test rather than deleting it.

Dimension 4 — Error Handling & Failure Modes

HIGH Happy-path only — no negative, business-error, or transport-failure coverage

Evidence: deal-sheet-wip.api.spec.ts:36-113 (all 3 tests)  ·  fixtures/graphql.fixture.ts:44-51 (silent error shape)

// graphql.fixture.ts — on a non-200, errors becomes [] and data becomes undefined:
if (response.ok() === false) { data = undefined; errors = []; }
// so res.errors.toHaveLength(0) PASSES on an HTTP failure; only res.raw.ok() guards it,
// and res.data.generateDealSheetUri.dealSheetUri would then throw a TypeError, not assert-fail.

Why it matters: Every test exercises the success path. There is no coverage of: an invalid/mismatched partyId/opportunityIds, a missing required input, the empty-hedge path (TC-2, manual only), or a server error. Note the fixture's failure shape: on a non-200 it returns errors: [], so toHaveLength(0) would pass on a failed request — the only thing standing between a 500 and a green test is the res.raw.ok() ordering, and a body-level GraphQL error on a 200 would surface only because data is then read. The suite has no test that deliberately drives any of these, so the mutation's failure behavior is entirely uncharacterized by this file.

Call path / reach: The error-swallowing in the shared fixture (graphql.fixture.ts:44-46) affects every API test in the suite; this file just never probes it.

Severity rationale: High — an unexercised failure mode on a money-adjacent mutation; no test would catch a regression that turns valid input into an error or vice versa.

Remediation: Add at least one negative test (e.g. mismatched account/opportunity) using requestExpectingBusinessError and assert on the business-error field, per the fixture's documented IMA error model.
