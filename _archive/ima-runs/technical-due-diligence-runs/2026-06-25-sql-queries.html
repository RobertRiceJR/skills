Verdict — the three things to fix first in this slice

These six modules are the heart of the layer and, on the whole, are careful work — every query is TOP-bounded, most apply the CI filter, and the merge finders show real engineering discipline. The risks are about how the SQL is assembled, not what it computes:

  1. [High] String parameters are interpolated straight into SQL, not bound. Utility codes, LOB names, note text, event-type names, and a party number are dropped into WHERE clauses with ${...} against a credentialed connection to live INT/QA/PROD. Inputs are test-controlled today, but the layer offers no parameterization barrier — and the Industrial config one folder over already does this correctly with .input().
  2. [Medium] The CI filter is copy-pasted ~80+ times, never extracted. The same RelationshipManagerId = 1 + 1-year CreateDate window is duplicated across nearly every query — and drift is already visible (a 2-year window in two account queries). The file proves it knows how to extract a predicate (MERGE_SAFE_PARTY_PREDICATE) — it just didn’t do it for the most-repeated one.
  3. [Medium] Terminal-state filtering uses two mechanisms with drifting string sets. Some queries gate on ImaOpportunityStatusId NOT IN (7,8) (the documented convention); others on os.Name NOT IN (...) string lists that disagree across files ('Lost' present in some, absent in others) — two ways to express one rule, inconsistently.

Net: introduce parameter binding, extract the CI filter and the status gate into shared fragments, and replace the magic-ID literals with a named registry. High-value, mechanical, low-risk.

Phase 1 — Surface Map

Public surface & change-radius

  • Each module exports a queries<Domain> object of pure string-builder functions returning raw SQL. Consumed by the facade (sql.db.ts imports 5 of the 6) which wraps each in an executeQuery call.
  • queries.industrial.ts is the exception — it is not imported by the facade; it is imported directly by 3 spec files (contract.record.overview.e2e, contract.contacts.e2e, sql.queries.spec). So there are two access patterns in the layer (facade-mediated vs direct import) — see F10.
  • The merge finders in queries.party.ts (:619–746) are the most sophisticated: a shared MERGE_SAFE_PARTY_PREDICATE constant, normalized-name helper, and composable *Pairs fragments unioned by wrapMergePairs.

Change-radius: these strings reach the suite through the facade’s 228 call sites (TDD-015). A change to a shared fragment (if one existed) would be one edit; today a change to the CI filter is ~80 edits (F2).

Dimension 3 — Security Surface & Trust Boundaries

HIGH String parameters are interpolated into SQL, not bound — an injection surface on a credentialed live-DB connection

Evidence: queries.account.ts:159, 328–329, 357–358, 638–639 · queries.party.ts:279, 286, 421, 431 · queries.opportunity.ts:445

WHERE u.Code = '${utilityCode}'                       // queries.account.ts:159, 328, 357, 638
  AND lob.Name = '${lobName}'                          // queries.account.ts:329, 358, 639
WHERE p.PartyNumber = '${partyNumber}'                 // queries.party.ts:279 AND :286 (UNION halves)
WHERE en.Note = '${noteText}'                          // queries.party.ts:421
WHERE en.Note LIKE '% ${word} %'                       // queries.party.ts:431
WHERE et.Name = '${eventTypeName}'                      // queries.opportunity.ts:445

Why it matters: These are string-typed parameters interpolated directly into WHERE clauses on a connection that authenticates with a real service account against live INT/QA — and, by the same code path, PROD-adjacent environments. A value containing a single quote breaks the query; a crafted value injects SQL. Today every caller passes a test-controlled constant (a utility code like 'NIPSCO', a note like '#duplicate'), so there is no live exploit — which is why this is High, not Critical. But the layer provides no parameterization barrier at all for string inputs, and several of these builders are general-purpose (getProspectAccountWithNoHuByUtility(utilityCode, lobName), getPartiesWithNoteContainingWord(word)) — the moment one is fed a value derived from a fixture file, a CSV, or DB-sourced data, it becomes exploitable, and the en.Note LIKE '% ${word} %' case will also silently mis-match on any value containing % or '. The decisive point: the sibling Industrial config (TDD-016) binds the same class of parameter correctly with .input() — the layer already knows the right way and doesn’t use it here.

Call path / reach: Reached via the facade finders getOpportunityByUtilityCode, getProspectAccountWith(s)NoHuByUtility, getUlobByUtilityCode, getPartyEmailHistoryByPartyNumber, getPartyWithExactNote, getPartyWithNoteContainingWord, getOpportunityWithBrokerRemovedEvent — i.e. the account/party/opportunity test fixtures that pass utility/LOB/note/event values.

Severity rationale: High, with the security multiplier applied (credentialed, write-capable DB connection). Not Critical because current inputs are trusted constants and the connection is internal-network test infra — but it is the single finding in the whole layer with an external-risk shape, and the only correct long-term answer is binding.

Remediation sketch: Convert the builders that take string params to parameterized form — return { text, params } and have executeQuery apply request().input(...) per param (the pattern already in sql.industrial.config.ts:178/196/210). For the LIKE case, bind the value and escape %/_. This also fixes the latent quote-breakage, not just the injection.

Dimension 1 — Architecture, Coupling, Boundaries

MEDIUM queries.industrial.ts bypasses the facade — two competing access patterns in one layer

Evidence: sql.db.ts:32–36 imports only 5 query modules (no industrial); queries.industrial.ts consumed directly by contract.record.overview.e2e.spec.ts, contract.contacts.e2e.spec.ts, sql.queries.spec.ts

// sql.db.ts imports account, contract, enum, opportunity, party — NOT industrial:
import { queriesAccount }    from './queries/queries.account';
import { queriesContract }   from './queries/queries.contract';
import { queriesEnum }       from './queries/queries.enum';
import { queriesOpportunity} from './queries/queries.opportunity';
import { queriesParty }      from './queries/queries.party';
// queries.industrial.ts (14 builders) is imported DIRECTLY by specs instead.

Why it matters: Five of the six query modules flow through the sqlDB facade; the sixth, queries.industrial.ts (14 builders, 257 LOC), is imported straight into spec files. So the layer has two ways to reach a query — the reuse-first facade (which the /sql-data skill and the 109-finder manifest assume is the single front door) and a direct import that the manifest can’t see. A query reachable only by direct import is invisible to the reuse ladder, so it’s prone to being re-composed bespoke. (Note: these are the Industrial string builders, distinct from the Industrial pool finders in sql.industrial.config.ts reviewed in TDD-016 — the Industrial domain is split across two files and two access patterns.)

Severity rationale: Medium — coupling/consistency that undercuts the reuse model the rest of the tooling is built on.

Remediation sketch: Either route queries.industrial.ts through facade finders (consistent front door) or consciously document it as a direct-import module and reflect that in the manifest; don’t leave it ambiguous.

Dimension 5 — Tech Debt & Maintainability

MEDIUM The CI filter is copy-pasted ~80+ times and never extracted — and is already drifting

Evidence: e.g. queries.party.ts:5–6, 15–16, 36–37, 51–52, 79–80, 97–98, …; queries.opportunity.ts:11–12, 46–47, …; queries.account.ts:30–31, 94–95, 161–162, … vs the relaxed window at account:219, 254

WHERE p.RelationshipManagerId = 1
  AND p.CreateDate BETWEEN DATEADD(YEAR, -1, GETDATE()) AND DATEADD(WEEK, -2, GETDATE())   // repeated ~80+ times
// ...but two account queries quietly use a TWO-year window:
AND p.CreateDate BETWEEN DATEADD(YEAR, -2, GETDATE()) AND DATEADD(WEEK, -2, GETDATE())     // account:219, 254
// the file PROVES it can extract a predicate — it just didn't for this one:
const MERGE_SAFE_PARTY_PREDICATE = ` p.RelationshipManagerId = 1 AND p.IsMerged = 0 ... `;  // party:642–649

Why it matters: The CI filter is the single most-repeated fragment in the layer — the canonical convention from CLAUDE.md, duplicated verbatim across roughly 80+ query sites in three files. A change to the window (the documented convention) is ~80 edits, and drift is already present: two account HU queries use a 2-year window (with a comment explaining why), so “the CI filter” is silently not one thing. The kicker is that queries.party.ts already demonstrates the fix — MERGE_SAFE_PARTY_PREDICATE (:642) and NORMALIZED_PARTY_NAME (:652) are exactly the extract-a-shared-fragment pattern — it was simply never applied to the fragment that needed it most.

Severity rationale: Medium — a compounding maintainability tax and an active correctness-drift surface; the highest-leverage cleanup in the slice.

Remediation sketch: Export CI_PARTY_FILTER (and a parameterized ciPartyFilter({ years = 1 }) for the HU exceptions) from a shared fragments module, mirroring MERGE_SAFE_PARTY_PREDICATE; replace the ~80 literals. One window definition, one place to change it.

MEDIUM Terminal-state filtering uses two mechanisms, and the string variant drifts across files

Evidence: id-based at queries.opportunity.ts:128, 189, 394 + queries.party.ts:215, 604; string-based at queries.opportunity.ts:13, 27, 127, 160 vs queries.account.ts:258, 292, 427

AND cs.ImaOpportunityStatusId NOT IN (7, 8)             // the documented convention (7=Won, 8=Dead)
AND os.Name NOT IN ('Won', 'Dead', 'Closed')           // queries.opportunity.ts:13, 27, 127, 160 — string set A
AND os.Name NOT IN ('Won', 'Lost', 'Dead', 'Closed')   // queries.account.ts:258, 292, 427 — string set B (adds 'Lost')

Why it matters: One concept — “exclude terminal opportunities” — is expressed two ways: the id-based gate on ImaOpportunityCurrentStatuses (the convention, with 7=Won/8=Dead confirmed in the KB) and base OpportunityStatuses.Name string lists. Worse, the string lists disagree: account queries add 'Lost'; opportunity queries don’t — and the KB notes 8 is Dead, not Lost. So depending on which finder a test uses, the terminal-state definition differs, which means two finders that look interchangeable sample subtly different populations. This is exactly the kind of silent inconsistency that makes the reuse ladder (pick the “looser sufficient” finder) unsafe.

Severity rationale: Medium — correctness/consistency; can skew which opportunities a test sees without any visible symptom.

Remediation sketch: Standardize on the id-based ImaOpportunityStatusId NOT IN (7,8) gate (a shared fragment, like F2), and delete the string-name variants; if a base-status filter is genuinely needed, define one canonical set.

MEDIUM Hardcoded ID blocklists and magic agent IDs — brittle, INT-specific, no lookup

Evidence: queries.account.ts:183–185 (+ use at :215), 385 · queries.opportunity.ts:446 · queries.party.ts:600

// TODO: identify the Industrial-side prerequisite so this becomes a structural filter
//       rather than a hardcoded blocklist.                              // account.ts:183–184 — acknowledged debt
const HU_REJECTING_PARTY_IDS = '43377747, 32888822, 31468485, 31432617, 31494708, 31417310';  // :185
  AND pa.PartyId NOT IN (${HU_REJECTING_PARTY_IDS})                      // :215
  AND hu.RequestedByOrganizationAgentId <> 45576                          // account:385 / opp:446 — "system agent"
  AND t.OrganizationAgentId NOT IN (47894, 56199)                        // party:600 — RSA_LEADS, AUTOMATED_TESTER

Why it matters: Concrete INT row IDs are baked into query text: a six-party HU-rejection blocklist (with an explicit unresolved TODO at :183–184 noting it should be a structural filter), the “system agent” 45576, and the RSA_LEADS/AUTOMATED_TESTER agents 47894, 56199. If any of these rows is repurposed, deleted, or differs in QA, the query silently samples a different population — no error, just skewed or empty data. There’s no enum or lookup; the meaning lives only in comments. This is the kind of embedded-magic the overhaul should retire.

Severity rationale: Medium — brittle test data with a silent failure mode; INT-specific values that don’t travel to QA.

Remediation sketch: Resolve the agent IDs by name at runtime (an enum finder already exists for statuses/task types) and replace the HU blocklist with the structural Industrial-side gate its own TODO calls for; at minimum hoist all three into a named, commented constants module.

MEDIUM Magic numeric literals saturate the predicates with meaning that lives only in comments

Evidence: e.g. queries.account.ts:53–54 (status 7 / type 3), 444–445 · queries.industrial.ts:22, 44, 46 · queries.opportunity.ts:76, 145, 700 · queries.party.ts:14

WHERE ldc.LDCAccountStatusId = 7 AND ldc.LDCAccountTypeId = 3            // account:53–54
WHERE sl.ServiceStatusId IN (1, 6) ... AND sl.AccountTypeId IN (1,2,4)   // industrial:22, 44
  AND sl.UtilityLineOfBusinessId <> 79                                   // industrial:46
  AND osa.ImaSaleActivityId = 18 ... AND ot.OpportunityTransferReasonId = 20  // opp:76, 145
  AND e.EventTypeId = 101                                                // party:14 (note event)

Why it matters: Status ids, type ids, activity ids, event-type ids, transfer-reason ids, and supplemental flags are scattered as bare integers across ~2,700 lines. Most carry an inline comment, but the meaning is in prose, not in a typed constant, so the same id (7 = Won? or LDCAccountStatus? depends on the column) means different things in different files and can’t be checked. This is the maintainability cost the overhaul most directly addresses.

Severity rationale: Medium — pervasive; compounds with F4 (IDs) and makes the queries hard to read and safe-edit.

Remediation sketch: A typed enum/constants module (LdcAccountStatus.Active = 4, ImaOpportunityStatus.Won = 7, …) referenced in the builders; the repo’s schema-constants.md already enumerates most of these — generate from it.

LOW Hardcoded date literals that silently age

Evidence: queries.party.ts:300 (MergedDate >= '2026-04-01'), 433 (en.CreateDate > '2020-01-01')

WHERE MergedDate >= '2026-04-01' ... // party:300 getStuckMergedParties — fixed window, drifts from "recent"
AND en.CreateDate > '2020-01-01'      // party:431/433 — arbitrary floor

Why it matters: getStuckMergedParties pins a fixed start date ('2026-04-01') for “recent” merge attempts; as time passes it stops meaning “recent.” The '2020-01-01' note floor is an arbitrary literal. Neither is parameterized or relative (DATEADD), unlike the rest of the layer’s date handling.

Severity rationale: Low — narrow blast radius (diagnostics/note finders), but a literal that ages into wrongness.

Remediation sketch: Use relative windows (DATEADD) or a parameter, consistent with the CI filter’s approach.

LOW The CI filter is applied inconsistently — intentional omissions look the same as accidental ones

Evidence: RM-only (no date) at queries.party.ts:20–26, 131–141, 416–423; documented omission at queries.contract.ts:25–26, 43–45; intentional drop at queries.party.ts:638–640

Why it matters: Some finders apply RM=1 + the date window; some apply only RM=1; some neither. Several omissions are deliberate and documented (contracts live on pre-window parties — contract.ts:25–26; the merge predicate drops the window to surface old+new duplicate pairs — party.ts:638–640). Others (getPartiesWithCallOutcomes party:20–26, getPartiesWithExactNote party:416–423) omit the date window with no comment, so a reader can’t tell intentional from accidental. Once F2 extracts the filter, these become explicit opt-outs rather than silent divergences.

Severity rationale: Low — correctness ambiguity; mostly resolved as a side effect of F2.

Remediation sketch: After F2, make every non-standard query opt out explicitly (/* no date window: contracts predate it */) so the intent is on the page.

LOW SELECT * + ORDER BY <ordinal> in the enum/snapshot queries

Evidence: queries.enum.ts:4, 10, 16 · queries.party.ts:270, 310

SELECT TOP 50 * FROM Opportunity.ImaLineOfBusinessOpportunityTypes ORDER BY 1   // enum:4
SELECT * FROM Opportunity.OpportunityStatuses ORDER BY OpportunityStatusId       // enum:10
SELECT TOP 10 * FROM Party.MergeSnapshots ORDER BY 1                              // party:310

Why it matters: SELECT * couples the result shape to the table’s column order, while the facade casts the result to a narrow type ({ TaskTypeId, Name }) — so a schema column change drifts silently against the type (links to TDD-018). ORDER BY 1 (ordinal) is fragile if the select list changes. Low because these are reference-data reads, but they’re the easiest to tighten.

Severity rationale: Low — reference-data only; cosmetic + a small drift surface.

Remediation sketch: Select explicit columns and order by name, matching the rest of the layer.

LOW Sanitized-but-interpolated IN-lists, and a few deterministic finders that skip NEWID()

Evidence: IN-lists at queries.contract.ts:79–91, queries.party.ts:447–448, queries.opportunity.ts:784; deterministic order at queries.account.ts:436–453, 456–465, queries.opportunity.ts:133–147

WHERE c.ImaContractId IN (${idList || 'NULL'})       // contract:89 — idList is parseInt+filter+join (safe)
WHERE VictimOpportunityId IN (${victimOpportunityIds.join(',')})  // party:447 — number[] (safe-ish)
ORDER BY ldc.AccountId DESC   // account:452 — deterministic; comment says pick() randomizes in-memory

Why it matters: Two minor convention deviations. (1) IN-lists are interpolated rather than bound — they’re sanitized (parseInt+Number.isFinite for contract; number[] elsewhere), so no live injection, but it’s the same interpolation habit as F1 on the numeric side. (2) A handful of finders order deterministically (ORDER BY AccountId DESC) instead of NEWID() and rely on the facade’s in-memory pick() to randomize — documented in the comments, but a divergence from the NEWID() sampling convention that a reader must know to expect.

Severity rationale: Low — safe and partly documented; consistency with the sampling/parameterization conventions.

Remediation sketch: Bind IN-lists alongside F1; either add ORDER BY NEWID() to the deterministic finders or document the in-memory-pick contract in one place.

Dimension 2 — Test Coverage, Quality & Gaps

Partial mitigation — the SQL sanity spec exercises these strings

src/tests/sql/sql.queries.spec.ts runs the queries/* builders against the live schema and asserts they return rows, which is the safety net that keeps column drift (the repo’s documented 32+ findings) from going fully silent — and the reason this report does not raise a column-drift finding to High. The gap: that net does not cover the inline SQL in the facade (TDD-015 F5) or the Industrial pool finders (TDD-016 F7), and it asserts “rows come back,” not “the right rows” — a finder whose predicate is subtly wrong (e.g. the F3 status drift) still passes. No no-assertion or over-mocked tests observed in scope.

Dimension 4 — Error Handling & Failure Modes

No findings in this slice — error handling lives in the facade

The query modules are pure string builders with no control flow beyond ternaries (getPartiesWithMultipleAccounts party:124) and the sanitizing .map/.filter/.join on IN-lists. There is no try/catch, retry, or state mutation here to assess — empty-result and connection handling is the facade’s concern (TDD-015 F1/F4). One observation: getCiContractsByIds emits IN (${idList || 'NULL'}) (contract:89) so an empty id array yields IN (NULL) (zero rows) rather than a SQL syntax error — a deliberate, correct guard worth keeping.

Dimension 6 — Dependency Health & Version Risk

No findings — these modules import nothing

The query modules have zero imports (pure template-literal functions). No dependency, transitive, or version surface in this slice.

Dimension 7 — Performance Hotspots

No material findings — bounded queries; two known heavy-table cautions already handled

Every query is TOP-bounded (TOP 1…TOP 1000), and the documented heavy-table hazards are respected: nothing in scope does an unbounded ORDER BY on the 176K-row ULOB contract-term table (the KB timeout caution), and the merge finders deliberately drop a costly NOT-EXISTS that previously pushed the email finder past the mssql timeout (documented at queries.party.ts:631–637). The complex multi-JOIN/CROSS-APPLY finders (getPartyWithActiveRepTaskAndOtherOpp opp:326–365, the merge *Pairs unions) are heavier but bounded and run once per fixture, not in a loop. No N+1 in the SQL layer (N+1 risk, if any, is in how the facade sequences calls — TDD-015). No perf finding warranted.
