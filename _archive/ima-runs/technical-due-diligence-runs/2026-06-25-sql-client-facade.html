Verdict — the three things to fix first in this slice

This is the highest-fan-in code in the SQL layer: sqlDB is imported by ~47 files with 228 sqlDB.* call sites, and every one routes through two ~private helpers, getClient() and pick(). The design (decoupled query strings + a thin executor + random-row pick) is sound. Three things undercut it:

  1. [High] Every finder call creates a new connection pool that is never closed. getClient() news up a fresh SQLUtils (→ a new mssql ConnectionPool) per call, and executeQuery opens it but has no close(). The Industrial path closes its pool in a finally; the CRM path does not. Across a sharded run with 228 call sites, idle pools accumulate against shared INT/QA SQL Server — a silent resource leak that surfaces as login/“pool is draining” flake.
  2. [Medium] The empty-result contract is inconsistent across ~100 finders. Some throw a clear message, some return records[0] (an undefined leak), some default safely, and dozens call pick() on a possibly-empty array (an opaque faker error). A caller can’t predict how “no data” fails.
  3. [Medium] Raw SQL is embedded directly in the facade, bypassing the queries/ layer and its verification. Four finders inline SQL string literals in sql.db.ts instead of queries/*.ts — so they dodge the SQL sanity spec that proves column names, the one safety net protecting against the repo’s documented column-drift.

Net: keep the architecture, fix the connection lifecycle, standardize the empty-result contract, and pull the inline SQL back into the queries layer. None is a rewrite.

Phase 1 — Surface Map

Public surface & change-radius

  • SQLUtils (sql.client.ts:8–35) — holds one ConnectionPool, caches poolConnectPromise, exposes connect() and executeQuery<T>(query). No close()/dispose().
  • getClient() (sql.db.ts:52–55) — await sqlConfig() then new SQLUtils(config). Called at the top of nearly every finder → a fresh client per call.
  • pick() (sql.db.ts:57) — faker.helpers.arrayElement(records); in-memory random row select.
  • sqlDB (sql.db.ts:839–949) — the exported object aggregating 100+ finder methods (the generated manifest lists 109). The single import surface for the suite.

Change-radius: highest fan-in in the SQL layer. The layer is imported by ~47 files (33 specs + 5 data factories + fixtures/api), and sqlDB. appears 228 times across 37 files (e.g. opportunities.api.spec.ts ×44, party.merge.e2e.spec.ts ×13, calendar.e2e.spec.ts ×9). Every finder funnels through getClient/executeQuery/pick, so a change to any of those three radiates to the whole suite.

Dimension 1 — Architecture, Coupling, Boundaries

MEDIUM Raw SQL embedded in the facade bypasses the queries/ layer and its sanity-spec verification

Evidence: src/utils/sql/sql.db.ts:495–518, 827–835

const getTaskById = async (taskId) => {
  const sql = await getClient();
  const records = await sql.executeQuery(
    `SELECT TaskId, TaskStatusId, OrganizationAgentId FROM Party.Tasks WHERE TaskId = ${taskId}`);  // inline SQL in the facade
  return records[0];
};
const getOpportunityImaStatus = async (oppId) => { /* ...inline 5-line JOIN, sql.db.ts:503–508... */ };
const getOpportunityIsFlagged  = async (oppId) => { /* ...inline query, sql.db.ts:515–517...      */ };
const getBrokerName            = async ()      => { /* ...inline 9-line query, sql.db.ts:827–835... */ };

Why it matters: The layer’s contract is that query text lives in queries/queries.*.ts — where it is exercised by the SQL sanity spec (src/tests/sql/sql.queries.spec.ts) that proves the column names resolve. Four finders break that contract by inlining SQL in the facade. Those strings are not in the queries manifest and are not covered by the sanity spec, so they’re exactly the surface where the repo’s documented column-name drift (schema-constants.md, 32+ findings) lands silently — a renamed column returns 0 rows, not an error.

Call path / reach: getTaskById/getOpportunityImaStatus/getOpportunityIsFlagged back DB-verification asserts in opportunity + task specs; getBrokerName backs broker tests. A column rename breaks them with a confusing empty result, not a typed failure.

Severity rationale: Medium — consistency + a real gap in the verification net; not High because the affected finders are few and read-only.

Remediation sketch: Move the four inline queries into queries/queries.{opportunity,party}.ts so they ride the sanity spec, and parameterize the numeric IDs while moving them.

MEDIUM God facade — 100+ finders in a single 949-line module, ~47-file fan-in

Evidence: src/utils/sql/sql.db.ts:59–837 (finder defs), 839–949 (the sqlDB export object)

Why it matters: sql.db.ts is one module that mixes party, opportunity, account, contract, and Industrial-bridging concerns and exposes them as a flat 109-entry object. It is the single import for ~47 files / 228 call sites, so (a) it’s too large to reason about as a unit, (b) any churn in the shared getClient/pick helpers or the export shape radiates suite-wide, and (c) domain boundaries that the rest of the repo keeps separate (feature-first folders) collapse here into one file. This is the structural reason the overhaul feels risky — everything imports everything through one door.

Call path / reach: Whole suite. Splitting it is the highest-leverage architectural move but also the one with the widest blast radius, so it must be done behind a stable sqlDB re-export.

Severity rationale: Medium — coupling/maintainability tax that compounds; not High because the indirection (one object) currently insulates callers from internal layout.

Remediation sketch: Split into per-domain finder modules (finders/party.ts, …) that each import getClient/pick from a tiny core; keep sqlDB as a barrel that re-exports them, so the 228 call sites don’t move.

Dimension 2 — Test Coverage, Quality & Gaps

MEDIUM The facade’s own logic (pool lifecycle, empty-result handling, pick) has no direct test

Evidence: src/tests/sql/sql.queries.spec.ts (exercises query rows, not facade logic); SQLUtils at sql.client.ts:30–34

Why it matters: The SQL sanity spec runs the query strings and checks they return rows — valuable, and it’s why TDD-017 ranks column drift lower than it otherwise would. But nothing tests the executor itself: that connect() is reused, that a finder throws a useful error on 0 rows vs leaks undefined (F4), or that pools are released (F1). The single most-reached primitive in the layer is verified only transitively (“the specs pass, so it must work”) — and F1/F4 are precisely the failure modes that transitive coverage hides.

Severity rationale: Medium — a foundational primitive with 228 call sites and zero direct coverage; the gap is what lets F1/F4 stay invisible.

Remediation sketch: A small unit spec for SQLUtils (mocked pool): connect-is-memoized, executeQuery returns recordset ?? [], and a finder-contract test asserting the agreed empty-result behavior once F4 standardizes it.

Dimension 3 — Security Surface & Trust Boundaries

LOW Inline numeric interpolation in the facade — low risk today, wrong barrier

Evidence: src/utils/sql/sql.db.ts:497, 507, 516

`...WHERE TaskId = ${taskId}`            // number
`...WHERE ics.OpportunityId = ${oppId}`  // number

Why it matters: These interpolate number-typed params, so the injection risk is effectively nil at runtime. But they establish string interpolation as the house style at the executor layer, and the real string-typed injection surface lives one layer down in queries/* (see TDD-017 F1). Flagged here so the facade fix (F5, move inline SQL to queries) also parameterizes rather than re-interpolates.

Severity rationale: Low — numeric inputs, no exploitable path; preventative.

Remediation sketch: When relocating per F5, use request().input('id', id) as the model the rest of the layer should converge on.

No other security findings in this slice: SQLUtils takes no untrusted input, holds no secrets (config is fetched in TDD-016’s scope), and exposes no network/auth surface beyond the DB connection.

Dimension 4 — Error Handling & Failure Modes

HIGH Per-call connection pool is created and never closed — the CRM path leaks while the Industrial path doesn’t

Evidence: src/utils/sql/sql.db.ts:52–55 + src/utils/sql/sql.client.ts:16–34 (no close); contrast sql.industrial.config.ts:39–55

const getClient = async () => {
  const config = await sqlConfig();
  return new SQLUtils(config);            // NEW pool object on EVERY finder call (sql.db.ts:52–55)
};
// SQLUtils — opens the pool, never closes it:
async connect() { if (!this.poolConnectPromise) this.poolConnectPromise = this.pool.connect()...; return ...; }
async executeQuery(query) {
  const pool = await this.connect();
  const results = await pool.request().query(query);
  return results.recordset ?? [];          // no pool.close(), no finally (sql.client.ts:30–34)
}
// the Industrial side gets this right — it closes in a finally:
const pool = await getIndustrialPool(); try { ... } finally { await pool.close(); }  // sql.industrial.config.ts:39–55

Why it matters: Every sqlDB.* call constructs a brand-new SQLUtils, i.e. a new mssql ConnectionPool (default max 10 connections), opens it, and abandons it — there is no close() anywhere in the CRM path. The pool object stays alive holding its sockets until process exit/GC. With 228 call sites and a spec like party.merge.e2e.spec.ts opening ~13 pools, a single sharded run (the suite runs ~6 worker processes) accumulates many open pools against the shared INT/QA SQL Server. That pressure surfaces as intermittent login failures / “pool is draining” errors that read as flaky tests, not as a leak. The asymmetry is the tell: the Industrial finders close their pool in a finally; the far-more-trafficked CRM facade never does.

Call path / reach: Suite-wide — every CRM finder (the large majority of the 228 call sites). Bounded only by process lifetime, which is why it manifests as load-dependent flake rather than an outright failure.

Severity rationale: High — silent, suite-wide, and it stresses shared database infrastructure the whole team depends on. Not Critical: it degrades test reliability and shared-infra headroom, not production data/money/auth.

Remediation sketch: Make the CRM client a memoized singleton per worker (one pool, reused) — the cleanest fix and it also removes the per-call connect cost (F8). Alternatively give SQLUtils a close() and wrap finder bodies in try/finally as the Industrial path does. A singleton is preferable; mssql pools are designed to be long-lived and shared.

MEDIUM Inconsistent empty-result contract — throw vs undefined-leak vs safe-default vs opaque faker error

Evidence: src/utils/sql/sql.db.ts:143, 498, 688, 753 (four different behaviors)

if (records.length === 0) throw new Error('No merge-safe duplicate party pair found ...'); // :143 clear throw
return records[0];                                   // :498 getTaskById — UNDEFINED leak, no guard
if (records.length === 0) throw new Error(`No ${lobName} ULOB ...`); return records[0];  // :687–688 guarded
return records[0]?.ContactCount ?? 0;                // :753 safe default
return pick(records);                                // dozens of finders — pick([]) throws an opaque faker error

Why it matters: Four different no-data behaviors coexist across ~100 finders with no stated rule. getTaskById (:498) returns records[0] unguarded, so a missing task yields undefined that explodes as “cannot read property of undefined” somewhere downstream, far from the cause. The many bare return pick(records) finders throw a faker-internal error with no domain context when the DB has no matching row — which, given the documented INT data-window gaps, is a routine occurrence. A consumer cannot predict whether “no data” means a clear message, a silent undefined, a default, or an opaque throw.

Call path / reach: Every finder that returns a single row; most acute on the unguarded records[0] path and the bare-pick path.

Severity rationale: Medium — debuggability/maintainability tax across the layer; the undefined leak can produce errors arbitrarily far from the finder.

Remediation sketch: One helper — pickOrThrow(records, contextMsg) — used by all single-row finders; reserve a separate maybeOne() for the deliberately-nullable cases. Make “0 rows” always throw a domain-named error (it almost always means a data-setup miss).

Dimension 5 — Tech Debt & Maintainability

MEDIUM getClient() rebuilds config (a Key Vault fetch) on every finder call — no memoization

Evidence: src/utils/sql/sql.db.ts:52–54 → src/utils/sql/sql.config.ts:6–7

const getClient = async () => {
  const config = await sqlConfig();   // → getServiceAccount('service-account') — a Key Vault round-trip, EVERY call
  return new SQLUtils(config);
};

Why it matters: Beyond the pool leak (F1), getClient calls sqlConfig() on every invocation, and sqlConfig resolves credentials via getServiceAccount('service-account') (the azure-secrets-helper / Key Vault path). So each of the 228 call sites pays a secret-resolution cost plus a fresh connect. Even if the helper caches internally, the layer is structured to re-fetch-and-reconnect per finder — the opposite of how a pool is meant to be used.

Severity rationale: Medium — compounding per-call cost across the suite; folds into the same singleton fix as F1.

Remediation sketch: Resolve config once and build one shared client per worker (the F1 singleton); the secret fetch and the connect both collapse to once-per-process.

LOW Result-type definitions split between the facade and the types module

Evidence: src/utils/sql/sql.db.ts:198–205 (BulkTransferBatchItem), 666–674 (ValidUlob); vs src/types/sql/*

Why it matters: Most finder result types live in src/types/sql/, but BulkTransferBatchItem and ValidUlob are declared inline in the facade, and many finders return anonymous inline shapes (Promise<{ OpportunityId: number; PartyId: number }>). The “types layer” has no single home — covered in depth in TDD-018; noted here as the facade’s contribution to that scatter.

Severity rationale: Low — cosmetic/organizational; see TDD-018 for the consolidated finding.

Remediation sketch: Move BulkTransferBatchItem/ValidUlob into src/types/sql/ as part of the TDD-018 consolidation.

Dimension 6 — Dependency Health & Version Risk

LOW CommonJS-in-ESM require('mssql') shim, duplicated across the layer

Evidence: src/utils/sql/sql.client.ts:3–6 (also duplicated at sql.industrial.config.ts:7–8)

const require = createRequire(import.meta.url);
const sql = require('mssql') as typeof mssqlTypes;   // CJS shim; sql.config.ts uses `import * as mssql` instead

Why it matters: mssql is loaded three different ways across the layer — the createRequire shim here and in sql.industrial.config.ts, but a plain import * as mssql in sql.config.ts. The shim is a workaround for mssql’s CJS packaging under ESM; duplicating it (rather than a single internal module that exports the loaded sql) is a small drift surface. The dependency itself is healthy and pinned via package.json.

Severity rationale: Low — works today; inconsistency only.

Remediation sketch: One internal mssql.ts that does the createRequire once and exports the typed client; everything imports from it.

Dimension 7 — Performance Hotspots

No standalone findings — perf cost is the F1/F8 per-call pool + secret fetch

Traced the multi-query finders (getRandomOpportunityFromMultiOppAccountParty sql.db.ts:248–254; getInactiveChoiceIndustrialAccountWithOpportunity :449–458): they correctly reuse a single getClient() across 2–3 sequential queries — no per-query client churn, no N+1 inside a loop. The only material performance cost in this slice is structural: a new pool + a config/secret resolution on every finder call (F1, F8), addressed by the singleton fix. No tight-loop allocation, no unbounded fetch (every query is TOP-bounded — see TDD-017).
