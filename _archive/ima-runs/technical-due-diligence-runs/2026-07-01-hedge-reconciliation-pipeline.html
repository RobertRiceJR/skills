Verdict — the three findings that most move go / no-go

This is a one-off dev backfill, not a pipeline, and it writes straight into the IMA.Hedges system-of-record. As a reference artifact it is genuinely useful to the hub; as code to adopt or trust it is not — it has two data-loss/duplication defects and bypasses the production hedge path the hub already documents.

  1. Non-transactional delete-then-insert can wipe live hedge rows with no rollback. The DELETE commits in its own connection before the INSERT connection even opens — any interruption between them leaves the SoR emptied of these hedges and never refilled. → Dim 4 · Critical
  2. "Idempotent, safe to rerun" is false — confirmed live. The only unique index on IMA.Hedges is on HedgeKey, which the script sets to NEWID() (always unique); there is no constraint on the dedup tuple the README claims, and the code never checks. A rerun that skips the delete duplicates financial rows. → Dim 4 · Critical
  3. Relevance = reference-only, quarantined. It bypasses the canonical deal sheet → scrape → Pending.Hedges → IMA.Hedges (HedgeKey handshake, CONTRACT-954) path — so it must never be encoded as "how hedges reach IMA360." But it cheaply confirms the hub's open IMA.Hedges column-shape TBD and documents the Supply-side source + the Supply≠Industrial ContractId gotcha. Harvest the facts; discard the mechanism. → Relevance section

Dimension 1 — Architecture, Coupling, Boundaries

HIGH Writes directly to the IMA.Hedges SoR, bypassing the canonical scrape path; HedgeKey fabricated with NEWID()

Evidence: hedge_reconciliation.py:462-466

INSERT INTO IMA.Hedges (
    ContractTermId, HedgeTypeId, StartDate, EndDate,
    FixedRate, Value, CreateBy, CreateDate, ModifyBy, ModifyDate,
    HedgeKey, Margin, CommodityCost, AdditionDate
) VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?, NEWID(), ?, ?, ?)

Why it matters: The hub's own model node documents the production path as deal-sheet Hedges field → scrape → Pending.Hedges (CI-7513) → on save → IMA.Hedges with a populated HedgeKey (CONTRACT-954). This script skips Pending.Hedges and the HedgeKey handshake entirely, minting a random HedgeKey. Rows land in the SoR with no lineage back to any deal sheet, scrape record, or the RabbitMQ→Snowflake position feed that keys on HedgeKey. Downstream joins on HedgeKey find orphans.

Call path / reach: main:551 → insert_into_ima:447. Every inserted row is affected.

Severity rationale: High (money-path SoR + integration lineage) — not bumped to Critical because it is an intentional migration shortcut, not a claimed production path.

Remediation sketch: Treat strictly as a bounded historical backfill; if hedges must be created going forward, route through the scrape / POST api/Hedges path so the HedgeKey handshake and lineage hold.

Dimension 2 — Test Coverage & Gaps

HIGH Zero automated tests over subtle money-path logic

Evidence: whole folder — find returns only hedge_reconciliation.py + README.md; no test file, no assertion anywhere.

Why it matters: Four error-prone routines each feed a NOT NULL money column in the SoR and none is exercised by a test: the calendar-month date normalization (calculate_calendar_months:316), the ContractId regex (extract_industrial_contract_id:33), the fuzzy-match threshold logic (fuzzy_match_unmatched:139), and the FixedRate/Margin/CommodityCost math (calculate_hedge_fields:356). Each has edge cases (proved in Dim 4) that produce wrong-but-valid rows. The only safety net is manual inspection of DRY_RUN log output.

Severity rationale: The rubric rates "no test coverage on a money path" as Critical; held at High because realized blast radius is captured by the Dim 4 findings and this is a manual one-shot, not a deployed service.

Remediation sketch: Pure-function unit tests on the four feeders (esp. the date-normalization boundary cases) before any non-dry-run against a shared DB.

Dimension 3 — Security Surface & Trust Boundaries

LOW SQL IN-clause built by f-string interpolation (defused by an int() cast)

Evidence: hedge_reconciliation.py:214-223

term_ids_str = ','.join(str(int(tid)) for tid in contract_term_ids)
query = f"""
SELECT TermId, FromDate, ToDate, Margin
FROM IMA.ContractDetails
WHERE TermId IN ({term_ids_str})
  AND Margin IS NOT NULL ..."""

Why it matters: String-interpolated SQL is the classic injection smell, but every element is int()-cast first, so a non-numeric value raises before it reaches the query — not injectable as written. The risk is latent: if the cast is ever relaxed (e.g. to accept GUID term keys) it becomes a live injection into a query against the SoR DB. Every other query is a static string or uses bound ? params. No secrets in-file (Windows Trusted_Connection).

Remediation sketch: Parameterize the IN-list (? placeholders) so the safety doesn't depend on the upstream cast.

Otherwise no material findings — traced all six queries: five are static or fully parameterized; no hardcoded credentials; auth is Windows integrated. One trust note: writes stamp CreateBy='CI-Dev-LOAD' regardless of the real invoking Windows identity, so there is no actor audit trail on SoR mutations (folded into Dim 5).

Dimension 4 — Error Handling & Failure Modes

CRITICAL Non-transactional DELETE-then-INSERT → data-loss window on the system of record

Evidence: hedge_reconciliation.py:416-439 (delete) then a separate connection at :455-488 (insert)

# delete_old_hedges — its OWN connection + commit
with pyodbc.connect(INDUSTRIAL_CS) as conn:
    ...
    cursor.execute(delete_query)   # DELETE ... WHERE CreateBy='CI-Dev-LOAD'
    conn.commit()                  # :439  <-- committed HERE, before insert exists

# insert_into_ima — a DIFFERENT connection, opened afterward
with pyodbc.connect(INDUSTRIAL_CS) as conn:      # :455
    for idx, row in df.iterrows():
        cursor.execute(insert_query, (...))
    if not dry_run:
        conn.commit()              # :488  <-- only commit for the inserts

Why it matters: In a real run (DRY_RUN=False) the DELETE of every CI-Dev-LOAD hedge is committed in its own transaction at line 439, and only then does a second connection begin inserting. Any failure in between — an insert-loop exception (bad cast, constraint), a dropped connection, a killed process — leaves IMA.Hedges with all these hedges deleted and never restored. There is no surrounding transaction, no savepoint, no restore step. The main() try/except (:568) only logs and re-raises. On a financial system of record with no rollback, this is the maximum blast radius.

Call path / reach: main:550 (delete) → main:551 (insert), sequential, two transactions.

Severity rationale: Critical (data loss on a money-path SoR, no rollback).

Remediation sketch: Wrap delete + insert in a single transaction on one connection (autocommit=False, one commit() at the very end, rollback() on any exception). Ideally snapshot the deleted rows first.

CRITICAL "Idempotent / safe to rerun" is false — no enforcing constraint (confirmed live) and no code check

Evidence: README.md:117-122 & :404 claim; hedge_reconciliation.py:447-485 (insert has no existence check); live sys.indexes on IMA.Hedges

README §Idempotency: "safe to rerun ... Checks for exact matches:
   ContractTermId + StartDate + EndDate + FixedRate ... Skips existing hedges (no duplicates)"

-- Live on mssql-industrial-int (2026-07-01) — the ONLY unique index:
UQ_Hedges_HedgeKey  (is_unique=1)  ->  HedgeKey
-- No unique index on (ContractTermId, StartDate, EndDate, FixedRate).
-- And the script sets HedgeKey = NEWID() (line 466) => every row is unique by construction.

Why it matters: The advertised safety net does not exist at either layer. (a) insert_into_ima contains no "does this hedge already exist?" query — it unconditionally inserts every prepared row. (b) The DB cannot catch a duplicate either: the sole unique constraint is on HedgeKey, which is NEWID(), so duplicates on the business tuple are physically insertable. The drop_duplicates at :403 only dedups within a single run's DataFrame. The only thing preventing accumulation across runs is the fragile delete-first step — which, per the finding above, is a separate transaction. Run the insert twice without the delete (or after the delete fails) and you silently double the hedge coverage for every contract term.

Call path / reach: Any re-execution; compounds directly with the delete/insert split above.

Severity rationale: High → Critical (money path — duplicated hedge % / rates inflate contract coverage and downstream margin/unwind math).

Remediation sketch: Either add a real pre-insert existence check on the business tuple, or add a unique index to enforce it and use MERGE/upsert — and correct the README, which describes behavior the code does not implement.

HIGH Fuzzy-match fallback can attach a priced hedge to the wrong contract

Evidence: hedge_reconciliation.py:160-174

if len(matching_utility_contracts) == 0:
    matching_utility_contracts = ima_contracts          # <-- search ALL contracts
    min_score_threshold = MIN_MATCH_SCORE + 0.15         # 0.85
...
for _, ima_row in matching_utility_contracts.iterrows():
    score = fuzzy_match(customer, ima_row['Name'])
    if score > best_score:
        best_score = score
        best_match = ima_row['Id'] if score >= min_score_threshold else None

Why it matters: When the utility filter finds nothing, the code name-matches against every IMA contract at a 0.85 SequenceMatcher ratio and takes the best. Two similarly named customers ("City of Fostoria" vs "City of Fostonia") can exceed 0.85. A false match writes a real FixedRate + Margin hedge onto the wrong customer's contract term in the SoR — a financial mis-attribution with no confidence column, no review gate, and (per Dim 2) no test. The README notes only 299 of 559 hedges match directly; ~260 ride this fuzzy path, so the exposure is a large minority of all rows.

Severity rationale: High (money-path correctness, silent, unreviewed). Not Critical only because it corrupts individual rows, not the whole table.

Remediation sketch: Persist the match score + method per row; route sub-perfect matches to a manual review file instead of auto-writing; never fall back to the all-contracts search without human sign-off.

HIGH Calendar-month normalization can invert the date range (StartDate > EndDate)

Evidence: hedge_reconciliation.py:334-347

end_date_normalized = to_date + relativedelta(day=31)       # last day of the END month
if from_date.day == 1:
    start_date_normalized = from_date
else:
    start_date_normalized = from_date + relativedelta(months=1, day=1)  # 1st of NEXT month

Why it matters: A hedge that both starts mid-month and ends in that same month — e.g. Apr 12 → Apr 20 — normalizes to StartDate = May 1, EndDate = Apr 30: a negative-duration hedge. Nothing guards start > end; both columns are NOT NULL datetime2 so the DB accepts it. Such a row silently corrupts any month-coverage or weighted-average-rate summation (the hub's "December total" sanity math), and feeds unwind calculations. Short intra-month strips are exactly what a mid-term hedge layer looks like.

Severity rationale: High (silent invalid financial data, no guard, no test).

Remediation sketch: After normalization, assert StartDate <= EndDate; decide the business rule for sub-month strips (drop, or clamp to the covered month) explicitly rather than emitting an inverted range.

HIGH Fabricated "15% of fixed rate" margin persisted to a NOT NULL money column, unflagged

Evidence: hedge_reconciliation.py:365-367

df['Margin'] = df['ActualMargin'].fillna(df['FixedRate'] * 0.15).round(8)
df['CommodityCost'] = (df['FixedRate'] - df['Margin']).round(8)

Why it matters: When no overlapping ContractDetails margin is found, the script invents one — 15% of the fixed rate — and writes it (and the derived CommodityCost) into the SoR's NOT NULL Margin/CommodityCost columns (both decimal(12,8), confirmed live). Nothing distinguishes a fabricated margin from a real one downstream. The README frames the script as loading "actual contract margins ... instead of assuming a flat 15% cost" (§4) — yet the fallback does exactly the thing the README says it avoids, for every unmatched row.

Severity rationale: High (invented financial value on a money path, indistinguishable downstream).

Remediation sketch: Do not fabricate; route margin-less hedges to review, or add an explicit "estimated margin" flag column so downstream consumers can exclude them.

MEDIUM "First overlapping ContractDetails margin wins" — arbitrary pick among several

Evidence: hedge_reconciliation.py:296-298 (and the duplicate pass at :541-542)

if len(matching_details) > 0:
    merged_row['ActualMargin'] = matching_details.iloc[0]['Margin']   # first by FromDate

Why it matters: A hedge date range can overlap multiple ContractDetails rows with different margins over the strip. The code takes iloc[0] (earliest FromDate) and ignores the rest — no weighting, no "which detail actually applies to this strip" logic. The persisted margin can be the wrong one for part of the hedge window.

Remediation sketch: Define the intended selection (date-weighted average, or the detail covering the hedge midpoint) and apply it deterministically.

Dimension 6 — Dependency Health & Version Risk

MEDIUM No dependency manifest / pinning; deprecated pandas SQL path; ODBC Driver 17 hardcoded

Evidence: folder has no requirements.txt/pyproject.toml; hedge_reconciliation.py:8-13, :25-26, :90

import pyodbc, pandas as pd, from dateutil.relativedelta import relativedelta
df = pd.read_sql(query, conn)   # :90 — pyodbc connection, not a SQLAlchemy engine
"Driver={ODBC Driver 17 for SQL Server}; ..."   # :25-26

Why it matters: Three compounding reproducibility risks: (1) no pinned dependency set — the next runner resolves whatever pandas/pyodbc/dateutil is on the box, and the date-normalization + money math are version-sensitive; (2) pd.read_sql against a raw DBAPI connection is deprecated (pandas steers to SQLAlchemy engines) and emits warnings today, breakage later; (3) ODBC Driver 17 is hardcoded — Driver 18 (the current default install) flips Encrypt to yes and would reject this connection string without TrustServerCertificate. Any of the three can silently change behavior or fail the connection on a rebuilt machine.

Remediation sketch: Add a pinned requirements.txt; move to a SQLAlchemy engine for read_sql; make the ODBC driver version configurable / detect it.

Dimension 7 — Performance Hotspots

MEDIUM Full-table loads + O(hedges × terms) Python-loop join; mislabeled as a recurring "pipeline"

Evidence: hedge_reconciliation.py:185-203 (load all terms), :278-304 (nested per-row filter); README.md:94-96

ima_terms = load_ima_contract_terms()      # README: 197,074 terms, all into pandas
for idx, row in result.iterrows():          # per matched hedge...
    matching_terms = ima_terms[             # ...full-DataFrame boolean scan of 197k rows
        (ima_terms['ContractId'] == row['Id']) &
        (ima_terms['StartDate'] <= row['ToDate']) &
        (ima_terms['EndDate'] >= row['FromDate'])]

Why it matters: It pulls all ~38,771 contracts and ~197,074 contract terms into memory, then does the date-overlap join as a Python iterrows loop that re-scans the full 197k-row terms frame for every hedge — an O(hedges × terms) pattern the DB could do with an indexed range join. Tolerable for a one-shot backfill of ~600 hedges, but the README brands this a "pipeline" (implying recurring) and the algorithm would not scale to that use. A redundant second margin pass (:530-542) re-does work match_to_ima already computed at :290-302.

Remediation sketch: Push the date-overlap join into SQL (or a pandas merge keyed on ContractId first), and drop the duplicate margin pass. If it is truly one-shot, rename/document it as a backfill, not a pipeline.

Relevance to the deal-sheet-debug Skill Hub
