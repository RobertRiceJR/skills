Recommendation

Adopt a per-skill, per-contributor JSON-Lines usage log — .claude/skills/<skill>/runs.<name>.jsonl, one line per run, schema {date, by, ticket, quality, well≤15w, poorly≤15w, improve≤30w, model, duration, run?} — as the run-history source of truth, and generate the cross-skill view with _shared/scripts/skill-runs-rollup.mjs. The per-contributor filename is the load-bearing detail: it is exactly how the central ledger avoided merge conflicts after its 2026-06-18 split, and it is your hard constraint here.

Surfacing: the roll-up is the first item in a weekly/monthly cadence package carried by /dsu (a reference, not a new skill) — it fires when DSU runs on a new week, and the framework grows to monthly knowledge-sync / skills-drift later. Strong in-repo precedent: /comms-coach and /coverage-gap-scan already keep a per-skill runs.jsonl.

The central ledger is the one that hurts: verbose to write, hard to query "how did /three-amigos runs go," and a shared-file merge surface. The new log targets exactly that — and pulls duration/model from the session-stats layer it sits beside.

The design

file + schema per-skill, per-contributor JSONL — 9 fields

  • File: .claude/skills/<skill>/runs.<name>.jsonl — append-only, one JSON object per line, skill implied by the folder. <name> = lowercase first word of git config user.name.
  • Schema: date · by · ticket · quality (✅/⚠️/❌) · well (≤15w) · poorly (≤15w) · improve (≤30w) · model · duration · run? — your four prompts, plus the ledger's ticket+quality, plus model (the session model) and duration (best-effort from session-stats duration_minutes).
{"date":"2026-06-24","by":"Robert","ticket":"TDD-011","quality":"✅",
 "well":"Collision-safe census surfaced zero ghost skills and the true 62-skill count.",
 "poorly":"Glob and find undercounted by 3 (uppercase SKILL.md) until Node readdir caught it.",
 "improve":"Wire the census into /sync-skills as a standing drift gate; also flag SKILL.md casing.",
 "model":"claude-opus-4-8","duration":"~15m","run":"reference/runs/2026-06-24-skills-ecosystem-references.html"}

merge-safe why per-contributor files (your constraint)

  • Different users → different files → cross-user edits can never conflict on the same line. This is the exact fix the central ledger got on 2026-06-18 (split into -robert/-jess).
  • Append-only JSONL is the least-conflict-prone shape even for the rarer same-user concurrent case (new lines, not rewrites).
  • Demonstrated: a second contributor file (runs.jess.jsonl) aggregates alongside runs.robert.jsonl with zero overlap.

roll-up the cross-skill view is generated, not maintained

_shared/scripts/skill-runs-rollup.mjs walks every runs.*.jsonl, infers the skill from the folder, sorts newest-first, and prints the chronological table the ledger gave by hand (--json for machine output; --skill X to filter). Generated output → no merge surface. Pilot output (live):

| Date       | Skill                    | By     | Ticket  | Q | Model          | Dur  | What went well ...
| 2026-06-24 | /technical-due-diligence | Robert | TDD-011 | ✅ | claude-opus-4-8 | ~15m | Collision-safe census ...

cadence — wired surfaced by /dsu, not a new skill

  • The roll-up is the first active item in a weekly/monthly cadence package: dsu/reference/cadence.md (a reference, per your call — never a skill). /dsu Phase 4.5 reads it, and on the first DSU run of a new ISO week runs the roll-up and appends a "This week — skill activity" section to the standup post.
  • Fire guard: dsu/reference/cadence-state.json (gitignored) holds last-fire dates so a missed Monday still fires Tuesday and nothing double-fires (weekly = new %G-%V; monthly = new %Y-%m).
  • Designed to grow: planned rows already stub monthly knowledge-sync (/sync-knowledge), skills-drift (/sync-skills), and weekly repo-freshness. Add a row + flip to active.

Pilot, rollout & risks

pilot — done /technical-due-diligence

  • Final-step section rewritten to append runs.<name>.jsonl (9-field schema) instead of the central ledger.
  • Seeded with the real TDD-011 record; roll-up reproduces a ledger-style row and ignores the metric-style runs.jsonl files.

risk the real risk is write-discipline, not the format

  • Will the step run every time? The central ledger has the same dependency and mostly works because 14 skills carry an explicit final-step block. Mitigation: keep the block terse + identical (point to _shared/skill-run-log.md); a /sync-skills check can flag skills missing it.
  • duration capture is best-effort — there is no per-skill wall-clock today; pull from session-stats duration_minutes when a run ≈ a session, else omit. Don't block the log on it.
  • Transition split-brain — piloted skills log JSONL while others still log the ledger; the old ledgers stay frozen and the roll-up can ingest them once for a continuous history.

open — for the brainstorm decisions left

  • session-review: spans a session, not one skill — give it a session-level _shared/reviews/<date>.<name>.jsonl, or keep it on a slimmed central ledger?
  • Rollout breadth/order: pilot → the 13 other logging skills → all 59? Or orchestrators first?
  • Next cadence items to activate: monthly /sync-knowledge and /sync-skills are stubbed planned — turn either on now, or wait?
  • Roll-up reach: DSU weekly is wired; also expose on-demand in a revived /qe-dashboard view, or DSU-only for now?
