Verdict — the three findings that most change the decision

1. The IGS skill's flagship telemetry artifact does not exist. references/kql-recipes.md is a byte-identical duplicate of the system map (same MD5, zero KQL) — every telemetry phase, Quick Triage mode, and the Teams-bot automation depend on query templates that were silently lost in packaging. Adopt its methodology; do not trust its package until repaired. Flag to Andrew Willcoxon.

2. prod-support's fast lane runs on keys that are 3% implemented. The design mandates a greppable **Signature:** line on every KB incident block; only 1 of 35 blocks has one. Phase 3 degrades to symptom-text matching — exactly what the KB's own hardest-won lesson (CISD-1937, attributed wrong three times) forbids. One afternoon of backfill closes the gap.

3. These skills are complementary, not competing. IGS = stateless cross-system telemetry→code methodology (org backend, Excelergy-centric). prod-support = stateful IMA-360 incident memory + ticket lifecycle with safety gates the IGS skill lacks entirely (KB write-back, dedup, containment, prod-read gating). The value is the 9-item Harvest Map, not a winner.

Dimension 1 — Architecture, coupling, internal integrity

CRITICAL · IGS The KQL recipe library does not exist — kql-recipes.md is a byte-identical duplicate of the system map

Evidence: references/kql-recipes.md and references/igs-system.map.md — identical MD5, identical 10,294 bytes, verified this run:

md5sum references/*.md
7c04b89ae31ae52bf056434a57b5fb1f *references/kql-recipes.md
7c04b89ae31ae52bf056434a57b5fb1f *references/igs-system.map.md

The file's actual first line is # IGS System Map — it contains the service registry, not one line of KQL. Three load-bearing call sites depend on the advertised content: skill.md:69 — "run the standard health overview from references/kql-recipes.md: 1. Failure rate trend … 2. Top exceptions … 3. Top dependency failures … 4. Private Bytes pattern"; skill.md:182 (Quick Triage step 2) — "Run current memory state (last 15 min) — references/kql-recipes.md"; skill.md:142 advertises it as "KQL query templates for health overview, exception analysis, dependency analysis, memory pressure, cross-app correlation."

Why it matters: telemetry querying is this skill's core capability, and the failure is silent — the file exists, so no error surfaces; an agent (or Adam Perks's Teams bot, which cannot troubleshoot) improvises every KQL query while believing it's using the vetted library. The original recipe content is not recoverable from anywhere in the package.

Call path / reach: Phases 2 and 3, Quick Triage mode, Automated mode — i.e., every telemetry-driven run.

Severity rationale: Critical — the load-bearing artifact of the primary capability is absent, the failure mode is silent, and the content is unrecoverable from the package (skill-land equivalent of "entire subsystem unrecoverable").

Remediation sketch: author (or recover from Andrew's source) the real kql-recipes.md; add a packaging check that no two reference files hash equal.

HIGH · IGS Every citation of the service registry uses a filename that doesn't exist (igs-system-map.md vs actual igs-system.map.md)

Evidence: six occurrences across three files — skill.md:61 ("See references/igs-system-map.md"), skill.md:65, skill.md:144, ado-search-patterns.md:175 ("Repo IDs and default branches are in references/igs-system-map.md"), INTEGRATION_GUIDE.md:185, INTEGRATION_GUIDE.md:213. The file on disk is igs-system.map.md (dot, not hyphen). Zero in-package citations use the real name.

Why it matters: the registry is the Phase 1 entry step ("map the starting point to a service"). A literal-path read 404s; an interactive agent will usually recover by globbing, but the headless Teams-bot integration this skill was built for is exactly the consumer least able to recover. Combined with IGS-C1, two of the five reference files are mispackaged — the package was shipped without a smoke test of its own internal links.

Severity rationale: High — systematic (6/6 citations wrong) on the primary discovery path; partial mitigation is that search-based lookup still finds the file.

Remediation sketch: rename the file or fix the six citations; add a link-check to packaging.

HIGH · prod-support The fast lane's matching keys are 3% implemented — 1 of 35 incident blocks carries the mandated **Signature:** line

Evidence: the contract — skill.md:477: "Open each incident block with a **Signature:** line (the Phase 3 fingerprint) so future sessions can grep-match it — this is what makes the fast lane work"; mirrored at 00-index.md:73. The reality — a grep across knowledge-base/** this run found 35 ### Incident: blocks in 25 files and exactly one Signature line: crashes/IndustrialEntities-CustomerSegments-SqlTimeout.md:56 (added during the v1 smoke test).

Why it matters: Phase 3's known-vs-novel gate — the mechanism that keeps the common case cheap — must fall back to matching on free-text symptom descriptions. The KB's own most expensive lesson (00-index.md:18, CISD-1937: attributed wrong three times from symptom matching) is a 400-word warning against exactly that fallback. Severity rationale: High — the core mechanism of the skill's central design idea is unimplemented in the artifacts; silent, compounding, but cheap to fix and no data is lost.

Remediation sketch: one backfill pass writing 34 Signature lines from existing content (the fingerprints already exist as prose), plus a /sync-style check that counts ### Incident: vs **Signature:**.
