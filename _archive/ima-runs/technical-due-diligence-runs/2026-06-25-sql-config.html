Verdict — the three things to fix first in this slice

The environment routing (context.ts) is the strongest part of the whole layer: clean, centralized, fail-loud. The config files, though, carry the layer’s boundary and trust-boundary smells:

  1. [Medium] The “config” file is half a data-access layer. sql.industrial.config.ts holds 10 finder functions (HU-eligibility gates, contract-term readers, merge-verification queries) alongside the one thing that belongs there — getIndustrialPool. The CRM side keeps finders in sql.db.ts; the Industrial side mixes config, pool lifecycle, and queries in one 244-line file.
  2. [Medium] trustServerCertificate: true on every credentialed connection. Both config blocks disable TLS certificate validation, so the “encrypted” connection to CRM and Industrial (including QA) isn’t authenticated. Likely required by internal self-signed certs — but it’s a blanket default, not a scoped, documented decision.
  3. [Medium] Two near-identical config blocks, copy-pasted. CRM and Industrial repeat the same user/password/domain/timeout/TLS shape, differing only in server + database name — so a TLS or timeout policy change must be made in two places and will drift.

Net: extract a shared connection-config builder, scope the cert-trust decision, and relocate the Industrial finders out of the “config” file into the finder layer.

Phase 1 — Surface Map

Public surface & change-radius

  • sqlConfig() (sql.config.ts:6–21) — async; resolves creds via getServiceAccount('service-account'), targets Context.getCrmSqlServer(), database 'CRM'. Consumed only by getClient() in the facade.
  • getIndustrialPool(env?) (sql.industrial.config.ts:15–29) — returns a connected pool (database 'Industrial'); the caller owns close().
  • 10 inline Industrial finders (sql.industrial.config.ts:37–244) — filterAccountIdsWithServiceLocations, filterAccountIdsWithoutStagedHu, getActiveContractTerms, getContractTermsWith/WithoutSut, getNullLeenSlcaIdsForCustomer, getTriggerIsDisabled, resolveImaUserId, getColumnIsNullable, getSlcaLegalEntityNames.
  • Context (context.ts:37–61) — getCrmSqlServer() / getIndustrialSqlServer(env?) switch on NODE_ENV to a server hostname; throw on an unknown env.

Change-radius: sqlConfig feeds the entire CRM facade (one hop to all 228 call sites). getIndustrialPool + the inline finders are imported by the facade (10 finders re-exported on sqlDB) and the Industrial finders flow into HU/merge/contract-term tests. context.ts routing is reached by both DB paths and the app/API base URLs — the single env switch for the suite.

Dimension 1 — Architecture, Coupling, Boundaries

MEDIUM The “config” file is half a data-access layer — 10 finders live beside the pool factory

Evidence: src/utils/sql/sql.industrial.config.ts:37, 64, 91, 112, 130, 152, 175, 193, 207, 226

export const getIndustrialPool = async (env?) => { ... }                 // :15 — belongs in "config"
export const filterAccountIdsWithServiceLocations = async (...) => {...} // :37 — a FINDER
export const getActiveContractTerms = async (topN = 400) => {...}       // :112 — a FINDER
export const getContractTermsWithSut = async (topN = 400) => {...}      // :130 — a FINDER
export const getSlcaLegalEntityNames = async (slcaIds) => {...}         // :226 — a FINDER

Why it matters: The CRM half of the layer cleanly separates config (sql.config.ts) from finders (sql.db.ts). The Industrial half collapses them: a file named sql.industrial.config.ts is 90% query functions — HU-eligibility post-filters, Industrial contract-term readers, and party-merge verification helpers (CI-7759/7760) — with only getIndustrialPool actually being “config.” A reader auditing connection setup wades through 200 lines of business queries; a reader looking for an Industrial finder won’t think to look in a “config” file. The boundary the overhaul wants to draw is already drawn on the CRM side and ignored on the Industrial side.

Call path / reach: The facade imports all 10 from here (sql.db.ts:39–50) and re-exports them on sqlDB; moving them is a same-file-barrel change, low risk.

Severity rationale: Medium — coupling/naming boundary that makes the layer harder to reason about and to split during the overhaul.

Remediation sketch: Keep only getIndustrialPool (and the shared config builder, F2) in a config module; move the 10 finders into a finders/industrial.ts alongside the CRM finder split proposed in TDD-015 F6.

Dimension 3 — Security Surface & Trust Boundaries

MEDIUM trustServerCertificate: true disables TLS validation on every credentialed DB connection

Evidence: src/utils/sql/sql.config.ts:15–18 and src/utils/sql/sql.industrial.config.ts:24–27

options: {
  trustServerCertificate: true,   // accept ANY server cert — no validation
  encrypt: true                    // encrypted, but the endpoint is unauthenticated
}

Why it matters: encrypt: true with trustServerCertificate: true means the channel is encrypted but the server is not authenticated — a man-in-the-middle on the path to Crm.Sql.{INT,QA}.Igsenergy.net / industrial.sql.{int,qa}.igsenergy.net could present any certificate and harvest the service-account credentials and query traffic. It is duplicated across both config blocks, so the trust-all posture is the layer-wide default. On an internal corporate network with self-signed DB certs this is the usual pragmatic setting and the real-world risk is low — but it is a blanket default applied identically to every environment, not a scoped, documented decision, and it’s the kind of thing an acquirer flags on sight.

Severity rationale: Medium (security multiplier: it sits on credentialed DB connections). Held below High because the connections are internal-network and the data is non-production test traffic; the credential, however, is a real service account.

Remediation sketch: Pin the internal CA / server cert thumbprint instead of trust-all, or at minimum gate trustServerCertificate behind an explicit, commented “internal self-signed cert” flag in the shared config builder (F2) so the decision is visible and overridable per env.

Dimension 4 — Error Handling & Failure Modes

MEDIUM Each Industrial finder opens a brand-new pool (TCP connect + Key Vault fetch) per call

Evidence: src/utils/sql/sql.industrial.config.ts:16, 39–55, 92–106, 131–144

const pool = await getIndustrialPool();   // → getServiceAccount(...) + new ConnectionPool().connect(), EVERY call
try { ... } finally { await pool.close(); }  // correctly closed (unlike the CRM path, TDD-015 F1)

Why it matters: The Industrial finders get pool lifecycle right — every one closes in a finally (this is the correct pattern the CRM path is missing in TDD-015 F1). But each call still constructs a fresh pool, which means a credential resolution (getServiceAccount at :16) plus a full TCP connect per finder. The HU-eligibility path is the sharp edge: sqlDB.getProspectCiElectricAccountWithNoHu calls two Industrial finders back-to-back (filterAccountIdsWithServiceLocations then filterAccountIdsWithoutStagedHu), and getCiContractWithSut opens-and-closes an Industrial pool then opens a CRM client — three connect cycles for one data fixture.

Severity rationale: Medium — connect + secret cost per finder; the correct close means it’s a latency tax, not a leak.

Remediation sketch: A memoized per-worker Industrial pool (paralleling the CRM singleton in TDD-015 F1), with the inline finders sharing it; drop the per-call connect()/close() in favor of pool reuse.

Dimension 5 — Tech Debt & Maintainability

MEDIUM Two near-identical connection-config blocks, copy-pasted

Evidence: src/utils/sql/sql.config.ts:8–19 vs src/utils/sql/sql.industrial.config.ts:17–28

// sql.config.ts (CRM)                         // sql.industrial.config.ts (Industrial)
{ user, password, domain,                        { user, password, domain,
  server: Context.getCrmSqlServer(),               server: Context.getIndustrialSqlServer(env),
  database: 'CRM',                                 database: 'Industrial',
  requestTimeout: 60000,                           requestTimeout: 60000,
  options: { trustServerCertificate: true,         options: { trustServerCertificate: true,
             encrypt: true } }                                 encrypt: true } }

Why it matters: The two configs differ only in server and database; everything else — credential resolution, the 60s timeout, the TLS options — is duplicated verbatim. A change to the timeout, the TLS posture (F3), or pool sizing must be made in (at least) two places, and the credential-fetch call is copied rather than shared. This is the textbook drift surface the overhaul should close.

Severity rationale: Medium — duplication that compounds; the natural home for the F3 cert decision and any future pool-size tuning.

Remediation sketch: One buildSqlConfig({ server, database }) that resolves creds once and returns the common shape; CRM and Industrial call it with their two differing fields. The cert-trust flag lives there, once.

LOW Three different ways to load the mssql module across the layer

Evidence: sql.industrial.config.ts:7–8 (createRequire shim) · sql.config.ts:4 (import * as mssql) · sql.client.ts:3–6 (createRequire shim)

const require = createRequire(import.meta.url); const sql = require('mssql') as typeof mssqlTypes;  // industrial.config + client
import * as mssql from 'mssql';                                                                     // config.ts — different

Why it matters: The CJS-in-ESM require('mssql') shim appears in sql.client.ts and sql.industrial.config.ts, while sql.config.ts uses a plain ESM import. Three spellings of one dependency-load is a small inconsistency that complicates the eventual module split. (Same root as TDD-015 F9.)

Severity rationale: Low — works; consistency only.

Remediation sketch: One internal typed mssql re-export module consumed everywhere.

LOW Inconsistent IN-list parameterization within the same file

Evidence: interpolated IN-lists at sql.industrial.config.ts:49, 77, 100, 238 vs correct .input() at 178, 196, 210–213

WHERE PartyAccountId IN (${idList})     // :49 — idList = parseInt+Number.isFinite+join (safe-by-construction)
WHERE c.CustomerId = ${id}              // :100 — id = parseInt(...) (safe)
.input('triggerName', triggerName)...   // :178 — CORRECT parameterization
.input('userName', userName)...         // :196 — CORRECT (UserName carries a backslash)

Why it matters: The file does both: scalar string params are correctly bound with .input() (and the resolveImaUserId comment even notes “Parameterized — UserName carries a backslash”), but ID lists are interpolated. The interpolated lists are sanitized (every element is parseInt+Number.isFinite-filtered before join), so there is no live injection — but two styles for the same job is the inconsistency the overhaul should resolve, and it sets the wrong example next to the correct one.

Severity rationale: Low — safe today; consistency, and it pairs with the genuine string-interpolation finding in the queries layer (TDD-017 F1).

Remediation sketch: Standardize on table-valued parameters or repeated .input() placeholders for IN-lists; if interpolation is kept for sanitized integer lists, centralize it in one toIdList() helper with the guard, so the pattern is named and auditable.

Dimension 2 — Test Coverage, Quality & Gaps

LOW The HU-eligibility gates decide test data but are unverified

Evidence: sql.industrial.config.ts:37–86 (filterAccountIdsWith/WithoutStagedHu); sql sanity spec covers query strings, not these post-filters

Why it matters: filterAccountIdsWithServiceLocations and filterAccountIdsWithoutStagedHu silently return [] on no match, and they are the gates that decide whether an HU test gets runnable data (the finder chain in sql.db.ts:377–386 throws domain errors when they empty). They’re inline Industrial queries not covered by the SQL sanity spec (which runs the queries/* strings), so a column rename on the Industrial side (e.g. the documented IMA.ServiceLocations drift) would make them quietly return fewer/zero rows. Low because the downstream throws give a clear symptom — but the gate logic itself is unverified.

Severity rationale: Low — failure is loud at the call site even though the gate is untested.

Remediation sketch: Add the Industrial finders to the sanity-spec sweep (or a sibling Industrial sanity spec) so their column references are exercised against the live schema.

Dimension 6 — Dependency Health & Version Risk

LOW Correct, centralized env routing — with one cosmetic inconsistency (advisory)

Evidence: context.ts:5 (dotenv override), 37–46 (CRM), 52–61 (Industrial), 9–20 (base URLs)

dotenv.config({ override: true, path: ... });  // context.ts:5 — .env beats shell NODE_ENV (documented trap)
static getCrmSqlServer() { switch(this.env){ case 'int':...; default: throw ... } }  // :37–46 — throws (no prod)
static getBaseURL()      { switch(this.env){ ... case 'prod': return ''; ... } }      // :15 — returns '' for prod

Why it matters: The routing layer is the healthiest code in scope — one switch per resource, hosts confirmed against .mcp.json, fail-loud throw on an unknown env, and an explicit env override on the Industrial server for the INT/PRO migration verifiers. Two advisories: (1) dotenv.config({ override: true }) at module load means the .env file beats the shell NODE_ENV — the documented “green run against the wrong env” trap (known-gotchas.md); the config layer’s correctness silently depends on it. (2) Minor inconsistency: for prod, getBaseURL returns '' while getCrmSqlServer throws — two different “no prod” behaviors. No dependency/version risk in this slice.

Severity rationale: Low / informational — by-design behavior; the override trap is a documentation/awareness item, not a defect here.

Remediation sketch: Leave routing as-is; add a one-line comment at context.ts:5 pointing to the env-override gotcha, and make the prod behavior uniform (both throw, or both return a sentinel) if prod is ever wired.

Dimension 7 — Performance Hotspots

No standalone findings — perf cost is the per-call connect/secret fetch (F4)

The only performance concern in this slice is structural and already captured in F4 (a fresh pool + credential fetch per Industrial finder call). The queries these finders run are TOP-bounded and single-statement; no loops, no N+1, no unbounded result sets in the config layer itself.
